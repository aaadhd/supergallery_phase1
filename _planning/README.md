**서비스명**: Artier (SuperGallery Phase 1)

## 프로젝트 구성

Artier Phase 1은 **2트랙 단일 프로젝트**로 구성된다.

| 트랙 | 접두어 | 대상 사용자 |
|---|---|---|
| 사용자 앱 (User) | `USR-` | 시니어·중장년 작가, 감상자 |
| 운영 어드민 (Admin) | `ADM-` | 운영팀(Operator 권한) |

Phase 2에서 외부 주최자·파트너 작가 포털 등 새 트랙이 추가되면 새 접두어를 신설한다.

## 문서 갱신 규칙

> 요구사항 변경·정책 개정·새 화면 추가 등은 **반드시 이 폴더의 해당 문서에 직접 반영**한다. 폴더 바깥의 다른 설명 문서는 참고용이며, 최종 기준은 이 폴더가 가진다.

- 코드 리포지토리의 `CLAUDE.md`는 AI 코딩 세션용 컨텍스트(코딩 규칙·파일 인덱스)이며, 기획 기준은 **언제나 `_planning/`가 우선**한다.
- 설명 문서가 본 폴더의 정책과 충돌하면 **본 폴더가 이긴다**. 설명 문서는 후속 반영.
- 새 문서를 추가할 때는 이 README의 "기획 문서 목록"에 파일을 등록한다.

## 문서 읽는 순서

**처음 프로젝트를 보는 사람**
1. 이 파일(README) 읽고 전체 구조·핵심 결정 이해
2. `IA_ScreenList_v1.md` → 어떤 화면들이 있는지 감 잡기
3. 필요에 따라 `Policy_v1.md` / PRD 깊이 들어가기

**새 기능을 추가하려는 PM**
1. `Policy_v1.md`에 관련 정책이 있는지 확인
2. `IA_ScreenList_v1.md`에 해당 화면이 있는지 확인
3. 없으면 새로 정의해서 PRD·IA 동시 업데이트

**UI를 만드는 디자이너·프런트**
1. `PRD_User_v1.md` / `PRD_Admin_v1.md`에서 해당 화면 카드
2. `Policy_v1.md` §19(시니어 친화 UX) 및 노출 규칙·제약

**운영팀이 기능 이해가 필요할 때**
1. `Policy_v1.md` 해당 섹션(큐레이션·신고·초대 등)
2. `PRD_Admin_v1.md`에서 화면 동작

**개발팀이 핸드오프 받았을 때**
1. `DELTA.md` → 직전 핸드오프 이후 변경된 부분부터 확인
2. `IA_ScreenList_v1.md` → 화면 ID·경로·접근권한 매핑
3. `PRD_User_v1.md` / `PRD_Admin_v1.md` → 화면별 입력·처리·출력·수용기준
4. `Policy_v1.md` → 정책·SLA·수치 상한·§31 런칭 전 미해결 항목·§32 데이터 연쇄 정리
5. 데이터 모델·번들 분할·외부 연동 채널 선정·스토어 구현 등은 개발팀이 결정하여 코드 단일 소스로 운용

### 충돌 방지 구조 (단일 소스 고정)

아래처럼 **영역별 원문을 1개 문서로만** 관리한다.  
같은 영역의 원문을 다른 문서에 복제하지 않는다.

| 영역 | 단일 소스 문서 | 다른 문서 역할 |
|---|---|---|
| 정책/요건 | `Policy_v1.md` | IA/PRD는 정책을 화면에 매핑한 구현 명세 |
| 화면 구조/라우트 | `IA_ScreenList_v1.md` | PRD는 화면별 처리/AC 상세 |
| 사용자/어드민 화면 명세 | `PRD_User_v1.md`, `PRD_Admin_v1.md` | Policy/IA를 구현 단위로 풀어쓴 문서 |
| 일반 UI 카피 | `Copy_v1.md` | `Handoff_*`는 키 목록/적용 범위/체크리스트만 제공 |
| 법무 전문 원문 | `Handoff_Terms_v1.md`, `Handoff_Privacy_v1.md` | Policy/PRD는 반영 위치와 요구사항만 기록 |

운영 원칙:
- 원문 수정은 단일 소스 문서에서만 수행
- 나머지 문서는 **링크/키/적용 범위**만 갱신(원문 복붙 금지)

## 문서 버전 규칙

기획 문서를 수정할 때는 아래를 **같이** 처리한다. 세부 판별·푸시 단위 규칙은 프로젝트 루트 `CLAUDE.md`의 **「문서 동기화·버전 규칙」** 과 동일하다(여기서는 요약만).

### 1) 각 문서의 버전 표기

| 변경 유형 | 버전 | 예 |
|---|---|---|
| 내용 추가·변경·삭제 (하위 호환) | 마이너 +0.1 | v1.0 → v1.1 |
| 구조 전면 재편·대규모 개편 | 메이저 +1.0 | v1.2 → v2.0 |

파일명은 **`Policy_v1.md` 등 `_v1` 접미**를 유지한다. 마이너·패치 수준 변경마다 파일명을 바꾸지 않고, **같은 파일 내** 상단 변경 요약 + 말미 `## 문서 이력`으로만 추적한다. 메이저 개편으로 문서를 쪼개거나 아카이브(`_planning/archive/` 등)로 옮길 때만 파일명 분리를 검토한다.

### 2) 문서 이력 기록

각 문서의 **맨 마지막**에 `## 문서 이력` 섹션을 둔다. **같은 푸시 사이클 안의 추가 수정**은 동일 버전 행에 변경 내용만 이어 쓰고, **다음 푸시 이후 첫 수정**부터 새 버전 행을 추가한다(자세한 판별은 `CLAUDE.md`).

```markdown
## 문서 이력

| 버전 | 일자 | 작성 | 변경 내용 |
|------|------|------|----------|
| v1.3 | 2026-04-20 | PM | §24·§25·§26 신설 — 인터랙션·이벤트·저작권 보호 정책 |
| v1.2 | 2026-04-20 | PM | §12 신고·모더레이션 Phase 1 단순화(3액션+2회 자동 비공개) |
| v1.1 | 2026-04-19 | PM × Claude | Phase 2 용어 정비 — 2차 그랜드 오픈 한정 표기로 통일. 런칭 전 백엔드 연동 작업은 "런칭 전 백엔드 연동 후"로 교체. |
| v1.0 | 2026-04-19 | PM × Claude | 최초 작성 — 기존 분산 문서를 `_planning/` 단일 소스로 재편. 2트랙 구조 · 화면 ID 체계 · 버전 규칙 · 핵심 결정 · 언어 정책 · 외부 참고 문서 연결 포함. |
```

문서 **상단**에도 가장 최근 변경 요약 블록을 유지한다.

```markdown
> **v1.2 변경 요약 (2026-04-20)**:
> - §12 신고·모더레이션 Phase 1 단순화(3액션 + 2회 자동 비공개)
```

### 3) 이 README 동기화

새 기획 파일을 추가하거나 파일명을 바꿀 때만, 위 **「기획 문서 목록」** 트리·표·링크를 함께 갱신한다.

## 서비스 핵심 결정 요약

처음 보는 사람이 빠르게 서비스를 파악할 수 있게 중요한 결정 사항만 모았다. 자세한 내용은 각 문서 참조.

### 사용자·대상

| 항목 | 내용 |
|---|---|
| 타깃 사용자 | 시니어·중장년 순수미술 작가(50대 이상), 그리고 작품을 감상하는 일반 사용자 |
| 핵심 가치 | "시니어 작가도 어려움 없이 본인의 작품을 전시·공유할 수 있는 디지털 갤러리" |
| 기본 톤 | 친근·존중·여유. 기술·SNS 용어 최소화 |
| Phase 1 범위 | 기능 전반 완성 + 실서비스 출시까지 |
| Phase 2 범위 | 작품 카테고리 분류(art/fashion/craft/product), **어드민 권한 3단계 분리**(최고 운영자/에디터/뷰어), 댓글·메시지·그룹 커뮤니티 확장 |

### 플랫폼

| 항목 | 내용 |
|---|---|
| 사용자 앱 | 웹 (모바일 375px 우선 · 데스크톱 반응형, 시니어 친화 UX) |
| 어드민 | 데스크톱 1280px 우선 운영 콘솔 |
| Phase 1 백엔드 | 미연동(런칭 전 백엔드 연동 후 운영. 수단·구체 스택은 개발팀 결정) |

### 디자인 방향 (PM 정책 한정)

### 주요 기능

- 전시 업로드: 혼자 / 함께(참가자·강사 역할). 1~10장 이미지. 카메라 사진(EXIF 감지) 차단.
- 둘러보기 피드: 6 버킷 인터리빙(Pick·Theme·Featured·Personalized·Recent·Rest).
- 큐레이션: Pick(주간 10 · 영구 이력 배지) · 기획전(묶음) · 이벤트(응모 + 선정).
- 링크 공유 카드: 전시 링크를 카톡·문자·SNS에 올릴 때 썸네일 카드(미리보기 이미지) 자동 생성.
- 인터랙션(좋아요·저장·팔로우): 전시 단위 토글, 카운트 UI 비노출. 본인 작품 허용.
- 비가입자 초대: 작가가 전화번호 또는 이메일을 입력해 초대 → 검수 승인 시점에 발송. 가입자 식별자가 일치하면 가입 마지막 단계의 본인 확인을 거쳐 회원 슬롯으로 연결(Policy §3.5). 사후 해제(Disavow) 가능.
- 신고·모더레이션 (Phase 1): **3액션(삭제·기각·비공개 유지) + 같은 전시 2회 신고 자동 비공개**. 사용자 계정 제재 확장은 추후 별도 로드맵 검토.
- 이미지 저작권 보호: 우클릭·드래그·길게누름 차단, 피드·카드는 1200px 썸네일 제공(원본 해상도 직접 노출 금지), 파일명에 작가명·전시명 등 식별 정보 미포함.
- 알림: 사용자가 가진 연락처 기준으로 자동 라우팅 — 한국 전화번호 보유자는 카카오 알림톡(SMS 폴백), 그 외는 이메일.

## PM 확정 결정 (개발자 작업 제외)

아래 2개는 PM/법무/운영에서 **이미 확정한 결정값**이며, 개발자는 이 기준으로 구현/반영한다.

| 항목 | 확정 결정 | 개발 전달물 |
|---|---|---|
| N-10 약관·개인정보처리방침 법무 확정 | 사용자 모집 시작 전 확정본만 게시, 시행일은 모집 시작일과 동일, 마케팅 수신 동의는 선택 동의(기본 OFF), 권리 행사 채널은 USR-INF-07 단일 채널, DPO 정보는 USR-INF-06 공개 | 확정 약관/처리방침 링크 + 공지 문안 + 문의 템플릿 |
| N-6 데이터 이전 정책 | 이전 대상/제외 대상/마스킹 기준 고정(전화번호 해시+마스킹), 검증 기준(레코드 100% 일치 + 샘플 50건 + 핵심 플로우 PASS), 롤백 기준(핵심 실패 1건 또는 불일치 0.5% 초과), 공지 시점 D-7/D-1/당일 고정 | 이전 실행 체크리스트 + 롤백 체크리스트 + 공지문 3종 |

개발자에게 바로 전달할 항목:
- 법무 확정본 링크(약관/처리방침)
- 반영 대상 화면(USR-INF-05/06, USR-INF-07, ADM-INQ-01)
- 이전 성공/실패 판정 수치(100%, 50건, 0.5%)
- D-7/D-1/당일 공지문

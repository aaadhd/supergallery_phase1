## 프로젝트 구성

Artier Phase 1은 **2트랙 단일 프로젝트**로 구성된다.

| 트랙 | 접두어 | 대상 사용자 |
|---|---|---|
| 사용자 앱 (User) | `USR-` | 시니어·중장년 작가, 감상자 |
| 운영 어드민 (Admin) | `ADM-` | 운영팀(Operator 권한) |

## 문서 갱신 규칙

> 요구사항 변경·정책 개정·새 화면 추가 등은 **반드시 이 폴더의 해당 문서에 직접 반영**한다. 폴더 바깥의 다른 설명 문서는 참고용이며, 최종 기준은 이 폴더가 가진다.

- 코드 리포지토리의 다른 MD 문서(`CLAUDE.md`, `CURRENT_SPEC.md`, `POLICIES.md`, `product-policies.md` 등)는 구현 스냅샷이며, 기획 기준은 **언제나 `_planning/`가 우선**한다.
- 설명 문서가 본 폴더의 정책과 충돌하면 **본 폴더가 이긴다**. 설명 문서는 후속 반영.
- 새 문서를 추가할 때는 이 README의 "기획 문서 목록"에 파일을 등록한다.

## 문서 읽는 순서

**처음 프로젝트를 보는 사람**
1. 이 파일(README) 읽고 전체 구조 이해
2. `SystemArchitecture_v1.md` → 서비스 한 장 그림
3. `IA_ScreenList_v1.md` → 어떤 화면들이 있는지 감 잡기
4. 필요에 따라 `Policy_v1.md` / PRD 깊이 들어가기

**새 기능을 추가하려는 PM**
1. `Policy_v1.md`에 관련 정책이 있는지 확인
2. `IA_ScreenList_v1.md`에 해당 화면이 있는지 확인
3. 없으면 새로 정의해서 PRD·IA 동시 업데이트

**UI를 만드는 디자이너·프런트**
1. `DesignSystem_v1.md`의 공통 토큰·컴포넌트 먼저
2. `PRD_User_v1.md` / `PRD_Admin_v1.md`에서 해당 화면 카드
3. 필요 시 `Policy_v1.md`에서 노출 규칙·제약 확인

**운영팀이 기능 이해가 필요할 때**
1. `Policy_v1.md` 해당 섹션(큐레이션·신고·초대 등)
2. `PRD_Admin_v1.md`에서 화면 동작

## 문서 버전 규칙

기획 문서를 수정할 때는 아래 세 가지를 **같이** 처리한다.

### 1) 파일 버전 올리기

| 변경 유형 | 버전 | 예 |
|---|---|---|
| 내용 추가·변경·삭제 (하위 호환) | 마이너 +0.1 | v1.0 → v1.1 |
| 구조 전면 재편·대규모 개편 | 메이저 +1.0 | v1.2 → v2.0 |

파일명도 함께 바꾼다 (`Policy_v1.md` → `Policy_v1.1.md`).
구버전 파일은 아카이브 폴더(예: `_planning/archive/`)로 이동하고, 최신 파일의 **문서 이력** 섹션에 변경 요약을 남긴다.

### 2) 문서 이력 기록

각 문서의 **맨 마지막**에 `## 문서 이력` 섹션을 둔다. 수정할 때마다 한 행 추가.

```markdown
## 문서 이력

| 버전 | 일자 | 작성 | 변경 내용 |
|------|------|------|----------|
| v1.1 | 2026-04-20 | PM | §5 자동 정지 승격 허위 신고 횟수 3회 → 5회로 변경 |
| v1.0 | 2026-04-19 | PM × Claude | 최초 작성 |
```

문서 **상단**에도 가장 최근 변경 요약 블록을 유지한다.

```markdown
> **v1.1 변경 요약 (2026-04-20)**:
> - §5 자동 정지 승격 기준 허위 신고 3회 → 5회
```

### 3) 이 README 동기화

문서 버전이 바뀌면 위 "기획 문서 목록"의 파일명·링크도 함께 업데이트한다.

## 서비스 핵심 결정 요약

처음 보는 사람이 빠르게 서비스를 파악할 수 있게 중요한 결정 사항만 모았다. 자세한 내용은 각 문서 참조.

### 사용자·대상

| 항목 | 내용 |
|---|---|
| 타깃 사용자 | 시니어·중장년 순수미술 작가(50대 이상), 그리고 작품을 감상하는 일반 사용자 |
| 핵심 가치 | "시니어 작가도 어려움 없이 본인의 작품을 전시·공유할 수 있는 디지털 갤러리" |
| 기본 톤 | 친근·존중·여유. 기술·SNS 용어 최소화 |
| Phase 1 범위 | 기능 전반 완성 + 실서비스 출시까지 |
| Phase 2 범위 | 이벤트 공모 시스템 고도화, 운영자 내부 컬렉션, 작품 카테고리 분류, 커뮤니티 확장 |

### 플랫폼·기술

| 항목 | 내용 |
|---|---|
| 플랫폼 | 웹 (모바일 375px 우선 · 데스크톱 반응형) |
| 프런트 | 모바일 우선 · 시니어 친화 UX(44px 터치·16px 입력·3단 글꼴 조절) |
| 어드민 | 데스크톱 1280px 우선 |
| Phase 1 저장 | 브라우저 로컬 저장(클라이언트 단일) |
| Phase 2 저장 | BaaS 이관(Supabase 등) |

### 디자인 시스템

| 항목 | 내용 |
|---|---|
| 폰트 | Pretendard(주) + 시스템 기본 한글·영문 폴백 |
| Primary Color | 모호·딥 플럼 (`oklch(0.46 0.075 328)`) — 온라인 갤러리의 차분한 공기감 |
| 디자인 컨셉 | "작품이 주인공. UI는 뒤로 물러난다" — 차분·여유·존중 |
| 다크 모드 | 토큰 준비 완료, 설정 UI는 Phase 2 |

### 수익화·계정·인증

| 항목 | 내용 |
|---|---|
| 수익화 모델 | **Phase 1 무료.** Phase 2 이후 수익 모델 별도 논의 |
| 계정 모델 | 단일 사용자 계정(가족 계정 없음). 탈퇴 시 작가명 "작가 미상"으로 익명화, 작품은 유지 |
| 소셜 로그인 | 카카오 · 구글 · 애플 (Phase 1 모의 → Phase 2 OAuth 실연동) |
| 이메일 로그인 | 지원. Phase 1 모의 → Phase 2 이메일 소유권 인증 |
| 최소 연령 | 만 14세 |
| 역할 | Guest / Creator / Instructor(자동 파생) / Operator |

### 주요 기능

- 전시 업로드: 혼자 / 함께(참가자·강사 역할). 1~10장 이미지.
- 둘러보기 피드: 6 버킷 인터리빙(Pick·Theme·Featured·Personalized·Recent·Rest).
- 큐레이션: Pick(주간 10 · 영구 이력 배지) · 기획전(묶음) · 이벤트(응모·선정) · 내부 컬렉션(Phase 2).
- 인터랙션: 좋아요·저장·팔로우·공유·신고.
- 신고·제재: 4액션(삭제·경고·기각·비공개) + 자동 승격(경고 3회·허위 3회 → 7일 정지).
- 알림: 카카오 알림톡(한국)·SMS 폴백·이메일(해외·법적 의무).

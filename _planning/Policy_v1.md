## 1. 알림 채널 정책

### 1.1 한국 사용자 (전화번호 `+82`)

| 용도 | 1순위 | 폴백 | 비고 |
|---|---|---|---|
| 일반 알림(좋아요·팔로우·초대·검수 결과) | 카카오 알림톡 | SMS | — |
| 비밀번호 재설정 | 카카오 알림톡 | SMS | — |
| **법적 의무 알림**(약관·개인정보처리방침 변경) | 이메일 | — | 법적 기록 보존 목적 |

> 한국 사용자에게 이메일은 **법적 의무 알림 외에 사용하지 않는다**.

### 1.2 해외 사용자 (전화번호 `+82` 외)

| 용도 | 채널 | 비고 |
|---|---|---|
| 모든 알림 | 이메일 | 알림톡·SMS 미지원 |

### 1.3 채널 판정 규칙

- 전화번호 정규화 후 숫자 **7자리 이상** 필수. 그 미만은 잘못된 번호로 간주하고 발송 차단.
- `+82`로 시작하거나 한국 이동통신 형식(`01x`)으로 판정되는 번호는 "한국 사용자".
- 그 외는 "해외 사용자"로 이메일 단일 채널.

### 1.4 Phase 1 구현 상태

SMS·카카오 알림톡·이메일은 전부 **모의**(콘솔 로그 + 발송 로그 기록, 약 5%의 임의 실패 시뮬레이션). 런칭 전 실제 게이트웨이 연동 필요(SystemArch §5 참조).

## 2. 가입·본인인증 정책

### 2.1 필수 수집 정보

| 항목 | 필수 여부 | 검증 |
|---|---|---|
| 전화번호 | **필수** (모든 경로) | 본인인증 API(PASS 등) — **런칭 전 연동 필요**. 현재는 입력 형식 검증만. |
| 실명 | **필수** (모든 경로) | 본인인증 결과에서 자동 수집. 2~20자, 비속어 불가. |
| 생년월일 | **필수** | 만 14세 이상 검증. |
| 이메일 | 선택(한국), **필수**(해외·소셜 첫 가입) | 형식 + 중복 검사. |
| 닉네임 | **필수** | 2~20자, 비속어 불가. |
| 비밀번호 | 이메일 가입만 | 8자 이상 + 영문·숫자 각 1자 이상. |

### 2.2 만 14세 검증

- **한국**: 본인인증 결과로 자동 검증. 본인인증 API 연동 전까지는 온보딩에서 생년월일 입력으로 대체.
- **해외**: 가입 시 생년월일 입력 + 나이 계산.
- **소셜 가입**: 현재는 OAuth 결과만 신뢰(연령 검증 빈틈 — 한시적). **런칭 전** 본인인증 또는 생년월일 추가 수집으로 보완(SystemArch §10 N-4).

### 2.3 중복 가입 차단

- 이메일·전화번호 **중복 시 가입 차단**.
- 소셜 최초 가입 → 약관 동의 + 닉네임 수집 모달(USR-AUT-05) → 온보딩(USR-AUT-06).
- 온보딩은 가입 경로에서 이미 받은 정보를 **프리필**한다(사용자가 수정 가능).

| 가입 경로 | 프리필 대상 |
|---|---|
| 이메일 가입(USR-AUT-03) | 이메일 |
| 소셜 첫 가입(USR-AUT-05) | 닉네임 + 소셜 제공자 이메일 |
| SMS 초대 링크(USR-EXH-03) | 초대받은 전화번호·표시명 |

온보딩 종료 시 모든 프리필 임시 값은 일괄 정리된다.

### 2.4 세션·자동 로그인

- 로그인 세션은 토큰 형태로 보관. 현재는 모의 토큰(클라이언트 전용), **런칭 전** 실제 JWT + 백엔드 세션으로 교체.
- Phase 1엔 사용자 계정 차원 정지가 없다(§12.3). 관련 정지·차단·이의제기 플로우는 Phase 2에서 재설계.

## 3. 비가입자 초대 정책

### 3.1 발송 시점

- 업로드 시 **즉시 발송하지 않는다**. 전화번호는 전시 내 참여 작가 정보에 일시 보관.
- **검수 승인(approved) 시점에 발송**한다.
- 검수 반려 시 보류된 초대는 폐기.

### 3.2 채널

| 조건 | 채널 |
|---|---|
| 한국 전화번호 | 카카오 알림톡 → SMS 폴백 |
| 해외 전화번호 | 이메일 |

### 3.3 삭제·취소 처리

- 초대 보류 중 작품 삭제 시: **"초대된 비회원이 있습니다. 삭제하면 초대가 무효화됩니다"** 경고 표시. 삭제 자체는 차단하지 않음(게시자 자유).
- 발송 후 작품 삭제 시: 초대 링크 랜딩(USR-EXH-03)에서 "이 전시는 삭제되었습니다" 안내 + Artier 가입 CTA 유지.
- **발송 후 자동 비공개(§12.2)·확정 비공개 시**: 초대 링크 랜딩에서 "이 전시는 운영팀 검토 중이에요. 검토 완료 후 다시 확인해주세요" 안내 + Artier 가입 CTA 유지. 전시 본문은 비노출.

### 3.4 중복·가입자 보호

- 이미 가입된 전화번호에는 초대 발송 **차단**.
- 동일 `(전화번호 해시, 전시 ID)` 조합은 중복 발송 차단.
- 초대 발송 로그에는 전화번호를 **마스킹 형태**(예: `010-****-5678`)로만 저장. 원본 전화번호는 발행 직후 스크럽.

### 3.5 가입 시 자동 매칭

가입(PASS 본인인증 완료) 시 **전화번호가 일치하는 모든 초대**를 조용히 자동 연결한다. 실명 대조 없음 — PASS가 (신원 + 번호) 결합을 검증하므로 번호 단독으로 소유권이 보장된다.

| 케이스 | 처리 |
|---|---|
| 전화번호 일치 | 초대된 전시의 참여 작가 슬롯(`imageArtists[i]`)을 회원 슬롯으로 자동 승격. 가입 직후 안내 UI 없음 — 마이페이지 전시 탭에 자연 노출 |

자동 연결된 결과가 본인 작품이 아닌 경우(예: 발신자의 번호 오입력, 재활용 번호), 마이페이지에서 사후 **원복(disavow)** 할 수 있다. 원복 시 해당 슬롯은 **작가 미상(`'unknown'`)** 으로 전환되고, 번호·이름은 스크럽된다(§4.1 탈퇴 작가 표시와 동일 패턴). 경고·제재는 없다.

### 3.5.1 알림 전송 (왕복 피드백)

초대 자동 연결 및 원복 이벤트 시 **초대한 작가(inviter)** 에게 시스템 알림이 발송된다.

| 경로 | 알림 문구 |
|---|---|
| **자동 매칭 성공** | "초대한 '{name}' 님이 가입해 전시 '{title}'의 참여 작가로 연결되었어요." |
| **원복(disavow)** | "'{workTitle}' 의 참여 작가 슬롯이 작가 미상으로 전환되었어요. 필요 시 전시 편집에서 다시 초대해주세요." |

- **타입**: 둘 다 `system` — 알림 센터에서 회색 배경
- **workId 포함**: 알림 탭 시 해당 전시 상세로 이동
- 원복 시 경고 카운트 없음 — 발신자 오입력·재활용 번호 모두 악의성으로 판단하지 않음. 정보 전달 목적의 알림만 발송.

## 4. 탈퇴 작가 정책

### 4.1 작가 정보

| 항목 | 탈퇴 후 |
|---|---|
| 작가명 | **"작가 미상"** |
| 아바타 | 기본 이미지 |
| 프로필 페이지 | **접근 불가** |

> "작가 미상" 라벨은 **§4 탈퇴 작가**와 **§23.3 disavow된 슬롯(`type: 'unknown'`)** 에서 동일 문구로 노출된다. 열람자 UX 관점에서는 **"이 슬롯에 책임 있는 작가가 현재 없음"** 이라는 같은 의미로 취급한다. 내부적으론 출처가 다르지만(탈퇴/원복) Phase 1에선 사용자에게 구분 노출하지 않는다. Phase 2에서 필요성 있으면 라벨 분리 검토.

### 4.2 작품

| 항목 | 탈퇴 후 |
|---|---|
| 작품 이미지 | 유지 |
| 작품명·전시명·그룹명 | 유지 |
| 피드 노출 | **유지** |
| 좋아요·저장 | 차단(기존 인터랙션은 유지, 신규 불가) |
| 팔로우 | 차단 |
| 신고 | 차단 |

### 4.2.1 차단 적용 지점 (Phase 1 구현)

**사용자 앱 UI 가드**
- WorkDetailModal 액션 바: 탈퇴 작가의 작품이면 opacity 40% + pointer-events-none으로 시각·상호작용 차단 (좋아요·저장·팔로우·신고·공유 묶음).
- 프로필 페이지: 타인이 탈퇴한 경우 프로필 접근을 전용 안내 화면("탈퇴한 작가입니다")으로 전환 + "둘러보기로 돌아가기" CTA. 본인의 탈퇴는 §4.3 사유 수집 플로우에서 이미 처리.

**저장소 레벨 가드 (defense-in-depth)**
- **좋아요·저장 토글**: 대상 작품의 작가가 탈퇴 상태면 no-op (silent skip). UI 가드가 뚫렸을 때 최종 방어선.
- **팔로우 토글**: 대상 작가가 탈퇴 상태면 no-op.
- **신고 모달(CM-03) 제출**: 대상 작품의 작가가 탈퇴 상태면 에러 토스트 "탈퇴한 작가의 작품에는 신고할 수 없어요." 표시 후 제출 중단.

**차단 대상이 아닌 것 (유지되는 것)**
- 피드·검색에서 작품 노출 자체는 유지 (작가명만 "작가 미상"으로 표시).
- 기존에 좋아요·저장한 사용자 기록은 그대로 보존 (카운터도 유지).
- 운영자 어드민 액션(비공개·삭제 등)은 탈퇴 여부와 무관하게 정상 동작.

### 4.3 탈퇴 사유

USR-STG-03(탈퇴 모달)에서 필수 선택: **이용 빈도 낮음 / 타 서비스 이용 / 개인정보 우려 / 이용 불편 / 기타**.
운영 개선 반영 목적. 데모·분석용으로 최근 탈퇴 사유 1건만 남긴다.

### 4.4 재가입

- 탈퇴 후 같은 이메일·전화로 **즉시 재가입 가능**(대기 기간 없음).
- 재가입은 **신규 계정**으로 취급된다. 과거 계정의 작품·좋아요·저장·팔로우·신고 이력은 복구되지 않음.
- 과거 작품은 §4에 따라 "작가 미상" 상태로 피드에 계속 남음. 재가입한 계정과는 **자동으로 연결되지 않는다**.
- **"작가 재활성" 흐름**(과거 작품 복구 선택지 제공)은 Phase 2 확장 논의(SystemArch §10 N-6).
- 재가입 계정이 과거 탈퇴 이력을 갖는지는 Phase 1에선 식별하지 않는다(해시 대조 없음).

## 10. 공유 정책

### 10.1 공유 버튼 규칙

- **공유 버튼은 1개.** "공유하기"를 누르면:

### 10.2 공유 콘텐츠

- 전달 내용: `"[작가명] 님이 [전시명] 전시에 초대합니다. Artier에서 작품을 감상해 보세요. [링크]"`
- **링크만 단독 전달 금지.** 항상 멘트와 함께.

### 10.3 공유 URL 쿼리 분기

전시 URL은 진입 맥락을 쿼리 파라미터로 구분한다.

| 쿼리 | 랜딩 화면 | 용도 |
|---|---|---|
| (없음) | 전시 상세 모달(USR-EXH-01) | 피드에서 직접 탭 |
| `?from=invite` | 초대장 랜딩(USR-EXH-03) | 비회원 초대 링크 |
| `?from=credited` | 초대장 랜딩(USR-EXH-03) | 크레딧된 참여 작가 공유 |
| `?from=work` | 작품 공유 랜딩(USR-EXH-04) | 단일 작품 공유 |

공유 버튼은 항상 `?from=work` 또는 `?from=invite`를 **자동으로 부여**한다.

## 11. 카메라 사진 차단 정책

Artier는 순수미술 플랫폼이므로 스마트폰·DSLR로 촬영한 사진 업로드를 차단한다.

### 11.1 판별 방법

1. 이미지 파일의 실제 데이터(**magic bytes**)를 읽어 JPEG 여부 확인.
2. JPEG인 경우 EXIF 메타데이터에서 `Make`(제조사)·`Model`(모델) 정보가 있으면 **차단**.
3. 파일명이나 확장자가 아닌 **실제 파일 내용**으로 판별하므로 확장자 변경으로 우회 불가.
4. PNG·WebP·GIF 등 비-JPEG 포맷은 EXIF 구조가 없으므로 차단 대상 **아님**.

### 11.2 차단 시 동작

- 업로드 화면(USR-UPL-02)에 인라인 안내("Artier는 디지털 드로잉 작품 전용 플랫폼입니다") 배너 표시.
- 파일은 추가되지 않음(업로드 큐 진입 차단).

### 11.3 예외

현재 운영 기준으론 예외 없음. "필름 스캔"·"아트워크 촬영" 같은 정당한 사용 사례가 다수 접수되면 §20 확정 수치 재검토.

## 12. 신고·모더레이션 정책 (Phase 1)

Phase 1은 **작품 단위 모더레이션만** 다룬다. 사용자 계정 차원 제재(주의·시한부 정지·영구 정지·경고 카운터·허위 신고 카운터)는 **Phase 2**에서 재설계한다(§12.3 참조).

### 12.1 신고 처리 액션 (어드민)

운영팀은 접수된 신고에 대해 아래 3가지 중 하나를 선택한다(ADM-RPT-01).

| 액션 | 효과 | 상태 변경 |
|---|---|---|
| **삭제** | 해당 전시 영구 삭제 | `adminStatus: deleted` · 확인 다이얼로그(복구 불가) 필수 |
| **기각** | 신고를 부당하다 판정. 자동 비공개 상태였다면 **즉시 복원**. | `adminStatus: dismissed` |
| **비공개 유지** | 전시를 둘러보기·검색에서 제외. 작가 본인 프로필엔 보임. 작가는 수정해 재검수를 요청할 수 있다. | `adminStatus: hidden` |
| _(제거)_ | 액션 없이 큐에서만 제거(레거시 호환) | — |

> 좋아요·저장·Pick·신고·배지는 모두 **전시(컨테이너) 단위**. 개별 이미지(작품 1장) 단위의 인터랙션·배지는 없다.

### 12.1.3 신고 처리 알림 전송 (왕복 피드백)

자동 비공개 발동 시, 그리고 운영팀이 액션을 수행할 때 영향받는 사용자에게 시스템 알림이 생성된다. 작가·신고자가 적용된 조치를 인지하고 필요 시 대응(수정 등)할 수 있게 하기 위함.

| 이벤트 | 알림 수신자 | 알림 문구(요약) |
|---|---|---|
| **2회 자동 비공개** (§12.2) | 작품 작가 | "회원님의 전시 '{title}'이 신고 누적으로 자동 비공개 처리되었습니다. 운영팀 검토 대기 중." |
| **삭제** | 작품 작가 | "회원님의 전시 '{title}'이 신고 처리로 삭제되었습니다." |
| **기각 → 복원** | 작품 작가 | "'{title}' 전시가 검토 결과 정상 복원되었습니다." |
| **기각** | 신고자 | "접수하신 신고가 운영팀 검토 결과 기각되었습니다." |
| **비공개 유지** | 작품 작가 | "회원님의 전시 '{title}'이 비공개 처리되었습니다. 수정해 재검수를 요청할 수 있어요." |

- 알림 `type`: 모두 `system`
- 알림 센터(USR-NTF-01)에서 회색 배경 + 종 아이콘으로 표시
- 작품 관련 알림은 `workId` 포함(삭제는 예외 — 참조할 작품이 없음)

### 12.1.1 검수 반려 사유 이력 보존

별도 문맥으로, **콘텐츠 검수(ADM-REV-02)의 반려 액션**은 전시에 두 필드를 갱신한다:

- `rejectionReason`(현재 반려 사유 단일 값): 현재 상태가 `rejected`일 때만 의미 있음. 상태가 rejected에서 벗어나면(pending·approved로) **초기화**된다.
- `rejectionHistory`(이력 배열): 반려할 때마다 `{reason, rejectedAt}`을 **append**. 작가가 수정 재발행해도 **보존**된다.

**보존 목적**
- 운영 감사 — 과거 반려 판단 근거 추적
- 재범 추적 — 같은 작가·작품의 반복 반려 패턴
- 사유 분포 통계 — 서비스 정책 개선

**UI 노출 원칙**
- 사용자 앱 반려 사유 모달(USR-PRF-12): **현재 반려 사유만** 표시. 과거 이력 노출하지 않는다(사용자 혼란 방지).
- 어드민 작품 관리(ADM-WRK-01)·검수(ADM-REV-02): 필요 시 이력 요약(누적 N회 반려)·전체 목록 열람 가능.

---

### 12.1.2 검수 상태별 수정·삭제 정책

작가가 자기 전시에 대해 수정·삭제하는 것은 **모든 검수 상태에서 원칙적으로 허용**한다. 자기 콘텐츠 제어 + 개인정보 보호(즉시 내릴 권리) 원칙 준수.

**핵심 규칙: 상태 전이는 이미지 변경 여부로만 결정**

| 기존 상태 | 이미지 변경 (추가·삭제·교체·작가 재지정·커스텀 커버 변경) | 메타만 변경 (전시명·그룹명·작품명·커버 인덱스·이벤트 연결) |
|---|---|---|
| `approved` (공개 중) | → `pending` (재검수) | **`approved` 유지** (즉시 반영) |
| `pending` (검수 중) | `pending` 유지 (운영자가 최신판 검수) | `pending` 유지 (최신판 반영) |
| `rejected` (반려) | → `pending` (재검수) | **`rejected` 유지** (반려 사유 미해소) |

**근거**
- 검수 반려 사유 4종(저품질·스팸·부적절·저작권)은 모두 **이미지 기반** 판정. 메타데이터는 비속어 필터가 저장 시점에 처리하므로 운영자 재검토 불필요.
- 시니어 작가는 오탈자·실수 재고 가능성 높음. 메타 오탈자 때문에 24시간 lockout은 UX 부담 과다.
- 삭제까지 막으면 개인정보보호법 관점에서 문제.

**수정 진입 경고 (pending 중 수정)**
- `pending` 상태 전시의 점점점 "수정" 진입 시 확인 모달:
  - 제목: "검수 중인 전시를 수정할까요?"
  - 설명: "지금 수정하면 검수 대기에 변경 사항이 반영되고, 검수 대기 시간이 다시 시작됩니다."
  - [취소] / [수정하기]
- `rejected`는 반려 사유 모달이 이미 맥락을 제공하므로 추가 경고 없음.

**저장 후 토스트 (5종, 사용자에게 결과 명확히 고지)**

| 시나리오 | 토스트 |
|---|---|
| `approved` 유지 (메타만 변경) | "수정 사항이 바로 반영되었어요." |
| `approved` → `pending` (이미지 변경) | "이미지가 바뀌어 잠시 검수 대기예요. 공개되면 알려드릴게요." |
| `pending` 유지 | "수정 사항이 검수 대기에 반영됐어요." |
| `rejected` → `pending` (이미지 변경) | "수정한 작품이 재검수 대기에 들어갔어요." |
| `rejected` 유지 (메타만 변경) | "수정 사항이 반영됐어요. 공개되려면 반려 사유를 해결해주세요." |

**탭별 편집 범위**

| 탭 | 편집 범위 | 상태 전이 가능성 |
|---|---|---|
| **전시 탭** (USR-PRF-05) | 이미지·작가·메타 전부 | 위 매트릭스 전체 적용 |
| **내 작품 탭** (USR-PRF-06) | **작품명만** (인라인 편집) | **상태 변경 없음** (메타만이므로 원본 상태 유지) |
| **수강생 작품 탭** (USR-PRF-07) | 전시 탭과 동일 | 위 매트릭스 전체 적용 |

"내 작품" 탭은 이미지 변경을 지원하지 않는 구조라 이 정책과 자연스럽게 일치한다. 전시 탭·수강생 작품 탭은 같은 편집 경로(`/upload?edit=<id>`)를 공유하므로 동일 규칙 자동 적용.

**삭제 (모든 상태 공통)**
- 확인 다이얼로그(복구 불가 스타일) + 비회원 초대 대기 시 추가 경고 문구 "초대된 비회원이 있습니다. 삭제하면 초대가 무효화됩니다".
- 검수 상태와 무관하게 자유 허용. 24시간 내 삭제 시 AP -20 회수.

**운영자 race 처리 (런칭 전 보강)**
- 운영자가 승인·반려 버튼을 누르는 순간 작가가 이미 삭제한 경우 → 현재 no-op.
- **백엔드 연동 시** "이미 처리되었거나 삭제된 전시입니다" 에러 토스트 + 큐 자동 갱신을 구현.

### 12.2 2회 신고 자동 비공개

- 같은 전시에 **2번째 신고가 접수되는 순간**, 그 전시는 즉시 자동 비공개 처리된다.
- 자동 비공개 효과 = 운영팀 확정 비공개(`adminStatus: hidden`)와 동일. 둘러보기·검색·피드에서 제외, 작가 본인 프로필에서만 보임.
- 작가에게 시스템 알림 발송(§12.1.3).
- 이후 운영팀이 신고 큐에서 §12.1의 3가지 액션(삭제·기각·비공개 유지) 중 하나를 선택해 최종 처리한다.
- 운영팀이 **기각**으로 판정하면 해당 전시는 즉시 복원된다.
- 자동 비공개 상태에서도 작가는 §12.1.2에 따라 본인 전시를 수정할 수 있다. 수정 시 이미지 변경 여부에 따라 `feedReviewStatus` 전이(재검수)가 발생하지만, `adminStatus: hidden`은 운영팀 판정 전까지 유지된다.

### 12.3 Phase 1 범위 밖 (Phase 2 이관)

다음 항목은 Phase 2에서 재설계한다:

- 사용자 계정 단위 제재(주의·시한부 정지·영구 정지)
- 경고 카운터·허위 신고 카운터·자동 정지 승격
- 이의제기 절차·관찰 기간·선의 리셋·쿨다운
- 중대 위반(저작권 침해 확정·청소년 유해·정지 회피 등)에 대한 즉시 영구 정지

Phase 1은 **작품 단위 모더레이션**(§12.1·§12.2)과 **신고자 보호**(§12.4)로 한정한다.

### 12.4 신고자 보호

- 동일 `(신고자, 대상유형, 대상ID)` 조합당 **1회만** 허용.
- 신고한 전시·작가는 해당 **신고자에게만** 자동 숨김 처리.
- 숨김은 피드·검색·상세 모달에서 즉시 반영된다.

## 13. 업로드 유형·역할 정책

### 13.1 업로드 유형(2종)

업로드 화면의 첫 단계(USR-UPL-01)에서 **둘 중 하나 필수 선택**.

| 유형 | 설명 |
|---|---|
| **혼자 올리기** | 내 작품만으로 구성된 전시 |
| **함께 올리기** | 동호회·클래스·지인 등 여러 작가의 작품을 한 전시에 모음 |

### 13.2 함께 올리기의 역할(2종)

"함께 올리기"를 선택하면 이미지를 추가하기 전에 **본인 포함 여부**를 먼저 결정.

| 역할 | 본인 작품 포함 | 해당 사례 |
|---|---|---|
| **참가자** | **최소 1점 필수** | 동호회·클래스·지인 등 본인도 참여하는 그룹 전시 |
| **강사** | **포함 금지** | 수강생 작품만 대신 올리는 경우 |

### 13.3 역할별 강제 규칙

**참가자 (강사 OFF)**
- 본인 작품 없이 전시 등록 **차단**. 에러 메시지: "본인의 작품이 1점 이상 포함되어야 합니다".
- 검증 시점: 발행 핸들러에서 참여 작가 미지정 검증 직후.

**강사 (강사 ON)**
- 본인 작품 포함 **금지**. 작가 선택 리스트에서 본인 자동 제외.
- 강사 ON 전환 시 본인으로 지정된 이미지 슬롯은 **자동 초기화**.
- 업로드 화면 상단에 **"그룹 전시(강사)" 뱃지** 표시.

### 13.4 강사 표시 파생

- 강사 역할로 업로드한 이력이 **1건 이상**이면 프로필(USR-PRF-01)에 "수강생 작품" 탭이 **자동 노출**된다.
- 강사 여부 수동 토글 UI 없음 — 상태 불일치 방지 목적.
- 해당 작품 **모두 삭제 시** 탭 자동 제거.

### 13.5 금지(회귀 방지)

- 사용자 프로필에 강사 여부를 표시하는 **독립 플래그·토글 재도입 금지**.
- "강사 공개 목록" 류의 별도 저장소 재생성 금지(2026-04-13 단일화 완료).

## 14. 커버 이미지 정책

- 업로드 시 **커스텀 커버 이미지**를 별도로 지정할 수 있다.
- 지정된 커버는 **전시의 첫 장 표지** 및 **둘러보기 피드 썸네일**로 사용된다.
- 커버 이미지는 전시 이미지 배열에 **포함되지 않는다** → 이미지 개수 배지(예: 1/10)에 영향 없음.
- 커버 미지정 시: 전시 이미지 중 대표 이미지(없으면 첫 이미지)가 썸네일·표지로 사용.
- 전시 상세 모달에서 **커버 슬라이드는 블랙 배경 + 전시명 라벨**로 표시된다. 작가 정보 오버레이는 커버 위엔 노출하지 않는다.

## 15. 큐레이션·배지 정책

### 15.1 큐레이션 개념 4종

Artier의 큐레이션은 목적·수명·공개 여부가 다른 4가지 개념으로 구분한다.

| # | 개념 | 설명 | 공개 | 배지 |
|---|---|---|---|---|
| 1 | **Artier's Pick** | 운영팀이 매주 선정하는 추천 전시 (최대 10개, 교체) | 공개 | **영구** |
| 2 | **기획전** | 운영팀이 기존 전시를 주제·맥락으로 엮어 내보내는 컬렉션 (그룹전·작가전·특집전 포괄) | 공개 | 없음 |
| 3 | **이벤트** | 운영팀이 주제를 제시하고 사용자가 응모 → 운영팀이 선정. 주간 공모형 테마전도 이벤트의 한 종류 | 공개 | **선정작에만 영구** |
| 4 | **운영 내부 컬렉션** | 좋은 전시·작가·그룹을 운영팀이 내부 북마크 (발굴·추적 용도) | **비공개** | 없음 |

### 15.2 배지 체계 (공개)

공개 배지는 **3종뿐**이다. 응모작에는 배지 없음.

| 배지 | 발생 | 수명 |
|---|---|---|
| ★ Artier's Pick | Pick 선정 이력 (한 번이라도 받았으면) | **영구** |
| 이벤트 선정작 | 이벤트·공모형 테마전에서 선정된 이력 | **영구** |
| 강사 | 강사 역할로 업로드된 전시 | §13.4 발행 정책으로 결정 |

본인 전용 상태 배지(검수 "확인 중"·"게시 불가")는 위와 별개. 타인에게 보이지 않는다.

**배너 라벨은 제공하지 않는다.** NEW/HOT/EVENT/PICK 같은 부속 라벨을 두지 않는다.

### 15.3 Pick 운영 기준

- **상한**: 최대 10개. 주 단위로 교체(현재 운영 수동).
- **영구 배지**: 해제 시 현재 Pick만 해제, **한 번이라도 받은 이력 배지는 영구 유지**.
- **자동 초기화**: 현재는 운영팀 수동. **서버 스케줄러 연동 후** "매주 월요일 자정" 자동화.
- **20일 중복 제한**: 현재 미구현(운영 수동). **서버 검증 연동 후** 자동화.

**Pick 영구 배지 인플레이션 모니터링**
주간 10개 기준 연간 최대 500개가 영구 Pick 배지를 받는다. 2~3년 운영 시 배지 의미 희석 우려 → **분기별 누적 수치 점검**, 필요 시 "연도별 Pick" 구분자 도입 또는 선정 기준 재조정.

### 15.4 Pick vs 기획전 경계

| 구분 | Pick | 기획전 |
|---|---|---|
| 단위 | **1개 전시씩 독립 선정** | **3개 이상을 주제로 묶어 소개** |
| 배지 | 영구 부여 | 없음 |
| 교체 | 주간 | 자유(운영 판단) |

경계를 넘어서는 기획(예: "이번 주 봄 풍경 10선")은 **기획전**으로 운영. 운영자 간 일관성이 핵심.

### 15.5 이벤트 선정 시스템 (Phase 1 후순위 TODO)

배지 원칙은 확정, 구현은 후순위. 구현 시 포함해야 할 것:

- 어드민 이벤트 상세에 "선정 작품 관리" 섹션 추가(응모작 → 선정 체크)
- 공모형 주간 테마전도 이벤트의 한 종류로 관리(별도 스토어 대신 이벤트 스토어로 통합)
- **응모**(이벤트 연결)와 **선정**(배지 부여)을 개념·데이터 상 명확히 구분
- 선정 시점에 전시에 영구 배지 부여, 이벤트 종료 후에도 유지

Phase 2 이벤트 브랜드 정착 시(예: "Artier Awards") 배지에 `eventId` 연결 정보를 포함해 두어 나중에 이벤트별 라벨 세분화 가능성 확보.

### 15.6 Pick 선정 알림 전송 (왕복 피드백)

운영팀이 ADM-PCK-01에서 작품을 Pick에 추가하면 대상 작가에게 시스템 알림이 발송된다.

- **문구**: "회원님의 전시 '{title}'이 Artier's Pick으로 선정되었어요. 축하드려요!"
- **타입**: `pick` — 알림 센터에서 주황 배경 + 별 아이콘
- **workId 포함**: 알림 탭 시 해당 전시 상세로 이동

Pick 해제 시에는 알림을 발송하지 않는다(주간 교체 특성상 해제는 일상적 운영). 영구 이력 배지는 유지되므로 작가는 "한 번이라도 받았다"는 사실만 기억하면 충분.

## 16. 피드 랭킹·검색 스코어링 정책

### 16.1 피드 랭킹 (둘러보기·USR-BRW-01)

**작품 점수 공식**

```
base = log1p(likes) × 2.2 + log1p(saves) × 3.2
following_bonus = 3.5  (현재 사용자가 작가를 팔로우 중일 때)
total = base + following_bonus
```

**소스 버킷별 부스트**

| 소스 버킷 | 부스트 | 노이즈 범위 |
|---|---|---|
| Pick (Artier's Pick) | +14 | ±5 |
| Theme (주간 테마전) | +8 | ±6 |
| Featured (추천 작가) | +6 | ±6.5 |
| Personalized (팔로잉) | +7 | ±7.5 |
| Recent (14일 이내) | +3 | ±7.5 |
| Rest (일반) | 0 | ±10 |

**인터리빙 패턴**

```
pick → rest → theme → rest → personalized → rest → featured → rest → recent → rest
```

(최대 2사이클 반복 후 끝)

**핵심 제약**
- 같은 작품은 **하나의 버킷에만 노출**(중복 차단).
- 이미 본 작품은 각 버킷 **하단**으로 밀림.
- 댓글 수는 점수에 반영하지 않음(Phase 1 범위 밖. 댓글 기능 자체가 PRD Out of Scope).

### 16.2 피드 공개 규칙

- **승인(approved)** 상태의 전시만 공개 피드·검색·큐레이션에 노출.
- 검수 대기(pending) / 반려(rejected) → **본인 프로필에서만 보임**.
- 비공개(hidden) → 본인 프로필에서만 보임. 둘러보기·검색·신고자 숨김 모두 비노출.

## 19. 접근성·시니어 친화 정책

### 19.1 필수 원칙

1. **터치 타깃 ≥ 44px** — 모든 인터랙티브 요소에 최소 44×44px.
2. **입력 폰트 ≥ 16px** — iOS Safari 자동 줌 방지.
3. **글꼴 크기 사용자 조절 3단** — 작게(100%) / 보통(110%) / 크게(120%).
4. **확인 다이얼로그 일원화** — 브라우저 네이티브 confirm·alert 금지. 공통 확인 다이얼로그(CM-01) 사용.
5. **색만으로 정보 전달 금지** — 상태(승인/반려/정지)는 아이콘 + 라벨 동반.
6. **호버는 데스크톱만** — 포인터 정밀·뷰포트 1024px 이상에서만 호버 효과. 모바일·태블릿에선 호버 효과 없음.
7. **애니메이션 짧고 느리게** — 200ms 내외. 번쩍이는 효과 금지.

### 19.2 금지된 UI 패턴

| 패턴 | 사유 |
|---|---|
| 스프레드시트형 UI | 시니어 인지 부하 과다 |
| 다중 선택(Shift + Click) | 오조작 위험 |
| 키보드 Tab 이동 전제 UX | 모바일·터치 우선 |
| `window.confirm()` · `window.alert()` 직접 호출 | 시각 일관성 + 커스터마이즈 불가 |
| `dangerouslySetInnerHTML` | XSS — 필요 시 sanitize 후 사용 |
| Pinch 줌 외 고난도 GestureAPI 의존 | 시니어 제스처 숙련도 편차 |

### 19.3 다국어

- UI 기본 한국어(KO), 영어(EN) 전환 가능.
- 모든 문자열은 다국어 훅 경유. **하드코딩 금지**.
- 언어 전환 반응성: 전환 시 즉시 리렌더가 트리거되도록 정적 스냅샷 함수를 렌더 시점에 직접 호출하지 말 것.

| 운영 판단 필요 케이스(법적 이슈·대외 민감) | 긴급 이슈로 승격 | ADM-ISU-01 |
| 시스템 장애·데이터 손실 의심 | 체크리스트 차단됨 처리 + 복구 절차 | ADM-CKL-01 |

### 22.6 운영 데이터 보관 기준

| 데이터 | 보관 | 파기 |
|---|---|---|
| 탈퇴 작가 식별 정보 | 재가입 방지 목적의 해시만 유지 | **런칭 전 백엔드 이관 시** 보관 기간 명문화 |

| 운영자 감사 로그 | **5년**(append-only) | 보존 기간 만료 시 배치 파기([PRD_Admin §0.6](PRD_Admin_v1.md#06-감사-로그-audit-trail)) |
| 초대 발송 로그 | 최대 300건 순환 | 자동 오래된 항목 제거 |
| 매칭 로그 | 최대 200건 순환 | 자동 오래된 항목 제거 |
| 포인트 원장 | 최대 500건 순환 | 자동 오래된 항목 제거 |

### 22.7 운영자 감사 로그 원칙

모든 어드민 액션(검수 판정·신고 4액션·제재·큐레이션·배너·이벤트·공지·문의)은 `admin_audit_log`에 즉시 기록된다. 스키마·보존·조회 권한은 [PRD_Admin §0.6](PRD_Admin_v1.md#06-감사-로그-audit-trail) 단일 소스. 요약:

- **Append-only**: 수정·삭제 API 제공하지 않음. 운영자 본인도 자기 로그를 삭제 불가.
- **행위자 식별**: `actorId`·`actorRole`·`ip`·`userAgent`로 누가 언제 어디서 했는지 재구성 가능.
- **복원 기반**: 복구 불가 액션은 `targetSnapshot`에 실행 직전 필드값을 보관해 사후 복원 근거 제공.
- **조회 권한**: Phase 1은 Operator 전체. Phase 2는 **최고 운영자만** 열람 가능.
- **개인정보 마스킹**: `targetSnapshot` 포함 개인정보는 [PRD_Admin §0.4.3](PRD_Admin_v1.md#043-뷰어-read-원칙) 마스킹 규칙을 적용해 저장.

### 23.1 강사 표시 파생

- **파생식**: `isInstructor(user) = ∃ work ∈ works(user), work.isInstructorUpload === true`
- **저장 안 함** — 업로드 이력에서 계산. 별도 프로필 플래그·토글 없음(§17.1, CLAUDE.md 강사 단일화 정책).
- **영향**:
  - 프로필(USR-PRF-01)에 "수강생 작품" 탭(USR-PRF-07) 자동 노출.
  - 모든 해당 작품 삭제 시 자동 비노출.
  - 강사 ON 업로드는 **본인 작품 포함 금지**, 강사 OFF 그룹 업로드는 **본인 작품 최소 1점 포함 필수**(§13).

> **Phase 2 통합 검토**: 전시는 현재 `feedReviewStatus` · `adminStatus` · `isHidden` 3개 상태를 동시에 갖는 구조다. "공개 여부" 조합 판정이 복잡하여 Phase 2에서 단일 `visibility` enum으로 통합 예정([SystemArch §10 N-11](SystemArchitecture_v1.md#10-아직-열려-있는-항목)). Phase 1은 현 구조 유지.

### 23.2 검수(feedReviewStatus) 상태 전환

| 트리거 | 전환 |
|---|---|
| 신규 업로드 | `undefined | null` → `pending` (auto-approve 환경에선 즉시 `approved`) |
| 운영팀 승인 | `pending` → `approved` + 팔로워 알림 발송 + **보류된 비회원 초대 발송**(§3.1) |
| 운영팀 반려 | `pending` → `rejected` + 반려 사유 1종 필수 기록 |
| 편집 시 이미지 계열 변경 (`image` 배열·`imageArtists`·`customCoverUrl`) | 원본 상태 무시하고 `pending` 재설정 → 재검수 |
| 편집 시 메타만 변경 (전시명·그룹명·작품명·커버 인덱스·이벤트 연결) | 원본 상태 유지 |
| 반려 후 수정·재제출 | `rejected` → `pending` (재검수) |

### 23.3 `imageArtists[i]` 슬롯 상태 전환

| 슬롯 상태 | 조건 |
|---|---|
| `{ type: 'member', memberId, memberName, memberAvatar }` | 업로더가 회원을 직접 지정, 또는 자동 연결 승격 결과 |
| `{ type: 'non-member', displayName, phoneNumber? }` | 비회원 초대. 검수 승인 후 `phoneNumber`는 스크럽(§3.4) |
| `{ type: 'unknown' }` | disavow(마이페이지 "본인 작품 아님") 결과(§3.5). 번호·이름 제거 |

- **전환**: `non-member` → `member` (가입 시 자동 연결) / `member` → `unknown` (피초대자 disavow).
- **역전환 없음**: `unknown` → 자동 재연결 안 됨. 업로더가 전시 편집에서 수동으로 member/non-member를 재지정해야 복구.

### 23.4 자동 비공개 (작품 단위)

- 트리거: 같은 전시에 **2번째 신고**가 접수되는 순간(§12.2).
- 동작: `adminStatus`를 `hidden`으로 즉시 설정 → 둘러보기·검색·피드 비노출. 작가 본인 프로필에서만 보임.
- 알림: 작가에게 시스템 알림 1건 발송(§12.1.3).
- 해제: 운영팀 큐에서 "기각" 판정 시 자동 해제(복원). 기타 액션(삭제·비공개 유지)은 해당 상태로 확정.

### 23.5 이미지 처리 파이프라인

업로드 파일 1장당 처리 순서:

```
1. 파일 크기 검증 — 10MB 초과 시 거부 (§20)
2. MIME 검증 — JPEG/PNG/WebP/GIF 외 거부
3. 카메라 사진 차단 검증 — JPEG 매직바이트 + EXIF Make/Model 존재 시 거부 (§11)
4. WebP 변환 — 브라우저 지원 시 WebP Data URL로 재인코딩, 실패 시 원본 Data URL 보관
5. 단변 해상도 검증 — min(width, height) < 800 거부 (§20)
6. 큐 적재 — 이미지 수 상한 10장 초과 시 신규 선택 차단
```

**핵심 규칙**
- 위 순서를 바꾸면 의미가 달라진다. 예: WebP 변환 전에 EXIF 검증해야 카메라 사진 탐지 가능.
- 프로필 이미지는 별도 경로로 5MB / 원형 크롭.
- 커버 이미지: 원본 이미지를 **blur + scale + opacity**로 깔아 CSS만으로 톤 배경 구현 — dominant-color 추출 알고리즘 **미사용**.

### 23.6 그룹명·자동완성 파생

- **사용자 최근 그룹명**: 본인 업로드 이력에서 추출. 상한 50개 (§20).
- **중복 허용**: 2026-04-17 이후 그룹명 정규화·캐논 맵 폐기. 동일 문자열은 동일 그룹으로 간주하지 않음 — 표시 문자열만 사용.
- **자동완성 순서**: 빈 쿼리 시 내 최근 → 전체 작품 그룹명 (상한 30). 쿼리 있을 시 상한 20.

### 23.7 포인트 일일 상한·월 상한

| 이벤트 | 상한 | 재설정 기준 |
|---|---|---|
| 일반 업로드 +20 | 일 **2회** | 로컬 자정 |
| 일일 둘러보기 +5 | 일 **1회** | 로컬 자정 |
| 그룹 전시 개설(강사) +30 | 월 **1회** (강사-월 조합) | 월 1일 |
| 그룹 참여(비강사) +15 | 월 **5회** | 월 1일 |
| 업로드 AP 회수 -20 | 단건 | 업로드 후 24시간 이내 삭제 |

- 첫 업로드 +100은 일 상한 **무관**(당일 "일반 업로드 +20" 카운터에 포함되지 않음).
- 포인트 원장 FIFO 500건 유지.

### 23.8 검색 쿼리 전처리

- 쿼리 **trim** 후 공백 1개로 치환(연속 공백 합침).
- 대소문자 비구분 매칭(현지화 locale 별 case folding).
- 숫자만 입력된 쿼리도 정상 처리(ID/전화번호 검색 아님 — 콘텐츠 필드 대상).
- 빈 쿼리·1자 이하는 자동완성만 표시, 결과 페이지 이동 금지.

## 24. 인터랙션 정책 (좋아요·저장·팔로우)

### 24.1 단위

- 좋아요·저장은 **전시(Work) 단위**. 개별 이미지(piece) 단위 인터랙션은 Phase 2.
- 팔로우는 **작가(Artist) 단위**.

### 24.2 공통 제약

| 조건 | 동작 |
|---|---|
| 비로그인 | CM-02(로그인 유도) 표시 후 로그인 완료 시 원래 의도된 액션으로 이어감 |
| 탈퇴 작가 (§4.2) | 좋아요·저장·팔로우·신고 모두 **no-op**(silent) + UI 비활성(opacity 40%, pointer-events-none) |
| 자동·확정 비공개 전시 (§12.2, `isHidden: true`) | 본인 프로필 외에선 카드 자체가 노출되지 않아 인터랙션 경로 없음 |
| 본인이 신고한 대상 | 본인에게만 피드·검색·상세 모달에서 자동 숨김(§12.4) |

### 24.3 좋아요

- 단일 토글(좋아요/해제).
- **본인 작품 좋아요**는 기능상 허용되나 프로필 "좋아요" 탭(USR-PRF-08)에 노출되지 않음.
- 발생 시 작품 작가에게 시스템 알림 1건(탈퇴 작가 제외).
- **카운트 UI 비노출** — 전시 카드·상세 모달에 **상태 아이콘만** 표시. 내부 집계값(`Work.likes`)은 §16.1 피드 랭킹에 사용.

### 24.4 저장

- 좋아요와 동일 구조의 토글.
- **작가 알림 없음**(저장은 개인 북마크 성격).
- "저장" 탭(USR-PRF-09)은 본인만 열람. 본인 작품 저장은 허용되나 탭에서 숨김.
- 내부 집계값(`Work.saves`)은 §16.1 피드 랭킹에 사용.

### 24.5 팔로우

- **자기 팔로우 금지**: 본인 프로필에서 팔로우 버튼 자체 미노출.
- **탈퇴 작가 팔로우 금지**: 버튼 숨김 + 저장소 레벨 no-op(defense-in-depth).
- 발생 시 대상 작가에게 시스템 알림 1건.
- 팔로워 마일스톤(10·50·100명) 달성 시 **본인**에게 AP 적립(§7.2).
- 팔로우는 피드 **개인화(Personalized)** 버킷 가중치로 사용(§16.1).
- 팔로잉·팔로워 숫자는 프로필 헤더(USR-PRF-01)에만 노출. 탭 시 USR-PRF-04 모달.

### 24.6 데이터

- `artier_interactions`: `{ liked: string[], saved: string[] }` 배열 (§18).
- `artier_follows`: 팔로우한 작가 ID 배열.
- 브라우저 간 동기화 없음(Phase 1 로컬). 런칭 전 백엔드 이관 후 계정 단위 동기화.

### 24.7 Phase 2 유보

- 댓글·핀 코멘트·익명 인터랙션·리액션.
- 개별 piece 단위 좋아요·저장·북마크.

## 25. 이벤트·응모 정책

### 25.1 이벤트 생성·관리

- 이벤트는 **운영팀만** 어드민(ADM-EVT-01·02)에서 생성·수정·삭제한다. 사용자 직접 생성 불가.
- 필수 필드: 제목·설명·기간(`startAt`·`endAt`)·배너 이미지. 선택: 응모 규칙·상품·주최자.
- 공개 토글(`worksPublic`): 응모작을 이벤트 페이지에서 **공개**할지 여부. 기본 `false`(비공개 응모). 선정 후 공개 시 `true`.

### 25.2 응모 조건

- 사용자는 본인 전시를 업로드·편집 중에 **하나의 이벤트에만** 응모 가능(`Work.linkedEventId`).
- 응모 시점에 이벤트가 **진행 중**(`now >= startAt && now <= endAt`)이어야 한다. 종료된 이벤트는 선택 불가.
- 응모한 전시는 **검수 승인(§12.1.2)** 이후 이벤트 목록에 노출(`worksPublic: true`일 때).
- 동일 전시의 응모 이벤트 변경은 **언제든 가능**. `linkedEventId`는 메타 필드이므로 §23.2 규칙상 재검수는 **트리거되지 않는다**. 즉 승인된 전시의 이벤트 연결만 바꿔도 즉시 새 이벤트 목록에 반영된다.

### 25.3 응모작 노출

- `worksPublic: true` 이벤트: 응모작이 이벤트 상세 페이지(USR-EVT-02)에 공개. 검수 승인(`approved`) + `isHidden: false` 조건을 모두 만족해야 노출.
- `worksPublic: false` 이벤트: 응모작은 운영팀만 어드민(ADM-EVT-03)에서 열람.

### 25.4 선정 시스템 (Phase 2)

- 당선작 자동 선정·투표·추천 작품 기능은 **Phase 2**. Phase 1은 응모 수집만 한다.
- Phase 1에서 운영팀은 ADM-EVT-03 응모자 현황을 확인하고, 필요 시 Pick(§15) 선정·기획전 구성으로 외부화한다.

### 25.5 이벤트 종료

- `endAt` 경과 시 이벤트는 자동으로 "종료" 상태 표시. 신규 응모 차단.
- 기존 응모작은 계속 목록에 노출(`worksPublic` 유지). 운영팀이 수동으로 비공개 전환 가능.

### 25.6 이벤트 삭제

- 운영팀 수동 삭제 시 복구 불가 확인 다이얼로그.
- 응모된 전시의 `linkedEventId`는 null로 일괄 정리(전시 자체는 유지).
- 삭제 사실은 감사 로그에 기록.

## 26. 이미지 저작권 보호 정책

Artier는 작가의 원본 작품을 보호한다. 완벽한 다운로드 차단은 불가능하지만, **우연한·일상적 복제**는 차단한다.

### 26.1 프런트 차단 (모든 플랫폼)

| 대상 | 차단 방식 |
|---|---|
| 우클릭 메뉴 | `oncontextmenu` 이벤트 차단(`preventDefault`) |
| 드래그·드롭 | `ondragstart` 차단. "이미지를 드래그할 수 없습니다" 조용한 실패(에러 토스트 없음) |
| 길게 눌러 저장(모바일) | `-webkit-touch-callout: none` + `user-select: none` CSS |
| 이미지 선택·복사 | `user-select: none` |

구현 경로: 모든 작품 이미지 노출은 **전용 이미지 컴포넌트** (§SystemArch §13.5 `CopyrightProtectedImage`) 하나를 경유한다.

### 26.2 피드·상세 이미지 해상도 제한

- 피드·카드: 썸네일은 **단변 최대 1200px** 제공. 원본 해상도(최대 수천 px) 직접 노출 금지.
- 상세 모달: 상세 뷰어는 원본 해상도 표시 가능(줌 뷰어 포함). 단, **다운로드 경로 차단**은 §26.1을 유지.

### 26.3 URL 노출 최소화

- 이미지 URL은 화면에 노출하지 않는다(개발자 도구로만 확인 가능).
- URL 끝의 원본 파일명은 난수 ID 기반이며 원본 파일명을 포함하지 않는다.

### 26.4 워터마크

- Phase 1은 워터마크 **미적용**(시니어 작가의 작품 감상 경험 우선).
- Phase 2에서 선택 가능한 "내 작품에 워터마크 적용" 토글 검토.

### 26.5 저작권 신고 대응

- 저작권 침해 신고 사유(§12.1 "저작권")가 접수되면 우선 **자동 비공개**(§12.2)에 따라 2회 신고 시 즉시 노출 차단.
- 운영팀은 저작권 신고를 **72시간 내** 우선 처리(§22.2 신고 SLA).
- 저작권자 본인이 아닌 제3자 신고도 접수 가능하되, **운영팀 검토 후**에만 확정 조치.

### 26.6 허용되는 복제

- **스크린샷**은 기술적으로 차단 불가. 서비스 이용약관에서 **"개인 감상 목적 외 재배포 금지"**를 명시(§21 LP-7 참조).
- 작가 본인은 본인 작품을 자유롭게 내보낼 수 있어야 함. Phase 2에서 "내 작품 원본 내보내기" 기능 검토.

### 26.7 API·크롤링 대응 (Phase 2)

- 검색 엔진 크롤러는 `robots.txt`로 이미지 경로 분리 고려(§11 SEO 연동).
- 일반 스크래퍼 차단은 Phase 2 백엔드 연동 후 rate-limit·봇 탐지로 대응.

## 27. 계정 정보 편집·비밀번호 재설정

### 27.1 프로필 편집 (USR-PRF-02)

사용자는 본인 프로필에서 다음 필드를 언제든 편집한다:

| 필드 | 편집 가능 | 제한 |
|---|---|---|
| 닉네임 | **O** | 2~20자, 비속어 불가 |
| 한 줄 소개(headline) | **O** | 20자 |
| 바이오 | **O** | 200자 |
| 위치 | **O** | 11개 국가/지역 + 기타 드롭다운 |
| 관심사 칩 | **O** | 15개 중 선택 |
| 외부 링크 | **O** | 6개 플랫폼 |
| 프로필 사진 | **O** | 5MB, 원형 크롭 |

### 27.2 고정 필드 (Phase 1)

다음 필드는 **본인 정보**의 핵심으로 편집 불가:

| 필드 | 이유 |
|---|---|
| **실명** | PASS 본인인증 결과(§2.1). 변경 시 본인인증 재수행 필요 — Phase 2 |
| **전화번호** | PASS 본인인증 결과 + 초대 매칭 키(§3.5). 변경 시 재인증 필요 — Phase 2 |
| **생년월일** | 만 14세 검증 근거(§2.2) |
| **이메일** (이메일 가입자) | 로그인 ID. 변경 플로우 + 인증 메일 — Phase 2 |

> Phase 2에서 본인인증 재수행 후 실명·전화 변경, 이메일 변경 플로우 제공 예정.

### 27.3 비밀번호 재설정 (이메일 가입자만)

**플로우** (USR-AUT-07 · Login.tsx)

1. 로그인 화면 "비밀번호를 잊으셨나요?" → 이메일 입력 폼
2. 등록된 이메일이면 **재설정 링크 발송**. 등록 여부는 프라이버시상 회신에 공개하지 않음("등록된 이메일이면 곧 도착합니다")
3. 링크는 **1시간** 유효
4. 링크 클릭 → 새 비밀번호 입력(§2.1 비밀번호 규칙 동일) → 완료
5. 변경 완료 시 사용자에게 **확인 이메일** 1건 발송("비밀번호가 변경되었습니다. 본인이 아니면 즉시 재설정하세요")

**Phase 1 구현 상태**

- 이메일 발송은 **모의**(콘솔 로그 + 발송 로그만). 런칭 전 실 SMTP 연동.
- 데모 QA용 링크: `/login?demo=reset-expired` / `/login?demo=reset-newpw`

### 27.4 닉네임 변경 주기

- 변경 횟수·주기 제한 **없음**(Phase 1). 시니어 작가의 오탈자 수정·작명 변경 자유를 우선.
- 악용이 관찰되면 Phase 2에서 주기 제한 도입 검토.

## 28. 다국어·locale 정책

### 28.1 지원 언어

- **Phase 1**: 한국어(`ko`) · 영어(`en`) 2종.
- 전체 i18n 사전은 [`Copy_v1.md`](Copy_v1.md) §6 단일 소스.

### 28.2 초기 locale 결정

로그인 이전·이후 모두 아래 순서로 결정:

1. 사용자가 설정에서 명시적으로 선택한 값 → `artier_locale` 저장 (최우선)
2. 저장값 없으면 브라우저 `navigator.language`:
   - `ko*` 접두 → 한국어
   - 그 외 → 영어
3. 런타임 중 변경은 `useI18n().setLocale()`만 허용. 스냅샷 함수 직접 사용 금지([SystemArch §13.4](SystemArchitecture_v1.md)).

### 28.3 전환 UX

- 헤더 언어 토글 버튼 또는 설정(USR-STG-01)에서 전환.
- 전환 즉시 전체 화면 리렌더(한글↔영문 실시간 교체). 새로고침 불필요.

### 28.4 미번역 키 fallback

- 누락된 키는 **키 문자열 자체**를 반환(예: `upload.err.fileTooBig`).
- 빌드·배포 전 누락 없는지 자체 린트로 확인(ko/en 키 수 동일).

### 28.5 Phase 2 확장 후보

- 제3 언어(일본어·중국어) 추가
- 지역 기반 자동 locale(IP·`Accept-Language` 서버측 판정)
- 날짜·숫자 포매팅 라이브러리 도입(현재는 단순 `toLocaleString` 사용)

## 29. 쿠키·분석 동의 정책

### 29.1 쿠키 분류

| 카테고리 | 용도 | 동의 필요 |
|---|---|---|
| **필수** | 세션 토큰·로그인 유지·글꼴 크기 설정 | 불필요(서비스 제공 목적) |
| **분석** | GA4 이벤트·페이지 조회 | **필수**(사용자 명시 동의) |
| **마케팅** | — (Phase 1 미사용) | Phase 2에서 정의 |

### 29.2 동의 수집 (CM-05 배너)

- 최초 방문 시 **하단 고정 배너** 노출.
- 옵션: "모두 동의" / "필수만" / 개인정보처리방침 링크.
- 선택 결과는 `artier_cookie_consent` localStorage 키에 저장.
- 배너는 선택 완료 시 즉시 사라짐. 이후 설정(USR-STG-04 가정)에서 재조정 가능.

### 29.3 3자 처리자 목록 (Phase 1)

| 서비스 | 용도 | 동의 카테고리 |
|---|---|---|
| Google Analytics 4 (GA4) | 행동 분석 | 분석 |
| Netlify | 정적 호스팅·CDN | 필수 |
| (런칭 전 연동) Supabase | 데이터 저장 | 필수 |
| (런칭 전 연동) 카카오 알림톡 | 초대 발송 | 필수(본인 동의로 수집된 연락처 발송) |
| (런칭 전 연동) 이메일 발송 업체 | 알림·비밀번호 재설정 | 필수 |

### 29.4 동의 철회

- 설정에서 언제든 동의 철회 가능.
- 철회 시 분석 스크립트 즉시 unload + 향후 이벤트 발송 차단.
- 이미 수집된 분석 데이터는 GA4 보관 정책(14개월) 하에 자동 만료.

### 29.5 법적 근거

- 한국 개인정보보호법·정보통신망법 준수.
- GDPR 대상 사용자(EU)에 대해서는 분석 쿠키 기본 **비활성**, 명시 동의 후에만 활성(개별 처리).
- 상세 법무 검토 포인트는 [§21 LP-7·LP-8·LP-9](#21-법무-체크포인트) 참조.

## 1. 알림 채널 정책

채널은 **사용자가 어떤 연락처를 가지고 있는지**에 따라 자동으로 정해진다. 인앱 알림은 모든 회원에게 항상 발송된다.

### 1.1 채널 라우팅

| 사용자 | 일반 알림 | 법적 의무 알림 |
|---|---|---|
| 한국 전화번호 + 이메일 보유 | 카카오 알림톡 → 실패 시 SMS | 이메일 |
| 한국 외 전화번호 + 이메일 보유 | SMS | 이메일 |
| 이메일만 보유 | 이메일 | 이메일 |

- **인앱 알림**(§24)은 모든 회원에게 항상 발송. 위 외부 채널과 별개로 동작.
- **법적 의무 알림**(약관·개인정보처리방침 변경)은 모든 사용자에게 이메일로 발송 — 법적 기록 보존 목적.
- 한국 전화번호 보유자에게 일반 알림은 알림톡·SMS만 사용하고 이메일은 보내지 않는다(시니어 이메일 미수신 가정).
- 비카카오 가입자가 알림톡·SMS 채널을 켜려면 Settings에서 전화번호를 **추가**할 수 있다(§2.1, §27.2).

### 1.2 Phase 1 구현 상태

알림톡·SMS·이메일 발송은 Phase 1에서 모두 모의 동작이며, 런칭 전 실제 발송 채널을 연동한다(채널·공급사 선정은 개발팀 결정).

## 2. 가입·본인인증 정책

### 2.1 가입 옵션과 필수 수집 정보 (region 분기 폐기)

가입은 **모든 사용자에게 동일한 4가지 옵션**으로 진입한다. region 분기 없음.

| 가입 옵션 | 본인인증 효과 |
|---|---|
| **카카오** | 동의 시 전화번호·생년월일·이메일이 카카오 계정 정보로 자동 수집된다. |
| **구글** | 이메일이 자동 수집된다. 전화번호·생년월일은 수집하지 않음. |
| **애플** | 이메일이 자동 수집된다(릴레이 이메일 옵션 가능). 전화번호·생년월일은 수집하지 않음. |
| **이메일 인증 링크** | 이메일 입력 후 일회용 링크로 본인 인증(§2.5). |

**모든 사용자 공통 필수**:

| 항목 | 필수 | 비고 |
|---|---|---|
| 이메일 | 필수 | 4가지 옵션 모두 이메일 보유. 카카오 가입은 카카오 측 동의 항목 설정으로 이메일을 받아온다. |
| 닉네임 | 필수 | 2~20자, 비속어 불가. 소셜 가입은 받아온 정보를 자동 채워준 뒤 수정 가능. |
| 생년월일 | 필수 | 만 14세 검증(§2.2). 카카오는 자동, 그 외는 직접 입력. |
| 약관 동의 | 필수 | "만 14세 이상이며 가입에 동의" 자기 명시 포함(§2.2). |

**선택 수집** (가입 후 Settings에서 추가 가능):
- 전화번호 — 카카오 가입자는 자동 수집. 그 외 사용자는 Settings에서 추가하면 알림톡·SMS 알림과 전화번호 초대 매칭이 활성화된다.

**Phase 1 수집하지 않는 항목**:
- 실명 — 가입·온보딩·Settings 어느 단계에서도 입력 받지 않는다. 매칭은 전화번호 단독, 식별·표시는 닉네임으로 충분하므로 시니어에게 추가 입력 부담을 지우지 않는다. Phase 2에서 정산·세무 등 실제 용도가 생기면 그때 도입을 재검토한다.

### 2.1.1 거주 국가 — 자동 수집 안 함

가입 시점에 **사용자 거주 국가를 자동으로 알아내거나 저장하지 않는다**. 가입 화면도 한국·해외 분기 없이 모든 사용자에게 동일한 4가지 옵션(§2.1)을 노출한다.

- 알림 채널은 사용자가 가진 연락처에 따라 자동 결정(§1)
- 초대 매칭은 발신자가 입력한 연락처 일치 여부로 판정(§3.5)
- **작가 출신국 공개 표시**는 프로필 편집의 **위치 항목**(§27.1, 11개 국가/지역 + 기타에서 선택)을 통해 사용자가 직접 선택

법적 지역별 처리가 필요해질 경우 그 시점에 본 정책을 보강한다(현재 미정).

### 2.2 만 14세 검증

자가 입력 + 4단 정책 가드로 글로벌 표준에 부합하는 검증을 구성한다.

| 가드 | 내용 |
|---|---|
| **1. 가입 시 즉시 차단** | 생년월일 입력값으로 만 14세 미만 즉시 가입 차단. 시스템 검증. |
| **2. 약관 자기 명시** | 약관 동의 체크박스 옆 "만 14세 이상이며 가입에 동의합니다" 자기 명시. |
| **3. 거짓 입력 책임** | 약관 본문에 "허위 생년월일로 가입 시 즉시 계정 삭제 + 법적 책임" 명시. |
| **4. 사후 신고 채널** | 사용자·운영팀이 14세 미만 의심 신고 → 운영팀 검증 후 즉시 차단·삭제. ADM-MBR-02에 처리 액션. |

**카카오 가입자**는 카카오 계정에 이미 검증된 생년월일이 있어 신뢰도가 더 높다. 그 외 가입자는 직접 입력 + 위 4단 가드.

**만 14세 미만 가입은 예외 없이 차단**한다. 법정대리인 동의 기반 가입 경로는 도입하지 않는다.

**근거**: 글로벌 콘텐츠 플랫폼 표준(Instagram·TikTok·Twitter·Facebook 모두 자가 입력 + 정책 가드). 자가 입력 자체는 검증 못 하지만 시스템 차단 + 약관 + 사후 처리로 법적 면책 + 운영적 차단을 동시에 달성.

### 2.3 중복 가입 차단

- 이메일은 모든 사용자 필수 → **이메일 중복 검사** 1종이 기본.
- 전화번호 보유자(카카오 가입자·Settings에서 추가한 사용자)는 추가로 전화번호 중복 검사.
- 소셜 최초 가입 → 약관 동의 + 닉네임 수집 모달(USR-AUT-05) → 온보딩(USR-AUT-06).
- 온보딩은 가입 경로에서 이미 받은 정보를 **프리필**한다(사용자가 수정 가능).

| 가입 경로 | 프리필 대상 |
|---|---|
| 이메일 가입(USR-AUT-03) | 이메일 |
| 소셜 첫 가입(USR-AUT-05) | 닉네임 + 소셜 제공자 이메일 |
| SMS 초대 링크(USR-EXH-03) | 초대받은 전화번호·표시명 |

온보딩 종료 시 모든 프리필 임시 값은 일괄 정리된다.

### 2.4 세션·자동 로그인

- 로그인 세션은 토큰 형태로 보관. 현재는 모의 토큰(클라이언트 전용), **런칭 전** 실제 JWT + 백엔드 세션으로 교체.
- Phase 1엔 사용자 계정 차원 정지가 없다(§12.3). 관련 정지·차단·이의제기 플로우는 Phase 2에서 재설계.

### 2.5 이메일 인증 링크(매직 링크) — 가입·로그인

매직 링크는 §2.1 가입 4가지 옵션 중 **이메일 옵션 단독**의 인증 방식이다(카카오·구글·애플은 각 서비스 측에서 본인 인증을 처리한다). 비밀번호를 수집하지 않는다. 이메일 가입과 이메일 로그인 모두 **일회용 인증 링크를 발송 → 사용자가 링크를 열어 인증**하는 방식으로 동작한다. 시니어 타깃 특성상 비밀번호 분실·재설정이 이탈의 큰 원인이 되므로 Phase 1부터 경로 자체를 제거한다.

**공통 토큰 규칙**

- 토큰은 **일회용** (1회 소비 시 폐기), **30분** 유효.
- 같은 이메일로 새 토큰을 발급하면 기존 미사용 토큰은 즉시 무효화된다.
- 클릭 시 `/auth/verify?token=…`로 진입 → 서버 검증 후 세션 발급(로그인) 또는 가입 이어가기(신규).
- 다른 브라우저·기기에서 열어도 동작하지만, "같은 브라우저에서 열기" 안내를 화면에서 제공한다(세션 연속성 이슈 방지).

**가입 플로우 (USR-AUT-03)**

1. AuthSheet → "이메일로 가입하기" → 이메일 1회 입력 → **"인증 메일 보내기"**
2. 발송 완료 상태 화면: 스팸함 안내 · 30초 쿨다운이 걸린 **재전송** · **이메일 수정**
3. 사용자가 메일 링크 클릭 → `/auth/verify` → 이메일 인증 완료 상태로 **프로필 입력 단계(닉네임 + 생년월일)**로 이동
4. 프로필 → 약관 동의 → 가입 완료 → 온보딩(USR-AUT-06)

**로그인 플로우 (USR-AUT-02b)**

1. AuthSheet → "이메일로 로그인" → 이메일 1회 입력 → **"로그인 링크 보내기"**
2. 발송 완료 상태 화면 (가입과 동일한 재전송·수정 UI)
3. 사용자가 메일 링크 클릭 → `/auth/verify` → 세션 발급 → `redirectTo` 경로로 이동
4. 미가입 이메일도 동일 문구로 응대(계정 존재 여부 노출 금지, §2.5 프라이버시).

**계정 정지 처리** — 정지 상태의 이메일로 링크 발송 요청 시 발송하지 않고 즉시 정지 안내 메시지를 반환한다(§12.3).

**Phase 1 구현 상태**

- 이메일 발송은 **모의** — 매직 링크 스토어에 토큰 적재 + 발송 로그. 런칭 전 실제 이메일 발송 서비스(SMTP) 연동([§31 N-5](#31-런칭-전-미해결-항목)).
- QA 편의를 위해 발송 완료 화면에 "데모: 받은 링크 열기" 버튼을 노출(가장 최근 발행 토큰으로 바로 이동). 실 서비스 빌드에서 제거.

**발송 실패·지연 시 폴백 (런칭 전 필수 구현)**

실제 이메일 발송 서비스(SMTP) 연동 후 발송 장애·스팸 차단·오타로 링크가 사용자에게 도달하지 않을 가능성이 있다. 사용자가 경로에 **갇히지 않도록** 다음 폴백을 제공한다:

- **30초 쿨다운 재전송** — 발송 완료 화면의 "다시 보내기" 버튼(Phase 1 구현 완료). 연속 남용 방지는 서버 측 레이트 리밋으로(분당 3회, IP+이메일 조합).
- **이메일 주소 수정** — 발송 완료 화면에서 돌아가 주소 교정 후 재요청(Phase 1 구현 완료).
- **발송 실패 감지** — 게이트웨이에서 bounce·reject 콜백 수신 시, 대기 중인 사용자 화면에 "메일이 반송됐어요. 주소를 확인해 주세요" 상태로 전환(런칭 전 구현).
- **토큰 만료 후 재발급 UI** — 30분 경과 후 `/auth/verify?token=…` 진입 시 만료 화면 → **재발급 버튼 1클릭으로 새 토큰 발송**(Phase 1 AuthVerify에서 "가입 다시 시작" / "로그인 다시 시작" CTA로 구현됨, 런칭 시 수동 입력 단계 건너뛰도록 이메일 프리필 보강).
- **대체 수신 채널 안내** — 3회 이상 재전송 후에도 수신 실패 보고 시 고객센터(문의하기) 안내. 한국 사용자 한정으로 장기적으로 카카오 알림톡 경로 병행 검토.

이메일 수신 습관이 약한 시니어 타깃 특성상 **첫 진입 1회가 가장 큰 관문**이다 — 이 관문에서 함정 없는 경로를 확보하는 것을 최우선으로 한다.

**세션 지속**

- 링크 검증으로 발급된 세션은 §2.4 규칙을 따른다(토큰 영속, 기기 교체·명시적 로그아웃 시까지). 매직 링크는 **첫 진입·재로그인 시에만** 사용된다.

## 3. 비가입자 초대 정책

회사가 외부 채널로 직접 발송하지 않는다. 회원 본인이 초대 링크를 받아 친구에게 직접 보내고, 친구가 가입 후 본인 작품을 명시적으로 선택하면 자동으로 연결되는 구조. 친구의 동의 없이 회사가 비회원의 전화번호·이메일을 받지 않는다.

### 3.1 초대 링크 발급과 공유

- 작가는 그룹 전시에서 비회원 작가의 **이름만** 입력한다. 전화번호·이메일은 받지 않는다.
- 전시가 검수를 통과하면 **전시 단위로 초대 링크 1개**가 활성화되고, 작가의 마이페이지 전시 카드에 "친구에게 알리기" 버튼이 노출된다.
- 작가가 버튼을 누르면 모바일은 휴대폰 공유 시트, PC는 링크 복사·메일 보내기를 띄운다. 카톡·문자·이메일·단톡방 등 작가가 원하는 채널로 친구에게 보낸다.
- **공유 권한은 전시 업로더에게만**. 그룹 전시의 다른 회원 작가에게는 공유 버튼이 노출되지 않는다.

### 3.2 본인 작품 찾기 (가입자 본인 선택)

초대 링크를 받은 친구가 클릭하면 전시 미리보기 + 가입 안내가 나온다. 가입을 마치면 **본인 작품 찾기** 화면이 한 번 나타난다.

| 화면 요소 | 내용 |
|---|---|
| 헤더 | "혹시 이 중에 본인 작품이 있나요?" |
| 안내 | "본인이 그린 그림이 아닌데 선택하면 작가님께 잘못 연결됐다는 알림이 갑니다." |
| 카드 | 비회원 자리의 작품마다 카드 1장(작품 이미지 + 작가가 적어둔 표시명 + 선택 버튼). 자리가 1장이어도 자동 연결되지 않고 같은 카드 형식으로 노출 |
| 스킵 | "여기 없어요. 그냥 둘러볼게요" — 자동 연결 없이 둘러보기로 이동 |

선택 시 확인 화면이 한 번 나타난다.

> '{표시명}' 자리를 회원님 작품으로 연결할까요?  
> 잘못 클릭하셨다면 작가님께 알려주세요. 작가님이 마이페이지에서 풀어주실 수 있어요.

"네, 맞아요"를 누르면 즉시 연결된다. 카드 표시명은 가입자의 닉네임·프로필 이미지로 자동 전환된다.

작가가 잘못 연결을 풀어둔 자리(작가 미상)는 후보 카드에 나타나지 않는다.

### 3.3 자동 연결 후 알림과 동시 선택

- 연결이 완료되면 작가에게 **정보용 알림 1건**이 가입자 닉네임과 작가가 적어둔 표시명을 함께 담아 전달된다. 별도 액션 없음.
- 두 가입자가 같은 자리를 동시에 선택하려 하면, 첫 사람만 연결되고 두 번째 사람에게는 "이미 다른 분이 본인 작품으로 연결했어요" 안내가 노출된 뒤 카드 목록이 새로 그려진다.

### 3.4 검수·삭제·취소·만료

| 상황 | 초대 링크 처리 |
|---|---|
| 검수 대기·반려 | 활성화 안 됨. 친구가 클릭하면 "이 전시는 검수 중이에요. 잠시 후 다시 확인해주세요" 안내. 작가가 수정 후 재승인 받으면 자동 활성화 — **친구가 받은 링크는 그대로 유효** |
| 검수 승인 | 활성화. 친구가 클릭하면 본인 작품 찾기 흐름 진행 |
| 자동 비공개(§12.2) | 활성 유지하되 랜딩에서 "운영팀 검토 중" 안내 |
| 작품 삭제·작가 탈퇴·비회원 자리 0·발급 후 90일 경과 | 영구 만료. 클릭 시 "초대 링크가 만료됐어요. 보내주신 분께 다시 받아주세요" |

영구 만료된 링크는 다시 살아나지 않는다. 만료는 친구가 클릭하는 시점에 즉석 평가된다(별도 일정 작업 없이 자연 만료). 작가는 마이페이지에서 새 링크를 다시 발급할 수 있다.

**Phase 1 한계**: 초대 링크 정보가 회원 본인 기기 안에만 보관되어 다른 기기에서 친구가 가입하면 링크 활성 상태를 알 수 없다. 친구가 같은 기기에서 가입하면 무관. 검색엔진·SNS 미리보기에서 링크 노출이 의도와 다르게 발생할 수 있고, 일부 봇은 차단 신호를 무시한다. 두 한계는 백엔드 도입 후 정합한다(§31).

### 3.5 잘못 연결됐을 때

- **가입자 자가 해제 진입점은 두지 않는다**. 본인 작품이 아닌데 잘못 연결됐으면 가입자가 작가에게 카톡 등으로 직접 알리고, 작가가 마이페이지에서 그 자리를 작가 미상으로 풀어준다.
- 작가가 회원을 직접 추가한 경우도 같은 방식. 알림은 정보용 1건만 — 거부 액션 없음.
- 자리를 풀면 작가 미상으로 전환되고 가입자의 닉네임·아바타는 그 자리에서 사라진다(§4.1과 동일). 경고·제재 없음.

토큰 없이 자력 가입한 사람은 자동 연결을 받지 않는다. 본인 작품과 연결되고 싶으면 작가에게 직접 부탁해 작가가 자리를 채워준다(서로 아는 관계 전제).

### 3.6 운영팀 신호

작가가 같은 전시에서 자리를 자주 푸는 패턴(도용 의심)은 운영 정보 로그로 적재한다. Phase 1엔 자동 액션이 없고, 누적 데이터는 추후 정책 보강 근거로 활용한다.

## 4. 탈퇴 작가 정책

### 4.1 작가 정보

| 항목 | 탈퇴 후 |
|---|---|
| 작가명 | **"작가 미상"** |
| 아바타 | 기본 이미지 |
| 프로필 페이지 | **접근 불가** |

> "작가 미상" 라벨은 **§4 탈퇴 작가**와 **§23.3 작가 미상 자리**에서 동일 문구로 노출된다. 열람자 UX 관점에서는 **"이 자리에 책임 있는 작가가 현재 없음"** 이라는 같은 의미로 취급한다. 내부적으론 출처가 다르지만(탈퇴/사후 해제) 사용자에게 구분 노출하지 않는다. 필요성 있으면 추후 라벨 분리 검토.

### 4.2 작품

| 항목 | 탈퇴 후 |
|---|---|
| 작품 이미지 | 유지 |
| 작품명·전시명·그룹명 | 유지 |
| 피드 노출 | **유지** |
| 좋아요·저장 | 차단(기존 인터랙션은 유지, 신규 불가) |
| 팔로우 | 차단 |
| 신고 | 차단 |

### 4.2.1 차단 적용 지점 (Phase 1 구현)

**사용자 앱 UI 가드**
- WorkDetailModal 액션 바: 탈퇴 작가의 작품이면 opacity 40% + pointer-events-none으로 시각·상호작용 차단 (좋아요·저장·팔로우·신고·공유 묶음).
- 프로필 페이지: 타인이 탈퇴한 경우 프로필 접근을 전용 안내 화면("탈퇴한 작가입니다")으로 전환 + "둘러보기로 돌아가기" CTA. 본인의 탈퇴는 §4.3 사유 수집 플로우에서 이미 처리.

**저장소 레벨 가드 (defense-in-depth)**
- **좋아요·저장 토글**: 대상 작품의 작가가 탈퇴 상태면 no-op (silent skip). UI 가드가 뚫렸을 때 최종 방어선.
- **팔로우 토글**: 대상 작가가 탈퇴 상태면 no-op.
- **신고 모달(CM-03) 제출**: 대상 작품의 작가가 탈퇴 상태면 에러 토스트 "탈퇴한 작가의 작품에는 신고할 수 없어요." 표시 후 제출 중단.

**차단 대상이 아닌 것 (유지되는 것)**
- 피드·검색에서 작품 노출 자체는 유지 (작가명만 "작가 미상"으로 표시).
- 기존에 좋아요·저장한 사용자 기록은 그대로 보존 (카운터도 유지).
- 운영자 어드민 액션(비공개·삭제 등)은 탈퇴 여부와 무관하게 정상 동작.

### 4.3 탈퇴 사유

USR-STG-03(탈퇴 모달)에서 필수 선택: **이용 빈도 낮음 / 타 서비스 이용 / 개인정보 우려 / 이용 불편 / 기타**.
운영 개선 반영 목적. 데모·분석용으로 최근 탈퇴 사유 1건만 남긴다.

### 4.4 재가입

- 탈퇴 후 같은 이메일·전화로 **즉시 재가입 가능**(대기 기간 없음).
- 재가입은 **신규 계정**으로 취급된다. 과거 계정의 작품·좋아요·저장·팔로우·신고 이력은 복구되지 않음.
- 과거 작품은 §4에 따라 "작가 미상" 상태로 피드에 계속 남음. 재가입한 계정과는 **자동으로 연결되지 않는다**.
- **"작가 재활성" 흐름**(과거 작품 복구 선택지 제공)은 데이터 이전 정책 수립 시 함께 논의([§31 N-6](#31-런칭-전-미해결-항목)).
- 재가입 계정이 과거 탈퇴 이력을 갖는지는 Phase 1에선 식별하지 않는다(해시 대조 없음).

| 전시 상세 화면 | **전시 상세** | Exhibition detail | 작품 상세(이미지 1장이 아님) |
| 운영 서비스 권한 | **운영팀** | Operator | 어드민·관리자(사용자 대면 문구), 운영자(혼용 시 운영팀으로 통일) |
| 신고 누적 시스템 차단 | **자동 비공개** | Auto-hide | 자동 숨김(신고자 본인 한정 숨김과 구분 필요) |
| 신고자 본인 한정 숨김 | **신고자 한정 자동 숨김** | Reporter-only hide | 자동 숨김(자동 비공개와 혼동 우려) |

> **검수 vs 검토**: 신규 업로드·편집 후 피드 노출 전 운영팀이 수행하는 심사는 **검수**로 통일. "검토"는 법무·UX·기능 로드맵 의사결정 같은 다른 맥락에만 사용. "신고 처리" 맥락은 **처리**로 쓴다.

> **자동 비공개 vs 신고자 한정 자동 숨김**: 두 개념을 혼동하지 말 것. **자동 비공개**(§12.2)는 신고가 누적되어 시스템이 모든 사용자에게 차단하는 것. **신고자 한정 자동 숨김**(§12.4)은 신고를 한 본인에게만 더 이상 보이지 않게 하는 개인화 기능.

> 코드 내부 타입 `Work` = UX 상 "전시(Exhibition)". 내부 타입명에만 과거 명이 남아 있다. **런칭 전 데이터베이스 설계 시** 코드의 `Work` 타입을 `Exhibition`으로 이름 변경하는 것을 검토한다.

## 7. 포인트(AP) 정책

### 7.1 UI 노출

- **Phase 1: 프런트에 포인트 UI 미노출.** 백그라운드 적립만 동작.
- `/points` 경로는 `/`로 리다이렉트.
- 포인트 적립 토스트·삭제 시 회수(-20 AP) 등 **어떤 포인트 신호도 사용자에게 노출하지 않는다**(백그라운드 로그만).
- 어드민에서만 포인트 원장 열람 가능. 사용자 UI 복원은 추후 별도 로드맵으로 검토(2차 오픈 고정 아님).

#### 설계 의도 — 비노출은 버그가 아니라 의도

- Phase 1의 포인트는 **내부 적립(소급 데이터) 용도 전용**이다. 사용자에게 보여주지 않는 것이 의도된 설계.
- 존재 이유: 추후 포인트 UI를 공개할 경우, **Phase 1부터 활동한 기존 사용자의 과거 이력을 소급 반영**하기 위함. 공개 시점은 별도 결정.
- 따라서 **포인트 관련 사용자 인지 부족을 UX 이슈로 다루지 않는다** — 삭제 시 회수(-20 AP) 등 페널티도 토스트·배지 없이 조용히 기록만 한다. 포인트 UI 공개 결정 전까지 이 상태를 유지.
- 재검토 체크포인트: 포인트 UI 공개 검토 착수 시 §7.2 적립 이벤트 표를 기준으로 소급 반영 로직 확정(이벤트 시점 기록은 이미 원장에 남음).

- **자동 저장 없음.** 수동 "초안 저장" 버튼 클릭 시에만 저장.

## 9. 작품명·전시명·그룹명 정책

### 9.1 공통 규칙

- 최대 **20자**.
- **빈 문자열·공백만 저장 불가** (에러 토스트).
- 비속어 포함 불가(한/영 47개 금칙어 필터, 필드별 에러 메시지).

### 9.2 필수 여부 및 빈값 표시

| 필드 | 필수 | 빈값 표시 |
|---|---|---|
| 전시명 | **필수**. 빈 값으로 전시 등록 불가. | — |
| 작품명 | 선택. 빈 값 허용. | 카드·상세에서 **"(제목 없음)"** 회색 텍스트로 표시(시스템이 "무제"로 자동 채우지 않음). |
| 그룹명 | 그룹 전시(참가자·강사 역할) 시 **필수**. 혼자 전시는 미입력. | — |

> **"무제" vs "(제목 없음)" 구분 원칙**: 미술계에서 "무제(Untitled)"는 작가가 의도한 정식 작품명이다. 시스템이 빈값을 "무제"로 자동 채우면 작가가 의도한 무제와 입력 누락이 구분되지 않는다. 따라서:
> - 작가가 직접 입력한 **"무제"** → 그대로 노출 (작품명 자리, 일반 텍스트)
> - 빈값 → **"(제목 없음)"** 으로 노출 (작품명 자리, 회색 + 괄호로 시스템 placeholder임을 시각·의미 양쪽으로 구분)
>
> 시니어 친화 근거: 작품명 줄이 비어 있으면 "왜 비어있지·내가 입력 안 했나·에러인가" 의심을 유발한다. 정해진 자리에 명시적 한국어 텍스트가 있어야 안정감 있다. "—" 같은 추상 기호는 시니어에게 의미가 즉시 읽히지 않아 부적합.

### 9.2.1 작품명(piece title) 편집 권한·진입점

- **편집 권한**: **업로더 본인**만 가능. 그룹 전시 참여 작가는 본인이 그린 그림이라도 **업로더가 아닌 경우 작품명을 직접 수정할 수 없다.**
- **편집 진입점은 단일**: **USR-PRF-05(전시 탭)** 의 본인 업로드 전시 카드 → 인라인 편집 모달(USR-PRF-11). USR-PRF-06(내 작품 탭)에서는 **편집을 제공하지 않는다**(보기·Disavow 전용).
- **이 단일 진입점 정책의 설계 근거**: 시니어 사용성. 같은 외형의 카드(본인 업로드 piece와 그룹전 참여 piece)가 한 화면(내 작품 탭)에 섞여 있을 때 어떤 카드는 편집되고 어떤 카드는 안 되면 시니어가 가장 헷갈린다. 내 작품 탭은 "본 작품 모음 갤러리"로 단순화하고, 수정은 "전시 탭 단일"로 멘탈 모델을 정리한다. 클릭 1회 추가의 비용보다 두 진입점이 만드는 헷갈림이 더 크다는 판단.
- **참여 작가의 대안**: ① 업로더에게 구두로 수정 요청(오프라인 관계) ② USR-PRF-06의 **"본인 작품 아님"**(Disavow, §3.5)으로 슬롯을 `'unknown'`(작가 미상)으로 전환.
- **설계 근거(권한)**: 이미지 추가·제거·순서·커버 등 전시 구성 권한은 업로더에게 있다. 작품명도 전시 구성의 일부이므로 **"구성에 영향을 미치는 메타는 업로더 권한"**으로 일관되게 적용한다. "내가 내릴 수 없는 그림의 제목만 고칠 수 있다"는 비대칭을 제거한다.
- **저장 시 동기화**: 제목은 piece 단위 저장. 동일 전시를 다른 사용자가 보면 즉시 반영. 재검수 트리거 없음(§12.1.2).

### 9.3 그룹명 자동완성

- **중복 허용**(2026-04-17 이후. 정규화·캐논 맵 제거).
- 사용자 본인 최근 그룹명 히스토리 상한 50개 유지.
- 자동완성 후보: 빈 쿼리 최대 30개, 입력 중 최대 20개.

### 9.4 닉네임

- 닉네임: 2~20자, 비속어 불가.
- 실명은 Phase 1 어떤 가입 경로에서도 수집하지 않는다. Phase 2에서 정산·세무 등 실제 용도가 생기면 그때 도입을 재검토한다.

## 10. 공유 정책

### 10.1 공유 버튼 규칙

- **공유 버튼은 1개.** "공유하기"를 누르면:
  - **모바일**: 기기에 설치된 앱(카톡·문자·라인 등) 선택 화면
  - **PC**: 초대 멘트 + 링크가 클립보드에 복사 + 토스트 "링크가 복사되었어요"

### 10.2 공유 콘텐츠

- 전달 내용: `"[작가명] 님이 [전시명] 전시에 초대합니다. Artier에서 작품을 감상해 보세요. [링크]"`
- **링크만 단독 전달 금지.** 항상 멘트와 함께.

### 10.3 공유 URL 쿼리 분기

전시 URL은 진입 맥락을 쿼리 파라미터로 구분한다.

| 쿼리 | 랜딩 화면 | 용도 |
|---|---|---|
| (없음) | 전시 상세 모달(USR-EXH-01) | 피드에서 직접 탭 |
| `?from=invite` | 초대장 랜딩(USR-EXH-03) | 비회원 초대 링크 |
| `?from=credited` | 초대장 랜딩(USR-EXH-03) | 크레딧된 참여 작가 공유 |
| ~~`?from=work`~~ | ~~작품 공유 랜딩(USR-EXH-04)~~ | **Deprecated (2026-04-20)** — 작품 단위 공유 UI 제거됨. 레거시 링크 호환용으로 라우트만 유지. |

공유 버튼은 항상 `?from=invite`를 **자동으로 부여**한다. 작품 단위 공유(`?from=work`)는 2026-04-20 제거됨 — "공유는 전시 단위가 기본 단위"라는 원칙 확립.

## 11. 카메라 사진 차단 정책

Artier는 순수미술 플랫폼이므로 스마트폰·DSLR로 촬영한 사진 업로드를 차단한다.

### 11.1 판별 방법

1. 이미지 파일의 실제 데이터(**magic bytes**)를 읽어 JPEG 여부 확인.
2. JPEG인 경우 EXIF 메타데이터에서 `Make`(제조사)·`Model`(모델) 정보가 있으면 **차단**.
3. 파일명이나 확장자가 아닌 **실제 파일 내용**으로 판별하므로 확장자 변경으로 우회 불가.
4. PNG·WebP·GIF 등 비-JPEG 포맷은 EXIF 구조가 없으므로 차단 대상 **아님**.

### 11.2 차단 시 동작

- 업로드 화면(USR-UPL-02)에 인라인 안내("Artier는 디지털 드로잉 작품 전용 플랫폼입니다") 배너 표시.
- 파일은 추가되지 않음(업로드 큐 진입 차단).

### 11.3 예외

현재 운영 기준으론 예외 없음. "필름 스캔"·"아트워크 촬영" 같은 정당한 사용 사례가 다수 접수되면 §20 확정 수치 재검토.

## 12. 신고·모더레이션 정책 (Phase 1)

Phase 1은 **작품 단위 모더레이션만** 다룬다. 사용자 계정 차원 제재(주의·시한부 정지·영구 정지·경고 카운터·허위 신고 카운터)는 추후 별도 로드맵에서 재설계한다(§12.3 참조).

### 12.0 신고 사유

사용자 신고는 **단일 카테고리**로 단순화한다. Phase 1 시니어 디지털 드로잉 작가 커뮤니티에서 사용자가 신고할 만한 가장 명확한 케이스는 **타인의 작품을 본인 것처럼 올린 도용**이며, 운영팀이 모든 신고를 직접 검토하므로 사전 카테고리 분류가 의사결정 가치를 더하지 않는다.

| 라벨 | 한 줄 설명(신고 화면 노출) |
|---|---|
| **이 작품은 작가 본인의 그림이 아니에요** | 다른 사람의 작품을 본인 것처럼 올린 것 같아요 |

- 자유 메시지 입력란(200자 이내) — 사용자가 추가 정황(원본 출처 링크 등)을 적을 수 있음.
- 부적절·스팸·매체 부적합 등 다른 모더레이션 사유는 **검수 단계**(§11)와 **운영팀 직권 등록**(ADM-RPT-01) 채널로 처리. 사용자 신고 카테고리에는 두지 않는다.
- 작품의 완성도(저품질)는 사용자 신고 영역이 아니다.

**처리 SLA**: §22.2 영업일 24시간 적용.

### 12.1 신고 처리 액션 (어드민)

운영팀은 접수된 신고에 대해 아래 3가지 중 하나를 선택한다(ADM-RPT-01).

| 액션 | 효과 |
|---|---|
| **삭제** | 해당 전시 영구 삭제. 확인 다이얼로그(복구 불가) 필수. |
| **기각** | 신고를 부당하다 판정. 자동 비공개 상태였다면 **즉시 복원**. |
| **비공개 유지** | 전시를 둘러보기·검색에서 제외. 작가 본인 프로필엔 보임. 작가는 수정해 재검수를 요청할 수 있다. |
| _(제거)_ | 액션 없이 큐에서만 제거(레거시 호환). |

> 좋아요·저장·Pick·신고·배지는 모두 **전시(컨테이너) 단위**. 개별 이미지(작품 1장) 단위의 인터랙션·배지는 없다.

### 12.1.3 신고 처리 알림 전송 (왕복 피드백)

자동 비공개 발동 시, 그리고 운영팀이 액션을 수행할 때 영향받는 사용자에게 시스템 알림이 생성된다. 작가·신고자가 적용된 조치를 인지하고 필요 시 대응(수정 등)할 수 있게 하기 위함.

| 이벤트 | 알림 수신자 | 알림 문구(요약) |
|---|---|---|
| **2회 자동 비공개** (§12.2) | 작품 작가 | "회원님의 전시 '{title}'이 신고 누적으로 자동 비공개 처리되었습니다. 운영팀 검토 대기 중." |
| **삭제** | 작품 작가 | "회원님의 전시 '{title}'이 신고 처리로 삭제되었습니다." |
| **기각 → 복원** | 작품 작가 | "'{title}' 전시가 검토 결과 정상 복원되었습니다." |
| **기각** | 신고자 | "접수하신 신고가 운영팀 검토 결과 기각되었습니다." |
| **비공개 유지** | 작품 작가 | "회원님의 전시 '{title}'이 비공개 처리되었습니다. 수정해 재검수를 요청할 수 있어요." |

- 알림 `type`: 모두 `system`
- 알림 센터(USR-NTF-01)에서 회색 배경 + 종 아이콘으로 표시
- 작품 관련 알림은 `workId` 포함(삭제는 예외 — 참조할 작품이 없음)

### 12.1.1 검수 반려 사유 이력 보존

별도 문맥으로, **콘텐츠 검수(ADM-REV-02)의 반려 액션**은 전시에 두 종류의 정보를 남긴다.

- **현재 반려 사유**(단일 값): 지금 반려 상태에 표시되는 사유. 작가가 수정해 재검수 대기로 넘어가거나 승인되면 초기화된다.
- **반려 이력**(누적 기록): 반려가 발생할 때마다 사유와 시각이 한 줄씩 누적. 작가가 수정 재발행해도 사라지지 않는다.

**보존 목적**
- 운영 감사 — 과거 반려 판단 근거 추적
- 재범 추적 — 같은 작가·작품의 반복 반려 패턴
- 사유 분포 통계 — 서비스 정책 개선

**UI 노출 원칙**
- 사용자 앱 반려 사유 모달(USR-PRF-12): **현재 반려 사유만** 표시. 과거 이력 노출하지 않는다(사용자 혼란 방지).
- 어드민 작품 관리(ADM-WRK-01)·검수(ADM-REV-02): 필요 시 이력 요약(누적 N회 반려)·전체 목록 열람 가능.

### 12.1.2 검수 상태별 수정·삭제 정책

작가가 자기 전시에 대해 수정·삭제하는 것은 **모든 검수 상태에서 원칙적으로 허용**한다. 자기 콘텐츠 제어 + 개인정보 보호(즉시 내릴 권리) 원칙 준수.

**핵심 규칙: 검수 상태 변경은 "이미지가 바뀌었는지"만으로 판정**

이미지 변경 = 이미지 추가·삭제·교체·참여 작가 재지정·커버를 직접 새 이미지로 변경.
메타만 변경 = 전시명·그룹명·작품명·커버 인덱스(이미 올린 것 중 어느 장을 커버로 쓸지)·이벤트 연결.

| 기존 상태 | 이미지 변경 시 | 메타만 변경 시 |
|---|---|---|
| **공개 중** | → 검수 대기(재검수) | **공개 중 유지**(즉시 반영) |
| **검수 대기** | 검수 대기 유지(운영자가 최신판 검수) | 검수 대기 유지(최신판 반영) |
| **반려** | → 검수 대기(재검수) | **→ 검수 대기**(강제 재검수, 아래 예외 참조) |

**근거**
- 검수 반려 사유는 **이미지 기반** 판정(매체 부적합·부적절·저작권 등). 메타데이터는 비속어 필터가 저장 시점에 처리하므로 운영자 재검토 불필요.
- 시니어 작가는 오탈자·실수 재고 가능성이 높음. 메타 오탈자 때문에 긴 시간 잠그면 UX 부담 과다.
- 삭제까지 막으면 개인정보보호법 관점에서 문제.

**예외: 반려 상태에서 메타만 고쳐 발행한 경우 → 강제 재검수**

작가가 USR-PRF-12 반려 사유 모달의 "수정하기" → USR-UPL-03 진입 후 발행하면 사용자 의도가 **"이 전시의 재검수를 요청한다"** 이다(원본 정책의 "메타만 = 상태 유지" 원칙과 충돌).

따라서:
- 반려 상태에서 **이미지 변경 없이 메타만 수정한 발행**도 → 강제로 검수 대기 + 현재 반려 사유 초기화.
- 반려 이력은 그대로 보존.
- 발행 직후 작가에게 토스트 "다시 검수가 시작되었어요" 노출.
- USR-UPL-03의 발행 CTA 라벨을 반려 편집 모드 한정 **"다시 검수 요청하기"** 로 변경(작가에게 의도 확인 효과).
- 운영자 입장: 메타만 변경된 재검수 건은 ADM-REV-01 큐에서 "재검수(메타만)" 부가 라벨로 노출 — 1차 사유가 무엇이었든 이미지가 그대로면 통상 동일 결과로 빠르게 처리.

이 예외는 "반려 → 사용자 의도된 재검수 요청"의 명시적 분기로, IA USR-UPL-03 / PRD_User §5.1.2 AC-R4와 정합한다.

**수정 진입 경고 (검수 대기 중 수정)**
- 검수 대기 중인 전시의 점점점 "수정" 진입 시 확인 모달:
  - 제목: "검수 중인 전시를 수정할까요?"
  - 설명: "지금 수정하면 검수 대기에 변경 사항이 반영되고, 검수 대기 시간이 다시 시작됩니다."
  - [취소] / [수정하기]
- 반려 상태는 반려 사유 모달이 이미 맥락을 제공하므로 추가 경고 없음.

**저장 후 토스트 (5종, 사용자에게 결과 명확히 고지)**

| 시나리오 | 토스트 |
|---|---|
| 공개 중 유지(메타만 변경) | "수정 사항이 바로 반영되었어요." |
| 공개 중 → 검수 대기(이미지 변경) | "이미지가 바뀌어 잠시 검수 대기예요. 공개되면 알려드릴게요." |
| 검수 대기 유지 | "수정 사항이 검수 대기에 반영됐어요." |
| 반려 → 검수 대기(이미지 변경) | "수정한 작품이 재검수 대기에 들어갔어요." |
| 반려 유지(메타만 변경) | "수정 사항이 반영됐어요. 공개되려면 반려 사유를 해결해주세요." |

**탭별 편집 범위**

| 탭 | 편집 범위 | 상태 전이 가능성 |
|---|---|---|
| **전시 탭**(USR-PRF-05) | 이미지·작가·메타 전부 | 위 매트릭스 전체 적용 |
| **내 작품 탭**(USR-PRF-06) | **본인 업로드 전시의 작품 제목만**(인라인 편집) · 참여 작가 자리는 읽기 전용 + 사후 해제만(§9.2.1) | **상태 변경 없음**(메타만이므로 원본 상태 유지) |
| **수강생 작품 탭**(USR-PRF-07) | 전시 탭과 동일 | 위 매트릭스 전체 적용 |

"내 작품" 탭은 이미지 변경을 지원하지 않는 구조라 이 정책과 자연스럽게 일치한다. 전시 탭·수강생 작품 탭은 같은 편집 화면을 공유하므로 동일 규칙이 자동 적용된다.

**삭제 (모든 상태 공통)**
- 확인 다이얼로그(복구 불가 스타일) + 비회원 초대 대기 시 추가 경고 문구 "초대된 비회원이 있습니다. 삭제하면 초대가 무효화됩니다".
- 검수 상태와 무관하게 자유 허용. 24시간 내 삭제 시 AP -20 회수.

**운영자 동시 처리 충돌(레이스) 처리** — 런칭 전 백엔드 연동 시 보강
- 운영자가 승인·반려 버튼을 누르는 순간 작가가 이미 삭제한 경우 → Phase 1은 조용히 무시.
- 런칭 전 백엔드 연동 시점에 "이미 처리되었거나 삭제된 전시입니다" 에러 토스트 + 큐 자동 갱신을 구현.

### 12.2 2회 신고 자동 비공개

- 같은 전시에 **2번째 신고가 접수되는 순간**, 그 전시는 즉시 자동 비공개 처리된다.
- 자동 비공개 효과 = 운영팀 "비공개 유지" 액션과 동일. 둘러보기·검색·피드에서 제외, 작가 본인 프로필에서만 보임.
- 작가에게 시스템 알림 발송(§12.1.3).
- 이후 운영팀이 신고 큐에서 §12.1의 3가지 액션(삭제·기각·비공개 유지) 중 하나를 선택해 최종 처리한다.
- 운영팀이 **기각**으로 판정하면 해당 전시는 즉시 복원된다.
- 자동 비공개 상태에서도 작가는 §12.1.2에 따라 본인 전시를 수정할 수 있다. 수정 시 이미지 변경 여부에 따라 검수 상태 전이(재검수)는 발생하지만, **자동 비공개 상태는 운영팀 판정 전까지 유지**된다.

#### 12.2.1 고지·처리 SLA

작가가 갑작스러운 비공개에 놀라거나 불만 CS가 쌓이지 않도록, **자동 비공개 트리거와 동시에 명확한 기한으로 절차를 안내**한다.

| 시점 | 대상 | 채널 | SLA |
|---|---|---|---|
| **자동 비공개 발동 즉시** | 작가 본인 | 시스템 알림 + 사용자 보유 연락처 기준 채널(§1 — 한국 전화번호 → 알림톡, 그 외 → 이메일) | **실시간** (15분 이내 전송 시도) |
| **운영팀 최종 판정** | — | 어드민 큐(ADM-RPT-01) | **접수 후 24시간 이내 목표** (영업일 기준, 주말·공휴일은 다음 영업일 09:00 기준) |
| **판정 결과 재고지** | 작가 + 신고자 | §12.1.3 알림 매트릭스 | 판정 액션 실행 즉시 |

**자동 비공개 작가 알림 문구 보강** (§12.1.3 "2회 자동 비공개" 행 연동):
- "회원님 전시가 신고 누적으로 **잠시 비공개**되었어요. 운영팀이 **영업일 24시간 안에** 확인해드리고, **문제 없으면 다시 공개**돼요. 궁금한 점은 문의하기로 연락 주세요."

**SLA 미준수 시 완화책** (운영 백오피스 관점):
- 접수 48시간 경과한 자동 비공개 건은 ADM-RPT-01 큐에서 **"SLA 초과" 배지** 표시(우선 처리 유도).
- 72시간 경과 건은 운영 대시보드 경고 + 슬랙 알림(런칭 전 구현).

### 12.3 Phase 1 범위 밖 (추후 이관)

다음 항목은 추후 별도 로드맵에서 재설계한다:

- 사용자 계정 단위 제재(주의·시한부 정지·영구 정지)
- 경고 카운터·허위 신고 카운터·자동 정지 승격
- 이의제기 절차·관찰 기간·선의 리셋·쿨다운
- 중대 위반(저작권 침해 확정·청소년 유해·정지 회피 등)에 대한 즉시 영구 정지

Phase 1은 **작품 단위 모더레이션**(§12.1·§12.2)과 **신고자 보호**(§12.4)로 한정한다.

### 12.4 신고자 보호

- 동일 `(신고자, 대상유형, 대상ID)` 조합당 **1회만** 허용.
- 신고한 전시·작가는 해당 **신고자 본인 한정**으로 자동 숨김 처리한다(§5.1 "신고자 한정 자동 숨김"). 다른 사용자에게는 영향 없음.
- 숨김은 피드·검색·상세 모달에서 즉시 반영된다.

## 13. 업로드 유형·역할 정책

### 13.1 업로드 유형(2종)

업로드 화면의 첫 단계(USR-UPL-01)에서 **둘 중 하나 필수 선택**.

| 유형 | 설명 |
|---|---|
| **혼자 올리기** | 내 작품만으로 구성된 전시 |
| **함께 올리기** | 동호회·클래스·지인 등 여러 작가의 작품을 한 전시에 모음 |

### 13.2 함께 올리기의 역할(2종)

"함께 올리기"를 선택하면 이미지를 추가하기 전에 **본인 포함 여부**를 먼저 결정.

| 역할 | 본인 작품 포함 | 해당 사례 |
|---|---|---|
| **참가자** | **최소 1점 필수** | 동호회·클래스·지인 등 본인도 참여하는 그룹 전시 |
| **강사** | **포함 금지** | 수강생 작품만 대신 올리는 경우 |

### 13.3 역할별 강제 규칙

**참가자 (강사 OFF)**
- 본인 작품 없이 전시 등록 **차단**. 에러 메시지: "본인의 작품이 1점 이상 포함되어야 합니다".
- 검증 시점: 발행 핸들러에서 참여 작가 미지정 검증 직후.

**강사 (강사 ON)**
- 본인 작품 포함 **금지**. 작가 선택 리스트에서 본인 자동 제외.
- 강사 ON 전환 시 본인으로 지정된 이미지 슬롯은 **자동 초기화**.
- 업로드 화면 상단에 **"그룹 전시(강사)" 뱃지** 표시.

### 13.4 강사 표시 파생

- 강사 역할로 업로드한 이력이 **1건 이상**이면 프로필(USR-PRF-01)에 "수강생 작품" 탭이 **자동 노출**된다.
- 강사 여부 수동 토글 UI 없음 — 상태 불일치 방지 목적.
- 해당 작품 **모두 삭제 시** 탭 자동 제거.

### 13.5 그룹 전시 공동 책임 안내

그룹 전시는 참여 작가들이 한 전시 컨테이너에 함께 노출되는 구조다. 한 작품에 신고가 누적되면(§12.2 자동 비공개) **전시 전체가 비공개**되어 다른 작가의 작품도 함께 영향받는다. 이 구조를 작가들이 발행 전 인지하도록 한다.

**발행 단계 안내**:
- 함께 올리기 발행 직전, 작가에게 안내 노출:
  > "그룹 전시는 참여 작가의 작품이 한 전시 안에 함께 노출돼요. 한 작품이라도 신고가 누적되면 전시 전체가 비공개될 수 있어요. 참여 작가들과 작품·저작권을 함께 점검해 주세요."
- 별도 동의 체크박스 없이 안내만 — 시니어 친화(추가 클릭 부담 최소화).
- 안내 문구는 [Copy_v1.md](Copy_v1.md)의 i18n 키로 관리.

**근거**:
- piece 단위 모더레이션은 시니어 인지 부담·운영 복잡도가 도매금 회피 가치보다 큼(검토 후 폐기).
- 발행 전 안내로 작가가 "공동 책임" 인지 → 차단 시 부당함이 아닌 **합의된 결과**로 받아들이게 한다.

### 13.6 그룹 전시 사후 강등(자동 개인전 전환)

발행된 그룹 전시에서 시간이 지나며 참여 작가가 빠져 **본인 1명만 남게 된 경우** 시스템이 전시를 자동으로 **개인전**으로 전환한다.

**트리거 조건**:
- 참여 작가 자리에서 본인 외 모든 작가가 사후 해제([§3.5](#3-비가입자-초대-정책))되어 작가 미상으로 바뀜
- 또는 다른 사정으로 본인 1명만 남는 상태가 됨

**작가 알림(필수)**:
- 1명 도달 시점에 **시스템 알림 1건** 발송: "전시 〈〈전시명〉〉이 참여 작가가 모두 빠지면서 개인전으로 변경되었습니다."
- 알림은 **1명 도달 시점에만 1회**. 그 이후 추가 해제가 발생해도 추가 알림 없음(알림 폭주 방지).
- 알림 채널은 §1 정책에 따른다.

**복원 경로**:
- 작가가 수정 모드에서 새 참여 작가를 추가하면 다시 그룹전으로 전환 가능. 자동으로 그룹전 복귀하지 않음(작가 명시적 액션 전제).

**검수 영향**:
- 사후 강등은 메타 변경에 해당. 검수 상태는 변경하지 않는다([§12.1.2](#1212-검수-상태별-수정삭제-정책)).

**근거**:
- 데이터 일관성: 그룹 전시는 정의상 ≥2명이 참여한 전시. 1명만 남은 그룹전은 정책상 어색한 상태.
- 사용자 인지: 사후 변환이 조용히 일어나면 작가가 다음에 본인 작품을 봤을 때 "왜 개인전이지?" 혼동. 알림으로 변경 사실을 명확히 통보.

### 13.7 금지(회귀 방지)

- 사용자 프로필에 강사 여부를 표시하는 **독립 플래그·토글 재도입 금지**.
- "강사 공개 목록" 류의 별도 저장소 재생성 금지(2026-04-13 단일화 완료).

## 14. 커버 이미지 정책

- 업로드 시 **커스텀 커버 이미지**를 별도로 지정할 수 있다.
- 지정된 커버는 **전시의 첫 장 표지** 및 **둘러보기 피드 썸네일**로 사용된다.
- 커버 이미지는 전시 이미지 배열에 **포함되지 않는다** → 이미지 개수 배지(예: 1/10)에 영향 없음.
- 커버 미지정 시: 전시 이미지 중 대표 이미지(없으면 첫 이미지)가 썸네일·표지로 사용.
- 전시 상세 모달에서 **커버 슬라이드는 블랙 배경 + 전시명 라벨**로 표시된다. 작가 정보 오버레이는 커버 위엔 노출하지 않는다.

## 15. 큐레이션·배지 정책

### 15.1 큐레이션 개념 4종

Artier의 큐레이션은 목적·수명·공개 여부가 다른 4가지 개념으로 구분한다.

| # | 개념 | 설명 | 공개 | 배지 |
|---|---|---|---|---|
| 1 | **Artier's Pick** | 운영팀이 매주 선정하는 추천 전시 (최대 10개, 교체) | 공개 | **영구** |
| 2 | **기획전** | 운영팀이 기존 전시를 주제·맥락으로 선별·구성한 컬렉션 (개수 제한 없음, 그룹전·작가전·특집전 포괄) | 공개 | 없음 |
| 3 | **이벤트** | 운영팀이 주제를 제시하고 사용자가 응모 → 운영팀이 선정. 주간 공모형 테마전도 이벤트의 한 종류 | 공개 | **선정작에만 영구** |
| 4 | **운영 내부 컬렉션** | 좋은 전시·작가·그룹을 운영팀이 내부 북마크 (발굴·추적 용도) | **비공개** | 없음 |

### 15.2 배지 체계 (공개)

공개 배지는 **3종뿐**이다. 응모작에는 배지 없음.

| 배지 | 발생 | 수명 |
|---|---|---|
| ★ Artier's Pick | Pick 선정 이력 (한 번이라도 받았으면) | **영구** |
| 이벤트 선정작 | 이벤트·공모형 테마전에서 선정된 이력 | **영구** |
| 강사 | 강사 역할로 업로드된 전시 | §13.4 발행 정책으로 결정 |

본인 전용 상태 배지(검수 "확인 중"·"게시 불가")는 위와 별개. 타인에게 보이지 않는다.

**배너 라벨은 제공하지 않는다.** NEW/HOT/EVENT/PICK 같은 부속 라벨을 두지 않는다.

### 15.3 Pick 운영 기준

- **상한**: 최대 10개. 주 단위로 교체(현재 운영 수동).
- **영구 배지**: 해제 시 현재 Pick만 해제, **한 번이라도 받은 이력 배지는 영구 유지**.
- **자동 초기화**: 현재는 운영팀 수동. **서버 스케줄러 연동 후** "매주 월요일 자정" 자동화.
- **20일 중복 제한**: 현재 미구현(운영 수동). **서버 검증 연동 후** 자동화.

**Pick 영구 배지 인플레이션 모니터링**
주간 10개 기준 연간 최대 500개가 영구 Pick 배지를 받는다. 2~3년 운영 시 배지 의미 희석 우려 → **분기별 누적 수치 점검**, 필요 시 "연도별 Pick" 구분자 도입 또는 선정 기준 재조정.

### 15.4 Pick vs 기획전 경계

| 구분 | Pick | 기획전 |
|---|---|---|
| 단위 | **1개 전시씩 독립 선정** | **개수 제한 없음 — 운영팀 재량** (1개 전시도 가능, 주제·맥락으로 선별·구성) |
| 배지 | 영구 부여 | 없음 |
| 교체 | 주간 | 자유(운영 판단) |

**경계는 "개수"가 아니라 "주제·맥락의 유무"**. 1개 전시라도 기획 주제가 명확하면 기획전으로, 여러 개라도 주제 없이 단순 추천이면 Pick으로 운영한다. 예: "이번 주 봄 풍경 10선"처럼 테마·스토리가 있으면 기획전. 운영자 간 일관성의 판단 기준은 개수가 아닌 **"이 묶음을 하나의 맥락으로 설명할 수 있는가"**.

### 15.5 이벤트 선정 시스템 (런칭 전 필수)

배지 원칙과 구현 모두 런칭 전 필수. 구현 시 포함해야 할 것:

- 어드민 이벤트 상세에 "선정 작품 관리" 섹션 추가(응모작 → 선정 체크)
- 공모형 주간 테마전도 이벤트의 한 종류로 관리(별도 스토어 대신 이벤트 스토어로 통합)
- **응모**(이벤트 연결)와 **선정**(배지 부여)을 개념·데이터 상 명확히 구분
- 선정 시점에 전시에 영구 배지 부여, 이벤트 종료 후에도 유지

이벤트 브랜드가 정착하면(예: "Artier Awards") 배지에 어느 이벤트에서 선정됐는지 연결 정보를 포함해 두어 추후 이벤트별 라벨 세분화 가능성을 확보한다.

### 15.6 Pick 선정 알림 전송 (왕복 피드백)

운영팀이 ADM-PCK-01에서 작품을 Pick에 추가하면 대상 작가에게 시스템 알림이 발송된다.

- **문구**: "회원님의 전시 '{title}'이 Artier's Pick으로 선정되었어요. 축하드려요!"
- **타입**: `pick` — 알림 센터에서 주황 배경 + 별 아이콘
- **workId 포함**: 알림 탭 시 해당 전시 상세로 이동

Pick 해제 시에는 알림을 발송하지 않는다(주간 교체 특성상 해제는 일상적 운영). 영구 이력 배지는 유지되므로 작가는 "한 번이라도 받았다"는 사실만 기억하면 충분.

## 16. 피드 랭킹·검색 스코어링 정책

### 16.1 피드 랭킹 (둘러보기·USR-BRW-01)

**작품 점수 공식**

```
base = log1p(likes) × 2.2 + log1p(saves) × 3.2
following_bonus = 3.5  (현재 사용자가 작가를 팔로우 중일 때)
total = base + following_bonus
```

**소스 버킷별 부스트**

| 소스 버킷 | 부스트 | 노이즈 범위 |
|---|---|---|
| Pick (Artier's Pick) | +14 | ±5 |
| Theme (주간 테마전) | +8 | ±6 |
| Featured (추천 작가) | +6 | ±6.5 |
| Personalized (팔로잉) | +7 | ±7.5 |
| Recent (14일 이내) | +3 | ±7.5 |
| Rest (일반) | 0 | ±10 |

**인터리빙 패턴**

```
pick → rest → theme → rest → personalized → rest → featured → rest → recent → rest
```

(최대 2사이클 반복 후 끝)

**핵심 제약**
- 같은 작품은 **하나의 버킷에만 노출**(중복 차단).
- 이미 본 작품은 각 버킷 **하단**으로 밀림.
- 댓글 수는 점수에 반영하지 않음(Phase 1 범위 밖. 댓글 기능 자체가 PRD Out of Scope).

### 16.2 피드 공개 규칙

- **승인(approved)** 상태의 전시만 공개 피드·검색·큐레이션에 노출.
- 검수 대기(pending) / 반려(rejected) → **본인 프로필에서만 보임**.
- 비공개(hidden) → 본인 프로필에서만 보임. 둘러보기·검색·신고자 숨김 모두 비노출.
- 상태 미설정 시 기본 노출(레거시 호환).

### 16.3 검색 스코어링 (USR-SRC-01)

| 필드 | 포함 점수 | starts-with 보너스 |
|---|---|---|
| 작품명(제목) | +10 | +6 |
| 작가명 | +8 | +4 |
| 전시명 | +8 | +4 |
| 그룹명 | +7 | — |
| 설명 | +3 | — |
| 태그 | +4 | — |

**통합 상위 매치**: 검색 결과 상단에 작가 1명 + 작품 2개 고정 노출.

### 16.4 검색 히스토리

- 로그인 사용자: 계정별 기록 유지.
- 게스트: 공용 기록.
- 로그인 시 **게스트 기록 → 계정 기록으로 병합**.
- 상한 **10개**, 검색어 최대 **100자**.

## 18. 데이터 보관·저장 정책

### 18.1 Phase 1 저장 경계

Phase 1은 **백엔드 서버 없이 브라우저 로컬 저장소**에 전체 상태를 보관한다. 작품·사용자 프로필·어드민 상태가 모두 클라이언트에 존재한다.

| 분류 | 보관 내용 | 관련 섹션 |
|---|---|---|
| 작품·초안 | 전시·초안 JSON, 스토리지 버전 | 코드 단일 소스(개발자 영역) |
| 사용자 상태 | 프로필·로그인·팔로잉·좋아요·저장 | §2·§7 |
| 가입 레지스트리 | 등록 이메일·전화 레지스트리, 프리필 임시 값 | §2.3 |
| 신고·비공개 | 신고 로그, 신고 처리 상태(비공개·삭제·기각) | §12 |
| 포인트 | 원장·상태·업로드 시각 | §7 |
| 초대 | 발송 로그·매칭 로그·본인 확인 대기 목록 | §3 |
| 어드민 | 배너·이벤트·Pick·기획전·체크리스트·멤버 상태 | §17 |

전체 키·구조는 코드 단일 소스(개발자 영역)에서 관리.

### 18.2 스토리지 버전 관리

- 작품 저장소에는 **버전 식별자**가 있다. 값이 바뀌면 부팅 시 작품 데이터를 새 기본값으로 다시 설정한다.
- 사용자 데이터 변환 체계는 **런칭 전 백엔드 이관 시** 한 번만 실행되는 데이터 이전 작업으로 처리([§31 N-6](#31-런칭-전-미해결-항목)).

### 18.3 자동 정리(Deprecated)

### 18.4 백엔드 이관 (런칭 전 필수)

- **목표**: Supabase 같은 클라우드 백엔드 서비스(BaaS)로 이관.
- **우선 이관 대상**: 전시·인터랙션·인증·신고·어드민 상태.

## 19. 접근성·시니어 친화 정책

### 19.1 필수 원칙

1. **터치 타깃 ≥ 44px** — 모든 인터랙티브 요소에 최소 44×44px.
2. **입력 폰트 ≥ 16px** — iOS Safari 자동 줌 방지.
3. **글꼴 크기 사용자 조절 3단** — 작게(100%) / 보통(110%) / 크게(120%).
4. **확인 다이얼로그 일원화** — 브라우저 네이티브 confirm·alert 금지. 공통 확인 다이얼로그(CM-01) 사용.
5. **색만으로 정보 전달 금지** — 상태(승인/반려/정지)는 아이콘 + 라벨 동반.
6. **호버는 데스크톱만** — 포인터 정밀·뷰포트 1024px 이상에서만 호버 효과. 모바일·태블릿에선 호버 효과 없음.
7. **애니메이션 짧고 느리게** — 200ms 내외. 번쩍이는 효과 금지.
8. **포커스 링 시각화** — 키보드 포커스는 반드시 가시화한다. 가시성 임계는 두께 **3px** 이상. 시각 표현 기법은 본 정책의 범위 밖.

### 19.2 금지된 UI 패턴

| 패턴 | 사유 |
|---|---|
| 스프레드시트형 UI | 시니어 인지 부하 과다 |
| 다중 선택(Shift + Click) | 오조작 위험 |
| 키보드 Tab 이동 전제 UX | 모바일·터치 우선 |
| `window.confirm()` · `window.alert()` 직접 호출 | 시각 일관성 + 커스터마이즈 불가 |
| `dangerouslySetInnerHTML` | XSS — 필요 시 sanitize 후 사용 |
| Pinch 줌 외 고난도 GestureAPI 의존 | 시니어 제스처 숙련도 편차 |

### 19.3 다국어

- UI 기본 한국어(KO), 영어(EN) 전환 가능.
- 모든 문자열은 다국어 훅 경유. **하드코딩 금지**.
- 언어 전환 반응성: 전환 시 즉시 리렌더가 트리거되도록 정적 스냅샷 함수를 렌더 시점에 직접 호출하지 말 것.

### 19.4 다크 모드

- **기본값**: **밝게(라이트)**. 시니어 사용자에게 가장 익숙·예측 가능하며, 작품 사진을 작가 의도한 색감 그대로 노출하는 데 유리. 시스템 환경설정(`prefers-color-scheme`) 자동 감지는 사용하지 않는다(시니어가 의도치 않은 다크 표시로 작품 색감 손해를 보지 않도록).
- **사용자 토글**: Settings(USR-STG-01)에 **밝게 / 어둡게 2옵션**. "시스템 따르기" 같은 추상적 라벨은 제공하지 않는다(시니어 친화).
- **선택 지속**: 사용자 선택은 로컬 영속(다음 방문 시에도 유지). 서버 동기화는 백엔드 연동 후.
- **작품 표시 영역은 다크 모드에서도 라이트 매트(흰 배경) 유지**: 카드 썸네일·전시 상세의 작품 이미지 표시 영역은 어떤 모드에서도 흰색 매트를 유지한다. 외곽 UI(헤더·배경·텍스트·버튼)만 다크 토큰을 적용한다. 미술관에서 검은 벽에 걸린 흰 매트 액자 메타포로, 작가가 의도한 색감을 다크 모드 사용자에게도 정확히 전달하기 위함이다. 다크 모드의 본래 효용(눈 피로 감소·야간 사용)은 외곽 UI 다크화로 충분히 달성된다.
- **시각 토큰**: 라이트·다크 각각의 컬러·콘트라스트 토큰은 개발팀 보유 디자인 시스템이 제공한다. 본 폴더는 토큰을 정의하지 않는다(작품 영역 라이트 매트 원칙은 강제, 매트의 정확한 색·padding·border 등 시각 디테일은 디자인 시스템 영역).

## 21. 법무 체크포인트

런칭 전 법무 검토가 필수인 문서·정책을 단일 목록으로 정리한다. 각 항목은 USR-INF·Policy 섹션과 연결된다.

### 21.0 법무 문서 단계별 마감 (운영 가드)

법무 문서는 단순히 "런칭 전"이 아니라 **사용 단계별로 마감 시점을 분리**한다. 시니어 사용자는 약관을 실제로 읽는 경향이 있어 placeholder 약관 상태에서 사용자가 모이면 신뢰도가 즉시 하락한다.

| 단계 | 시점 | 약관·개인정보처리방침 상태 허용 범위 |
|---|---|---|
| **개발·내부 테스트** | 현재 ~ 모집 직전 | **placeholder 허용** (검수·QA·내부 시연 목적) |
| **사용자 모집 시작** | 외부 베타·CBT 시작 시점 | **법무 확정본 필수**. placeholder 상태로 모집 진입 금지. |
| **정식 런칭** | Phase 1 정식 오픈 | 법무 확정본 + 변경 이력·시행일 명시 + DPO 지정·공개 완료 |

운영 가드:
- 사용자 모집 시작 전, 본 §21.1 표의 L-1·L-2 상태가 "확정"이 아니면 ADM-CKL-01 런칭 체크리스트에서 **블로커**로 표시한다(자동 체크).
- 어드민 대시보드(ADM-DSH-01)에 "법무 확정본 미완료" 경고가 모집 시점부터 떠야 한다.
- 사용자 모집 후 약관·처리방침 변경이 생기면 §1 법적 의무 알림 채널(이메일)로 모든 회원에게 변경 고지.

| 승인 후 소급 숨김 | 운영팀 재량으로 비공개 전환 가능 | 사용자에게 고지 필수 |

**작가 안내 문구**(업로드 발행 직후 노출): "주말·공휴일을 빼고 24시간 안에 공개돼요. 결과는 알림으로 알려드릴게요."

**검수 큐 SLA 배지**(ADM-REV-01): **2단계** — 정상 / 시한 초과(24영업시간 경과). 자동 비공개 큐의 4단계 배지(§12.2.1)와 별개로, 일반 검수는 단순 2단계만 둠(긴급도가 다름).

### 22.2 신고 SLA

| 항목 | 시한 | 비고 |
|---|---|---|
| 신고 처리 시한 | **영업일 24시간 이내** 운영팀 판정 | 사유 무관 동일 SLA |

자동 비공개(§12.2 — 같은 전시 2번째 신고)는 신고 접수와 동시에 즉시 발동되어 노출이 차단된다. 시간 흐름에 따른 피해 확산은 자동 비공개가 막고, 운영팀 판정은 영업일 24시간 내 작가 회복(기각 시 복원) 또는 확정 처리(삭제·비공개 유지)를 결정하는 절차다.

### 22.3 이의제기 절차

Phase 1은 작품 단위 모더레이션만 다루므로 정지 이의제기 절차가 없다(§12.3). 작품 처리 결과(삭제·비공개 유지)에 대한 문의는 USR-INF-07(카테고리: 신고·기타) 채널로 접수하고 운영팀이 일반 문의 SLA에 따라 응답한다. 계정 단위 이의제기는 추후 별도 로드맵에서 재설계한다.

### 22.4 어뷰즈 방지

- **반복 신고 차단**: 동일 `(신고자, 대상유형, 대상ID)` 조합 1회(§12.4).
- **신고 레이트 리밋**: 단일 사용자의 동시 다발 신고(분당 10건 이상)는 자동 보류 후 운영팀 검토.
- **배너·이벤트 활성 수 제한**: 배너 최대 5, Pick 최대 10 — 운영 부하 억제.

### 22.5 에스컬레이션 기준

| 상황 | 대응 | 경로 |
|---|---|---|
| 동일 계정 24시간 내 신고 5회 이상 | 해당 사용자 계정 검토 큐 진입 (추후 제재 체계 검토용) | ADM-MBR-01 |
| 특정 전시에 24시간 내 신고 10회 이상 | 이미 §12.2 2회 자동 비공개 상태. 긴급 큐 우선순위 상향 | ADM-RPT-01 · ADM-REV-01 |
| 운영 판단 필요 케이스(법적 이슈·대외 민감) | 긴급 이슈로 승격 | ADM-ISU-01 |
| 시스템 장애·데이터 손실 의심 | 체크리스트 차단됨 처리 + 복구 절차 | ADM-CKL-01 |

### 22.6 운영 데이터 보관 기준

| 데이터 | 보관 | 파기 |
|---|---|---|
| 탈퇴 작가 식별 정보 | 재가입 방지 목적의 해시만 유지 | **런칭 전 백엔드 이관 시** 보관 기간 명문화 |

### 22.7 운영자 감사 로그 원칙

모든 어드민 액션(검수 판정·신고 4액션·제재·큐레이션·배너·이벤트·공지·문의)은 `admin_audit_log`에 즉시 기록된다. 스키마·보존·조회 권한은 [PRD_Admin §0.6](PRD_Admin_v1.md#06-감사-로그-audit-trail) 단일 소스. 요약:

- **Append-only**: 수정·삭제 API 제공하지 않음. 운영자 본인도 자기 로그를 삭제 불가.
- **행위자 식별**: `actorId`·`actorRole`·`ip`·`userAgent`로 누가 언제 어디서 했는지 재구성 가능.
- **복원 기반**: 복구 불가 액션은 `targetSnapshot`에 실행 직전 필드값을 보관해 사후 복원 근거 제공.
- **조회 권한**: Phase 1은 Operator 전체. Phase 2는 **최고 운영자만** 열람 가능.
- **개인정보 마스킹**: `targetSnapshot` 포함 개인정보는 [PRD_Admin §0.4.3](PRD_Admin_v1.md#043-뷰어-read-원칙) 마스킹 규칙을 적용해 저장.

### 23.1 강사 표시 파생

- **판정 방법**: 사용자의 업로드 이력 중 강사 ON으로 발행된 작품이 1개라도 있으면 강사로 본다.
- **별도 저장 안 함** — 업로드 이력에서 매번 계산. 프로필에 강사 토글·플래그를 두지 않는다(§17.1).
- **영향**:
  - 프로필(USR-PRF-01)에 "수강생 작품" 탭(USR-PRF-07) 자동 노출.
  - 강사 ON 업로드를 모두 삭제하면 자동 비노출.
  - 강사 ON 업로드는 **본인 작품 포함 금지**, 강사 OFF 그룹 업로드는 **본인 작품 최소 1점 포함 필수**(§13).

> **상태 통합 결정(Phase 1)**: 전시는 본래 검수·신고 처리·자동 비공개가 따로 관리되어 운영팀이 "이게 왜 비공개인가"를 매번 조합으로 추론해야 했다. 이를 **단일 공개 상태 5종**(공개 / 검수 대기 / 반려 / 자동 비공개 / 비공개 유지)으로 통합한다([§31 N-11](#31-런칭-전-미해결-항목)).

### 23.2 검수 상태 전환

| 트리거 | 전환 |
|---|---|
| 신규 업로드 | (상태 없음) → **검수 대기** (자동 승인 환경에선 즉시 **승인**) |
| 운영팀 승인 | 검수 대기 → **승인** + 팔로워 알림 발송 + **보류된 비회원 초대 발송**(§3.1) |
| 운영팀 반려 | 검수 대기 → **반려** + 반려 사유 1종 필수 기록 |
| 편집 시 이미지 변경(이미지 추가·삭제·교체·참여 작가 슬롯·커버 직접 변경) | 원본 상태 무시하고 **검수 대기**로 재설정 → 재검수 |
| 편집 시 메타만 변경(전시명·그룹명·작품명·커버 인덱스·이벤트 연결) | 원본 상태 유지 |
| 반려 후 수정·재제출 | 반려 → **검수 대기** (재검수) |

### 23.3 참여 작가 슬롯 상태 전환

전시의 각 작품(이미지)마다 1명의 참여 작가 자리가 있다. 자리는 다음 세 가지 상태를 갖는다.

| 슬롯 상태 | 조건 |
|---|---|
| **회원 자리** | 업로더가 회원을 직접 지정했거나, 비회원 초대가 회원 가입 후 본인 확인을 거쳐 승격된 결과(§3.5) |
| **비회원 초대 자리** | 비회원에게 초대만 보낸 상태. 검수 승인 후 발신 식별자(전화번호·이메일)는 시스템에서 흔적 없게 정리됨(§3.4) |
| **작가 미상 자리** | 사후 해제(마이페이지 "본인 작품 아님") 결과(§3.5.3) 또는 자리 주인이 탈퇴한 경우(§4.1). 이름·연락처는 모두 제거됨 |

- **전환**: 비회원 초대 → 회원 자리(가입 후 본인 확인 시), 회원 자리 → 작가 미상(피초대자 사후 해제·탈퇴).
- **역전환 없음**: 작가 미상 자리는 자동으로 다시 채워지지 않는다. 업로더가 전시 편집에서 새로 회원·비회원으로 재지정해야 복구된다.

### 23.4 자동 비공개 (전시 단위)

- 트리거: 같은 전시에 **2번째 신고**가 접수되는 순간(§12.2).
- 동작: 즉시 자동 비공개 처리 → 둘러보기·검색·피드 비노출. 작가 본인 프로필에서만 보인다.
- 알림: 작가에게 시스템 알림 1건 발송(§12.1.3).
- 해제: 운영팀 큐에서 "기각" 판정 시 자동 해제(복원). 기타 액션(삭제·비공개 유지)은 해당 상태로 확정.

### 23.5 이미지 처리 파이프라인

업로드 1장당 거치는 검증·변환 순서. 정책상 **카메라 사진 차단**·**해상도 하한**·**이미지 수 상한**만 강제하고, 변환·인코딩 방식은 개발자 결정 영역.

| 단계 | 정책상 강제 |
|---|---|
| 파일 크기 상한 | 10MB(§20) |
| 허용 형식 | JPEG·PNG·WebP·GIF 외 거부 |
| 카메라 사진 차단 | EXIF에 카메라 정보가 남아 있으면 업로드 거부(§11) |
| 해상도 하한 | 짧은 변(width·height 중 작은 값) 800px 미만 거부(§20) |
| 이미지 수 상한 | 전시 1건당 10장 |

- **검증 순서가 결과에 영향을 미치는 부분만 고정**: 카메라 사진 차단 검증은 이미지 변환 **전**에 수행되어야 EXIF가 살아 있어 검출 가능. 이를 제외한 압축·인코딩·캐싱 방식은 개발자 결정.
- **프로필 이미지**: 5MB·원형 크롭(별도 경로).
- **커버 톤 배경**: 작품 이미지를 흐리게·확대·반투명으로 깔아 CSS만으로 톤 배경을 구현. 색상 추출 알고리즘은 사용하지 않음.

### 23.6 그룹명·자동완성 파생

- **사용자 최근 그룹명**: 본인 업로드 이력에서 추출. 상한 50개 (§20).
- **중복 허용**: 2026-04-17 이후 그룹명 정규화·캐논 맵 폐기. 동일 문자열은 동일 그룹으로 간주하지 않음 — 표시 문자열만 사용.
- **자동완성 순서**: 빈 쿼리 시 내 최근 → 전체 작품 그룹명 (상한 30). 쿼리 있을 시 상한 20.

### 23.7 포인트 일일 상한·월 상한

| 이벤트 | 상한 | 재설정 기준 |
|---|---|---|
| 일반 업로드 +20 | 일 **2회** | 로컬 자정 |
| 일일 둘러보기 +5 | 일 **1회** | 로컬 자정 |
| 그룹 전시 개설(강사) +30 | 월 **1회** (강사-월 조합) | 월 1일 |
| 그룹 참여(비강사) +15 | 월 **5회** | 월 1일 |
| 업로드 AP 회수 -20 | 단건 | 업로드 후 24시간 이내 삭제 |

- 첫 업로드 +100은 일 상한 **무관**(당일 "일반 업로드 +20" 카운터에 포함되지 않음).
- 포인트 원장 FIFO 500건 유지.

### 23.8 검색 쿼리 전처리

- 쿼리 **trim** 후 공백 1개로 치환(연속 공백 합침).
- 대소문자 비구분 매칭(현지화 locale 별 case folding).
- 숫자만 입력된 쿼리도 정상 처리(ID/전화번호 검색 아님 — 콘텐츠 필드 대상).
- 빈 쿼리·1자 이하는 자동완성만 표시, 결과 페이지 이동 금지.

## 24. 인터랙션 정책 (좋아요·저장·팔로우)

### 24.1 단위

- 좋아요·저장은 **전시(Work) 단위**. 개별 이미지(piece) 단위 인터랙션은 추후 별도 로드맵 검토.
- 팔로우는 **작가(Artist) 단위**.

### 24.2 공통 제약

| 조건 | 동작 |
|---|---|
| 비로그인 | CM-02(로그인 유도) 표시 후 로그인 완료 시 원래 의도된 액션으로 이어감 |
| 탈퇴 작가(§4.2) | 좋아요·저장·팔로우·신고 모두 조용히 무시 + 버튼 비활성(연하게 처리) |
| 자동·확정 비공개 전시(§12.2) | 본인 프로필 외에선 카드 자체가 노출되지 않아 인터랙션 경로 없음 |
| 본인이 신고한 대상 | 신고자 본인 한정으로만 피드·검색·상세 모달에서 자동 숨김(§12.4) |

### 24.3 좋아요

- 단일 토글(좋아요/해제).
- **본인 작품 좋아요**는 기능상 허용되나 프로필 "좋아요" 탭(USR-PRF-08)에 노출되지 않음.
- 발생 시 작품 작가에게 시스템 알림 1건(탈퇴 작가 제외).
- **카운트 UI 비노출** — 전시 카드·상세 모달에 **상태 아이콘만** 표시. 누적 집계는 피드 랭킹(§16.1)에만 내부 가중치로 사용.

### 24.4 저장

- 좋아요와 동일 구조의 토글.
- **작가 알림 없음**(저장은 개인 북마크 성격).
- "저장" 탭(USR-PRF-09)은 본인만 열람. 본인 작품 저장은 허용되나 탭에서 숨김.
- 누적 집계는 피드 랭킹(§16.1)에만 내부 가중치로 사용.

### 24.5 팔로우

- **자기 팔로우 금지**: 본인 프로필에서 팔로우 버튼 자체 미노출.
- **탈퇴 작가 팔로우 금지**: 버튼 숨김 + 저장 단계에서도 동작 무시(이중 가드).
- 발생 시 대상 작가에게 시스템 알림 1건.
- 팔로워 마일스톤(10·50·100명) 달성 시 **본인**에게 AP 적립(§7.2).
- 팔로우 관계는 피드 **개인화** 버킷 가중치로 사용(§16.1).
- 팔로잉·팔로워 숫자는 프로필 헤더(USR-PRF-01)에만 노출. 탭 시 USR-PRF-04 모달.

### 24.6 동기화 범위

- Phase 1은 좋아요·저장·팔로우 기록을 사용자 단말 로컬에 보관. 같은 사용자가 다른 기기·브라우저에서 접속하면 기록이 공유되지 않는다(목업 한계).
- 런칭 전 백엔드 연동 시점에 계정 단위 서버 저장으로 전환 — 기기 간 동기화·로그아웃 후 복구 가능.

### 24.7 추후 유보

- 댓글·핀 코멘트·익명 인터랙션·리액션.
- 작품 단위(전시 안의 개별 이미지) 좋아요·저장·북마크.

## 25. 이벤트·응모 정책

### 25.1 이벤트 생성·관리

- 이벤트는 **운영팀만** 어드민(ADM-EVT-01·02)에서 생성·수정·삭제한다. 사용자 직접 생성 불가.
- 필수 항목: 제목·설명·시작·종료 일시·배너 이미지. 선택: 응모 규칙·상품·주최자.
- **응모작 공개 여부 토글**: 응모작을 이벤트 페이지에서 공개할지 여부. 기본은 비공개 응모, 선정 후 공개 전환 가능.

### 25.2 응모 조건

- 사용자는 본인 전시를 업로드·편집 중에 **하나의 이벤트에만** 응모할 수 있다.
- 응모 시점에 이벤트가 **진행 중**(현재 시각이 시작·종료 일시 사이)이어야 한다. 종료된 이벤트는 선택 불가.
- 응모한 전시는 **검수 승인(§12.1.2)** 이후 이벤트 목록에 노출(이벤트의 응모작 공개 토글이 켜진 경우).
- 동일 전시의 응모 이벤트 변경은 **언제든 가능**. 이벤트 연결은 메타 정보이므로 §23.2 규칙상 재검수는 **트리거되지 않는다**. 즉 승인된 전시의 이벤트 연결만 바꿔도 즉시 새 이벤트 목록에 반영된다.

### 25.3 응모작 노출

- 응모작 공개 토글이 **켜진 이벤트**: 응모작이 이벤트 상세 페이지(USR-EVT-02)에 공개. 검수 승인 + 자동·확정 비공개 아님 조건을 모두 만족해야 노출.
- 응모작 공개 토글이 **꺼진 이벤트**: 응모작은 운영팀만 어드민(ADM-EVT-03)에서 열람.

### 25.4 선정 시스템

- 이벤트 선정은 **Phase 1 범위**로 처리한다.
- 운영팀은 ADM-EVT-03 응모자 현황을 기반으로 선정작을 확정하고, 선정 결과는 이벤트 상세/관련 전시 노출에 반영한다.
- 자동 투표·추천 알고리즘 도입 여부는 추후 별도 로드맵으로 검토한다(필수 아님).

### 25.5 이벤트 종료

- 종료 일시 경과 시 이벤트는 자동으로 "종료" 상태 표시. 신규 응모 차단.
- 기존 응모작은 계속 목록에 노출(응모작 공개 토글 그대로 유지). 운영팀이 수동으로 비공개 전환 가능.

### 25.6 이벤트 삭제

- 운영팀 수동 삭제 시 복구 불가 확인 다이얼로그.
- 응모된 전시의 `linkedEventId`는 null로 일괄 정리(전시 자체는 유지).
- 삭제 사실은 감사 로그에 기록.

## 26. 이미지 저작권 보호 정책

Artier는 작가의 원본 작품을 보호한다. 완벽한 다운로드 차단은 불가능하지만, **우연한·일상적 복제**는 차단한다.

### 26.1 프런트 차단 (모든 플랫폼)

| 대상 | 차단 방식 |
|---|---|
| 우클릭 메뉴 | `oncontextmenu` 이벤트 차단(`preventDefault`) |
| 드래그·드롭 | `ondragstart` 차단. "이미지를 드래그할 수 없습니다" 조용한 실패(에러 토스트 없음) |
| 길게 눌러 저장(모바일) | `-webkit-touch-callout: none` + `user-select: none` CSS |
| 이미지 선택·복사 | `user-select: none` |

구현 경로: 모든 작품 이미지 노출은 **공통 이미지 컴포넌트(예: `CopyrightProtectedImage`) 단일 진입점**을 경유한다(개발 가이드).

### 26.2 피드·상세 이미지 해상도 제한

- 피드·카드: 썸네일은 **단변 최대 1200px** 제공. 원본 해상도(최대 수천 px) 직접 노출 금지.
- 상세 모달: 상세 뷰어는 원본 해상도 표시 가능(줌 뷰어 포함). 단, **다운로드 경로 차단**은 §26.1을 유지.

### 26.3 URL 노출 최소화

- 이미지 URL은 화면에 노출하지 않는다(개발자 도구로만 확인 가능).
- URL 끝의 원본 파일명은 난수 ID 기반이며 원본 파일명을 포함하지 않는다.

### 26.4 워터마크

- Phase 1은 워터마크 **미적용**(시니어 작가의 작품 감상 경험 우선).
- 추후 사용자 선택형 "내 작품에 워터마크 적용" 토글 검토.

### 26.5 저작권 신고 대응

- 저작권 침해 신고("다른 사람의 그림이에요", §12.0)가 접수되면 §12.2 자동 비공개 규칙에 따라 같은 전시 2번째 신고 시 즉시 노출 차단.
- 운영팀은 §22.2 단일 SLA(영업일 24시간)에 따라 판정.
- 저작권자 본인이 아닌 제3자 신고도 접수 가능하되, **운영팀 검토 후**에만 확정 조치.

### 26.6 허용되는 복제

- **스크린샷**은 기술적으로 차단 불가. 서비스 이용약관에서 **"개인 감상 목적 외 재배포 금지"**를 명시(§21 LP-7 참조).
- 작가 본인은 본인 작품을 자유롭게 내보낼 수 있어야 함. 추후 "내 작품 원본 내보내기" 기능 검토.

### 26.7 API·크롤링 대응

- 검색 엔진 크롤러는 `robots.txt`로 이미지 경로 분리 고려(§11 SEO 연동).
- 일반 스크래퍼 차단은 **런칭 전 백엔드 연동 후** rate-limit·봇 탐지로 대응(§31 런칭 전 미해결 항목 영역).

## 28. 다국어·locale 정책

### 28.1 지원 언어

- **Phase 1**: 한국어(`ko`) · 영어(`en`) 2종.
- 전체 i18n 사전은 [`Copy_v1.md`](Copy_v1.md) §6 단일 소스.

### 28.2 초기 locale 결정

로그인 이전·이후 모두 아래 순서로 결정:

1. 사용자가 설정에서 명시적으로 선택한 값 → `artier_locale` 저장 (최우선)
2. 저장값 없으면 브라우저 `navigator.language`:
   - `ko*` 접두 → 한국어
   - 그 외 → 영어
3. 런타임 중 변경은 `useI18n().setLocale()`만 허용. 스냅샷 함수 직접 사용 금지(개발 가이드).

### 28.3 전환 UX

- 헤더 언어 토글 버튼 또는 설정(USR-STG-01)에서 전환.
- 전환 즉시 전체 화면 리렌더(한글↔영문 실시간 교체). 새로고침 불필요.

### 28.4 미번역 키 fallback

- 누락된 키는 **키 문자열 자체**를 반환(예: `upload.err.fileTooBig`).
- 빌드·배포 전 누락 없는지 자체 린트로 확인(ko/en 키 수 동일).

### 28.5 추후 확장 후보

- 제3 언어(일본어·중국어) 추가
- 지역 기반 자동 locale 판정(브라우저 언어 설정 기반)
- 날짜·숫자 포매팅 라이브러리 도입(현재는 단순 표시)

## 30. 개인정보 열람·정정·삭제 요청 정책

개인정보보호법 제35~37조, GDPR Art.15~17(Right of Access / Rectification / Erasure)에 따라 사용자는 자신의 개인정보에 대해 **열람·정정·삭제·처리 정지**를 요청할 권리를 갖는다.

### 30.1 요청 접수 채널

- **Phase 1**: USR-INF-07 문의하기(`/contact`)의 **"개인정보 열람·정정·삭제 요청"** 카테고리로 접수. 사용자 이름·이메일·요청 유형·세부 내용을 입력한다.
- **Phase 2**: 설정(USR-STG-01) 내 전용 메뉴 제공 검토(셀프 서비스 열람·내보내기·삭제).
- 대체 경로: 개인정보보호 책임자(DPO) 이메일 직접 연락 — 약관·개인정보처리방침 푸터에 노출.

### 30.2 요청 유형 (Phase 1 수동 처리 전제)

| 유형 | 대응 |
|---|---|
| **열람** | 회원가입 정보·업로드 작품·인터랙션·알림 수신 설정 등 자신의 계정 데이터 확인. 운영팀이 본인 확인 후 JSON·PDF 형태로 회신. |
| **정정** | 잘못된 정보 수정. 이름·생년월일 등 식별 핵심 필드는 운영팀 본인 확인 후 수동 반영. Phase 2에서 본인 확인·자가 정정 플로우 재설계(§27.2). |
| **삭제** | 탈퇴(§4)는 셀프 가능. 탈퇴 후에도 남는 작품·작가 미상 라벨 관련해 추가 삭제 요청은 이 채널에서 접수. |

## 32. 데이터 연쇄 정리 정책 (참조 무결성)

전시 1건이 삭제될 때(사용자 자가 삭제 · 어드민 삭제 · 신고 처리 "삭제" 모두 동일) **해당 전시 ID를 참조하는 모든 부속 레코드를 함께 정리**한다. 정책상 원칙: stale 참조가 어떤 저장소에도 남아선 안 된다.

### 32.1 전시 삭제 시 정리 대상

| # | 대상 | 정리 내용 |
|---|---|---|
| 1 | 전시 본체 | 전시 레코드 + 첨부 미디어 영구 삭제 |
| 2 | 포인트 원장 | 업로드 시각 레코드 제거. 업로드 후 24시간 이내 삭제면 AP −20 회수 (§7.2 회수 정책) |
| 3 | 인터랙션 | 모든 사용자의 좋아요·저장 배열에서 해당 전시 ID 제거. 작가 팔로워 수 델타는 유지 |
| 4 | 알림 | 해당 전시 ID를 참조하는 알림 레코드 제거 |
| 5 | 신고 큐 | 해당 전시를 대상으로 한 신고 레코드 제거 |
| 6 | 신고 중복 서명 | `<신고자>\|work\|<전시ID>` 서명 제거 (재신고 가능 상태로 환원) |
| 7 | 신고자 숨김 | 모든 신고자의 숨김 목록에서 해당 전시 ID 제거 |
| 8 | Pick / 기획전 | 현재 Pick 배열·이력 배지 / 기획전 묶음의 전시 ID 배열에서 제거 |

### 32.2 운영 원칙

- **모든 삭제 트리거가 동일한 정리 절차를 거친다**: 사용자 자가 삭제 · 어드민 삭제 · 신고 "삭제" 처리 모두 §32.1 표의 모든 대상이 정리되어야 한다.
- **부분 정리 금지**: 일부 항목만 정리되고 나머지가 남는 상태(stale 참조)가 발생해선 안 된다.
- **이벤트 ↔ 전시 연결**: 단방향(전시 → 이벤트, `linkedEventId` 필드)으로만 보관. 따라서 전시 삭제 시 별도 이벤트 측 정리는 없다(전시 본체와 함께 자연 소거). **이벤트 삭제** 트리거에서는 전시 측의 `linkedEventId`를 비워준다(단, 작품 자체는 보존).

## 33. 작품 단위 운영팀 문의 정책

사용자가 특정 작품에 대해 운영팀에 문의를 보낼 수 있는 채널. **작가에게 자동 전달되지 않는다** — 시니어 작가의 답변 부담을 만들지 않으면서, 운영팀이 어떤 종류의 문의가 들어오는지 데이터를 수집해 Phase 2 BM 가설을 검증하는 목적.

### 33.1 카테고리 (5종)

사용자가 작품 문의 모달에서 선택하는 카테고리. 카테고리 분포가 곧 BM 인사이트.

| 라벨 | 한 줄 설명 | 비엠 가설 |
|---|---|---|
| **작품 구입·소장 문의** | 이 작품을 구입하거나 소장할 수 있나요 | 디지털 작품 직거래·프린트 판매 |
| **라이선스·사용 문의** | 상업 사용·프린트·NFT 라이선스가 가능한가요 | 라이선스 시장 |
| **전시 협업·의뢰** | 오프라인 전시·기관 협업 의뢰 | 갤러리 중개 |
| **작품 정보 문의** | 도구·기법·원본 사이즈 등 정보 | 단순 호기심·교육 수요 |
| **그 외 (직접 적기)** | 100자 이내 자유 입력 | 카테고리 외 신규 BM 후보 발견 |

### 33.2 처리 흐름

1. **사용자**: WorkDetailModal 더보기 메뉴(...)의 "이 작품에 대한 문의" 진입 → 카테고리 + 메시지(200자) 입력 → 제출.
2. **로그인 필수**: 비로그인 사용자는 CM-02 로그인 유도 모달(§19·§USR-AUT-02). 어뷰즈 방어 + 문의자 정보 정확도.
3. **운영팀 큐**: ADM-INQ-01 문의함에 "작품 문의" 카테고리로 적재. workId·pieceIndex·문의자·카테고리·메시지가 함께 표시.
4. **작가 자동 전달 안 함**: 작가에게는 알림이 가지 않는다. 운영팀이 데이터를 모아 PM에게 인사이트를 정기 보고하고, 가치 있는 문의(예: 동일 작가에게 거래 의뢰가 다수 누적)에 한해 운영팀이 별도로 작가에게 정리해 전달.

### 33.3 답변 정책

| 카테고리 | 처리 |
|---|---|
| 작품 구입·소장 / 라이선스·사용 | Phase 1엔 직접 거래 기능이 없음. **표준 응답** 발송: "현재 Artier는 직접 거래 기능을 제공하지 않아요. 작가 프로필의 외부 링크(SNS·웹사이트·이메일)로 직접 연락해보세요." |

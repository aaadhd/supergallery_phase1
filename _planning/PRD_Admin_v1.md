## 1. ADM-DSH · 대시보드

### ADM-DSH-01 · 대시보드

**목적**: 런칭 준비 상태와 운영 건강성을 한 장으로 요약해 보여주기.
**트리거**: 어드민 진입 기본 경로(`/admin`), 사이드바 1순위.
**우선순위**: **P0**

#### 입력
- 작품 검수 상태 분포(pending·rejected 개수, rejectionHistory 누적 수)
- 신고 큐 미결 건수
- 경고 카운터 1회 이상 누적 회원 수
- 미결 이슈 수·체크리스트 진행률·파트너 현황
- 블로커(차단됨 체크리스트·긴급 이슈)

#### 구성 (두 개 섹션으로 분리)

**1) 콘텐츠 운영** — 운영자가 매일 확인하는 지표
- **검수 대기 카드** (→ `/admin/content-review`) — pending 건수 + 재검수(rejectionHistory > 0) 건수 인라인 표시 + "SLA 24시간"
- **반려 상태 카드** (→ `/admin/content-review?status=rejected`) — rejected 건수 + "작가 수정 대기 중"
- **미결 신고 카드** (→ `/admin/reports`) — pending 신고 건수 + "긴급 건 24시간 SLA"
- **경고 누적 회원 카드** (→ `/admin/members`) — 경고 1회 이상 회원 수 + "3회 누적 시 자동 정지"

**2) 런칭 준비** — 기존 지표 유지
- 미결 이슈 / 체크리스트 % / 파트너 작가 / 파트너 작품 / 블로커 경고 / 카테고리별 진행률 / 최근 업데이트

#### 수용기준
- AC-01: Given 체크리스트 중 "차단됨" 1건 / When 대시보드 로드 / Then 상단 블로커 경고 + 해당 항목 링크.
- AC-02: Given 검수 대기 23건(재검수 5건 포함) / When 로드 / Then 카드에 "23" + 서브 "재검수 5건 포함 · SLA 24시간" 문구 노출.
- AC-03: Given 모든 지표 0 / When 로드 / Then 카드 수치는 0이지만 카드 자체 비활성화 없이 표시.
- AC-04: Given "반려 상태" 카드 탭 / When 클릭 / Then ADM-REV-01에서 `?status=rejected` 필터 적용 상태로 이동.
- AC-05: Given 신고가 실시간 추가(다른 탭·사용자) / When `REPORTS_CHANGED_EVENT` 수신 / Then 미결 신고 카드 수치 자동 갱신(새로고침 없이).

#### 의존
- 엔티티: EXHIBITION, UNRESOLVED_ISSUE, LAUNCH_CHECK_ITEM, PARTNER_ARTIST, REPORT, SANCTION(경고 카운터)
- 연결 화면: ADM-REV-01, ADM-RPT-01, ADM-MBR-01, ADM-ISU-01, ADM-CKL-01, ADM-PTN-01, ADM-WRK-01

## 2. ADM-REV · 콘텐츠 검수

### ADM-REV-01 · 검수 큐

**목적**: 업로드된 전시를 상태별·기간별로 열람하고 상세 판정 진입점을 제공.
**트리거**: 사이드바 → 콘텐츠 검수. 대시보드 "검수 대기" 카드 탭.
**우선순위**: **P0**

#### 입력
- 전체 전시 중 검수 상태별 목록
- 필터: 상태(전체/대기/승인/반려), 업로드 기간(날짜 범위)
- 정렬: 업로드 시각 기본(최신순)

#### 구성
- **상단 필터 바**: 상태 토글(4종: 전체·대기중·승인·반려) + 기간 피커(시작~종료)
- **URL 동기화**: 상태 필터 ↔ `?status=all|pending|approved|rejected` 양방향 동기화(`replace` 모드, `all`은 쿼리 생략). 대시보드·외부 링크에서 특정 필터로 딥링크 가능(예: `/admin/content-review?status=rejected`).
- **대기 건수 배지**: 큐 상단에 "검수 대기 N건"
- **테이블 컬럼**: 썸네일 · 작품명 · 작가 · 업로드일 · 상태 · 액션(상세 진입)
- **재검수 배지**: 작품명 옆에 노출. `pending` 상태이면서 `rejectionHistory.length > 0`이면 "재검수 요청"(1회차) 또는 "재검수 N회차"(2회 이상) 앰버 배지. **클릭 시 팝오버로 이력 전체**(사유·일시·회차) 조회 가능 — 운영자가 과거 판정을 참고해 결정할 수 있도록 함.
- **빈 상태**: "조건에 맞는 항목이 없습니다" + 필터 초기화 버튼

#### 처리
1. 필터·기간 적용 후 페이지네이션 20건/페이지.
2. 행 탭 → ADM-REV-02 상세 모달.
3. 상태 배지 색: 대기 회색 · 승인 초록 · 반려 빨강.

#### 수용기준
- AC-01: Given 대기 23건 / When 기본 진입 / Then "검수 대기 23건" 배지 + 최신 업로드 순 20건.
- AC-02: Given 상태 "반려" 필터 / When 탭 / Then `rejected` 항목만 노출 + URL `?status=rejected`.
- AC-03: Given 필터 조합 결과 0건 / When 로드 / Then 빈 상태 UI + 필터 초기화 CTA.
- AC-04: Given `pending` + `rejectionHistory.length === 0` / When 행 렌더 / Then 재검수 배지 비노출.
- AC-05: Given `pending` + `rejectionHistory.length === 1` / When 행 렌더 / Then "재검수 요청" 앰버 배지 노출.
- AC-06: Given `pending` + `rejectionHistory.length === 3` / When 행 렌더 / Then "재검수 3회차" 앰버 배지 + 툴팁 "이전 반려 3회 후 수정되어 재검수 대기 중입니다".
- AC-07: Given `/admin/content-review?status=rejected` 직접 진입 / When 로드 / Then 반려 상태로 필터링 시작.
- AC-08: Given 상태 필터 "대기중" 선택 / When 선택 / Then URL이 `?status=pending`로 갱신되고 뒤로가기로 이전 상태 복원.

#### 엣지케이스
- EC-01: 다른 운영자가 처리한 항목이 큐에 남아있음 → 행 새로고침 버튼 노출.

#### 의존
- 엔티티: EXHIBITION
- 연결 화면: ADM-REV-02

### ADM-REV-02 · 검수 상세·판정

**목적**: 검수 대기 전시의 모든 이미지·메타를 확인하고 승인 또는 반려 판정.
**트리거**: ADM-REV-01 행 탭.
**우선순위**: **P0**

#### 입력
- 선택된 전시 전체 데이터(이미지 배열·커버·작가·그룹명·이벤트 연결·설명)
- 운영자의 판정 액션

#### 구성
- **이미지 캐러셀**: 전체 이미지 좌우 네비, 커버 슬라이드 포함
- **메타 패널**: 작가 프로필 링크, 그룹명, 이벤트 배지(연결 시), 설명, 태그
- **이미지별 작가 지정**: 회원/비회원 구분 표시, 비회원은 전화번호 마스킹
- **하단 CTA**: "승인" 녹색 · "반려" 빨강
- **이력 블록**: 과거에 반려·재검수된 이력(있을 경우)

#### 처리

**승인 액션**
1. `feedReviewStatus: 'approved'`로 전환.
2. 보류된 비회원 초대 목록을 알림 발송 큐에 추가([Policy §3.1](Policy_v1.md#3-비가입자-초대-정책)):
   - 한국 번호 → 카카오 알림톡(SMS 폴백)
   - 해외 번호 → 이메일
3. 팔로워(알림 설정 "팔로잉 신작" ON) 전원에 신작 알림 생성.
4. 토스트 "승인 처리되었습니다" + 큐 자동 갱신.

**반려 액션**
1. ADM-REV-03 반려 사유 모달 오픈.
2. 사유(4종: 저품질·스팸·부적절·저작권) 선택 필수.
3. 내부 메모(선택) 저장.
4. `feedReviewStatus: 'rejected'` + `rejectionReason` 기록.
5. **`rejectionHistory` 배열에 `{reason, rejectedAt}` 항목 append** — 작가가 수정 재발행해도 보존되는 이력([Policy §12.1.1](Policy_v1.md#121-검수-반려-사유-이력-보존)).
6. 보류된 비회원 초대 폐기.
7. 작가에게 시스템 알림 생성("전시명이 반려되었습니다 — 사유: …").
8. 토스트 "반려 처리되었습니다" + 큐 자동 갱신.

#### 수용기준
- AC-01: Given 승인 + 비회원 초대 2건 / When 확정 / Then 발송 큐 진입 + 팔로워 알림 생성(알림 설정 ON 대상).
- AC-02: Given 반려 + 사유 선택 / When 확정 / Then `rejectionReason` 기록 + 작가 알림 1건.
- AC-03: Given 동일 전시 이중 처리 시도 / When 다른 운영자가 먼저 처리 / Then 에러 "이미 처리된 항목입니다" + 큐 새로고침.
- AC-04: Given 작가가 반려 후 수정·재발행 / When 큐 재조회 / Then `pending`으로 재진입(기존 사유 제거).

#### 엣지케이스
- EC-01: 이벤트 연결 전시 반려 / When 이벤트 응모 자격 상실 / Then 이벤트 참여 이력에서 제거.
- EC-02: 삭제된 전시 상세 진입 시도 → "삭제된 전시" 안내 + 큐 복귀.

#### 의존
- 엔티티: EXHIBITION, INVITE, NOTIFICATION
- 정책: [Policy §12 신고·제재](Policy_v1.md#12-신고제재-정책) · [Policy §3 비가입자 초대](Policy_v1.md#3-비가입자-초대-정책) · [Policy §22.1 검수 SLA](Policy_v1.md#22-운영-리스크모더레이션-sla)
- 연결 화면: ADM-REV-03, ADM-WRK-01

### ADM-REV-03 · 반려 사유 선택 모달

**목적**: 반려 판정 시 사유를 일관된 라벨로 수집.
**트리거**: ADM-REV-02 "반려" CTA.
**우선순위**: **P0**

#### 구성
- **이전 반려 이력 미리보기** (조건부 — `rejectionHistory.length > 0`일 때만): 상단 앰버 박스에 최신순으로 사유·일시 리스트 노출. 재검수 판정 시 과거 판단 참고용.

- 확정 / 취소

#### 수용기준
- AC-01: Given 사유 미선택 / When 확정 시도 / Then 에러 "반려 사유를 선택해 주세요".
- AC-02: Given "저작권" 선택 + 내부 메모 / When 확정 / Then 기록 + 상위 모달로 반영.

#### 의존
- 엔티티: EXHIBITION.rejectionReason
- 정책: [Policy §12](Policy_v1.md#12-신고제재-정책)

## 3. ADM-PCK · Pick 관리

### ADM-PCK-01 · Artier's Pick 관리

**목적**: 매주 선정하는 Artier's Pick 관리(최대 10개). 이력 배지는 영구 유지.
**트리거**: 사이드바 → Pick 관리.
**우선순위**: **P0**

#### 입력
- 승인된 전시 중 Pick 후보 검색(전시명·작가명·기간 필터)
- 현재 Pick 목록(최대 10개)
- Pick 이력(배지 수여 누적)

#### 처리
1. 현재 Pick 섹션: 최대 10개 카드 + 해제 버튼.
2. 추가: 검색 결과에서 "+ Pick 추가" — 상한 10 초과 시 불가.
3. 선정 시:
   - 해당 전시에 `pick: true` + `pickBadge: true` 동시 부여
   - 작가에게 알림 1건
4. 해제 시: `pick: false`로 전환. `pickBadge`는 **영구 유지**(한 번 받은 이력 배지).
5. 히스토리 섹션: 최근 N주차 Pick 변화 로그.

#### 출력
- 현재 Pick 카드 그리드(최대 10)
- 추가·해제 테이블
- 히스토리 타임라인

#### 수용기준

- AC-02: Given Pick 선정 / When 확정 / Then 해당 전시 피드에서 Pick 버킷 진입 + 카드 배지 + 작가 알림.
- AC-03: Given Pick 해제 / When 확정 / Then `pick: false`로 전환되지만 `pickBadge` 배지는 유지.
- AC-04: Given 반려·비공개·삭제된 전시 / When Pick 추가 시도 / Then 차단.

#### 엣지케이스
- EC-01: Pick 중인 전시가 반려 전환되면 자동 해제 + 알림.
- EC-02: 20일 내 같은 전시 재선정 시도 — Phase 1 미구현(운영 수동). Phase 2에서 차단([Policy §15.3](Policy_v1.md#15-큐레이션배지-정책)).

### ADM-RPT-02 · 신고 상세·처리

**목적**: 신고 1건의 상세를 확인하고 4가지 처리 액션 중 하나 선택.
**트리거**: ADM-RPT-01 행 탭.
**우선순위**: **P0**

#### 입력

#### 구성
- 상단 메타: 신고자 정보(마스킹 이메일) · 신고 사유(5종 중 하나) · 상세 200자
- 본문: 대상 전시 이미지 캐러셀 또는 작가 프로필 미리보기
- 카운터 섹션: 경고 카운트 "N/3" · 허위 신고 카운트 "N/3"
- **4 액션 CTA**: 삭제(빨강·전시 한정) · 경고 · 기각 · 비공개(전시 한정)

#### 처리

**삭제 액션** (전시 신고 한정)
- ADM-RPT-03 파괴적 확인 다이얼로그 → 확정 시 전시 영구 삭제
- `adminStatus: 'deleted'`
- 연쇄 정리: 좋아요·저장·알림·Pick·기획전 참조 전부 정리
- 대상 작가에게 시스템 알림 "회원님의 전시 '{title}'이 신고 처리로 삭제되었습니다."([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))
- 에러: 작가 신고에 삭제 시도 → "작품 신고에 한해 삭제할 수 있습니다"

**경고 액션**
- 대상 작가 경고 카운터 +1
- `adminStatus: 'warned'`
- 3회 누적 시 자동 7일 정지 승격([Policy §12.2](Policy_v1.md#12-신고제재-정책))
- **자동 승격 임박 확인** — 대상 작가의 현재 경고 카운트가 2회 이미 누적된 상태에서 3회째 경고를 적용할 때 파괴적 확인 다이얼로그("경고 3회 누적 — 자동 7일 정지됩니다") 노출. 취소 시 액션 미실행.
- 대상 작가에게 시스템 알림: 기본 "신고가 접수되어 경고가 적용되었습니다. (누적 N/3회)" + 자동 승격 시 추가 알림 "경고 3회 누적으로 7일 정지되었습니다..." ([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))
- 에러: `targetArtistId` 없는 경우(데이터 결함) → "대상 작가 ID가 없어 경고를 적용할 수 없습니다"

**기각 액션**
- 신고자 허위 신고 카운터 +1
- `adminStatus: 'dismissed'`
- 3회 누적 시 신고자 자동 7일 차단
- **자동 차단 임박 확인** — 신고자의 현재 허위 신고 카운트가 2회 이미 누적된 상태에서 3회째 기각을 적용할 때 파괴적 확인 다이얼로그("허위 신고 3회 누적 — 신고자 7일 차단됩니다") 노출. 취소 시 액션 미실행.
- 신고자에게 시스템 알림: 기본 "접수하신 신고가 운영팀 검토 결과 기각되었습니다." + 자동 차단 시 추가 알림 "허위 신고 3회 누적으로 7일 차단되었습니다." ([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))

**비공개 액션** (전시 한정)
- `Exhibition.isHidden: true`
- `adminStatus: 'hidden'`
- 피드·검색·신고자 숨김 모두 비노출(작가 본인 프로필만 예외)
- 대상 작가에게 시스템 알림 "회원님의 전시 '{title}'이 신고 검토 결과 비공개 처리되었습니다." ([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))

#### 수용기준
- AC-01: Given 경고 처리 + 해당 작가 누적 3회 / When 확정 / Then 자동 7일 정지 + 정지 알림 + 큐 갱신.
- AC-02: Given 기각 처리 + 신고자 누적 3회 / When 확정 / Then 신고자 자동 7일 차단 + 알림.

- AC-05: Given 작가 신고에 삭제 시도 / When 확정 / Then 에러 토스트 + 삭제 미수행.

#### 엣지케이스
- EC-01: 이미 정지된 작가 대상 신고 → 경고 카운트만 누적, 단계 유지.
- EC-02: 신고자 계정이 탈퇴 / When 처리 시도 / Then 카운트 반영만 + 알림 스킵.

### ADM-MBR-02 · 회원 상세

**목적**: 회원 1명의 제재 지표·최근 신고를 확인하고 정지·해제 수행.
**트리거**: ADM-MBR-01 행의 이름 버튼 탭.
**우선순위**: **P0**

#### 입력
- 회원 기본 정보(이름·이메일·가입일·아바타)
- 현재 상태(활성/정지)
- 경고·허위 신고 카운터 (`sanctionStore.getWarningCount·getFalseReportCount`)
- 이 회원을 대상으로 한 최근 신고(`loadUserReports` 필터 상위 5건)

#### 구성 (Phase 1)
- **헤더**: 아바타 · 이름 · 이메일 · 가입일 · 닫기 버튼
- **제재 지표 3단** (그리드):
  - 상태(활성/정지) — 정지면 빨강, 활성이면 초록
  - 경고 N/3 — 2회 누적 시 빨간 배경 + "다음 경고 시 자동 7일 정지" 경고 문구
  - 허위 신고 N/3 — 2회 누적 시 빨간 배경 + "다음 기각 시 자동 7일 차단" 경고 문구
- **데모 사용자 고지** (조건부): "데모 사용자 — 정지 적용 시 전역 로그인 차단이 실제로 발동됩니다"
- **최근 받은 신고 섹션**: 상위 5건 리스트. 각 항목에 대상·사유·접수일·처리 상태(adminStatus).
  - 0건이면 빈 상태 박스 "이 회원을 대상으로 한 신고가 없습니다"
- **액션 CTA (상태별)**:
  - 활성: "정지" 버튼 → 상세 닫고 ADM-MBR-03(정지 단계 선택) 오픈
  - 정지: "해제" 버튼 → 즉시 해제(기존 플로우) + 상세 닫힘

#### Phase 2 예정
- 활동 카운트(작품·팔로워·좋아요·저장) — 백엔드 연동 후
- 제재 이력 타임라인(적용·해제 전체 기록) — 백엔드 이력 저장 후
- 이메일·전화 일부 편집

#### 수용기준
- AC-01: Given 이름 버튼 탭 / When 클릭 / Then 상세 모달 오픈, 제재 지표 3단과 최근 신고 노출.
- AC-02: Given 경고 카운트 2회 상태 / When 상세 로드 / Then 경고 카드 빨간 배경 + "다음 경고 시 자동 7일 정지" 경고 문구.
- AC-03: Given 대상 회원 대한 신고 0건 / When 상세 로드 / Then 빈 상태 박스 노출.
- AC-04: Given 상세에서 "정지" 탭 / When 클릭 / Then 상세 닫힘 + ADM-MBR-03 정지 단계 모달 오픈.
- AC-05: Given 정지 상태 상세에서 "해제" 탭 / When 클릭 / Then 즉시 해제 + 상세 닫힘 + 해제 토스트.
- AC-06: Given 데모 사용자 상세 / When 로드 / Then 지표 밑에 데모 고지 문구 노출.

#### 의존
- 엔티티: USER_PROFILE, SANCTION, REPORT
- 정책: [Policy §12](Policy_v1.md#12-신고제재-정책)
- 연결 화면: ADM-MBR-03

## 9. ADM-WRK · 작품 관리

### ADM-WRK-01 · 작품 관리

**목적**: 전시(작품 컨테이너) 검색·공개·삭제 관리.
**트리거**: 사이드바 → 작품 관리. ADM-REV-01·ADM-RPT-01 연계.
**우선순위**: **P0**

#### 입력
- 전시 검색·필터(상태·작가·기간·이벤트 연결 여부)
- 공개/비공개 토글(`isHidden`)
- 직접 삭제

#### 처리
1. 전시 목록 테이블(썸네일·작가·전시명·상태·업로드 시각).
2. 공개/비공개 토글(`isHidden`) — 피드·검색·신고자 숨김에서 즉시 반영.
3. 직접 삭제: 파괴적 확인 다이얼로그.
4. **연쇄 정리**: 삭제 시 아래 전부 일괄 정리(즉시 반영):
   - **메모리·영속화**: 전시 목록에서 제거, IndexedDB 미디어 삭제
   - **인터랙션**: 모든 사용자의 좋아요·저장 배열에서 해당 ID 제거
   - **알림**: 이 전시를 가리키는 모든 알림 레코드 제거
   - **신고**: 신고 큐에서 `targetId` 매칭 레코드 제거 + 신고 중복 서명·신고자 숨김 목록 정리
   - **큐레이션**: Pick 목록·기획전(theme.workIds) 배열에서 제거
   - **피드 상태**: "이미 본 작품" 기록에서 제거
   - **포인트**: 업로드 시각 레코드 제거(24h 이내 삭제면 AP -20 회수 선행)
   - 이벤트 연결(`linkedEventId`)은 **이벤트 삭제 시** 전시 측이 정리(반대 방향)

#### 출력
- 테이블 + 편집 액션
- 사용자 앱 피드·검색·프로필에 즉시 반영

#### 수용기준
- AC-01: Given `isHidden` ON / When 저장 / Then 피드·검색·타인 프로필에서 비노출, 본인 프로필엔 표시.
- AC-02: Given 직접 삭제 / When 확정 / Then 위 §4 연쇄 정리 8항목 모두 즉시 반영 + 토스트.
- AC-03: Given 삭제된 전시를 Pick에 포함하려 할 때 / When 시도 / Then 차단(이미 없음).

#### 의존
- 엔티티: EXHIBITION, INTERACTION, NOTIFICATION, REPORT, PICK, CURATION
- 연결 화면: ADM-REV-01, ADM-RPT-01, ADM-PCK-01, ADM-CUR-01

## 13. 어드민 공통 동작

### 13.1 감사·이력

모든 파괴적 액션(삭제·영구 정지)과 상태 변경(정지 단계 상승·기획전 비공개 전환·파트너 단계 변경)은 **이력에 기록**한다. Phase 1은 각 엔티티 내부의 `history` 필드에 스냅샷 저장. Phase 2에서 통합 감사 로그로 이관.

### 13.2 CSV 내보내기

회원·이벤트 참여자·파트너 작가·문의 내역 등은 **CSV 내보내기**가 가능하다(Phase 2). Phase 1은 UI만 노출하고 실제 다운로드는 모의.

### 13.3 운영 데이터 일괄 초기화

데모 환경에서 운영 데이터(배너·이벤트·Pick·기획전·파트너·체크리스트)를 스토리지 버전 변경 시 재시드한다. 운영 빌드에서는 수동 초기화가 필요하며, Phase 2 백엔드 연동 후 관리자 도구로 이관.

### 13.4 권한 관리 (Phase 2)

Phase 2 권한 3단계 분리 시 본 문서의 §0.4 권한 매트릭스가 세분화된다. 감사 로그와 함께 설계 예정([Policy §17.3](Policy_v1.md#17-어드민-접근역할-정책)).

- 데스크탑 1280px 우선의 운영 콘솔 톤(시니어 친화 44×44 규칙은 사용자 앱 한정)

```
운영 콘솔
├ 대시보드 (ADM-DSH)
│
├ ▸ 모더레이션
│   ├ 검수 큐 (ADM-REV)
│   └ 신고 처리 (ADM-RPT, 운영자 직권 신고 등록 포함)
│
├ ▸ 콘텐츠 운영
│   ├ Pick (ADM-PCK)
│   ├ 기획전 (ADM-CUR)
│   ├ 배너 (ADM-BNR)
│   └ 이벤트 (ADM-EVT)
│
├ ▸ 회원
│   └ 회원 관리 (ADM-MBR)
│
└ ▸ 운영
    ├ 문의함 (ADM-INQ)
    ├ 미결 이슈 (ADM-ISU)
    └ 런칭 체크리스트 (ADM-CKL, 정식 런칭 후 자동 제거)
```

폐기·통합으로 13 → 10개. 사이드바 라벨은 모두 명사로 통일("검수 큐", "신고 처리", "Pick", "기획전" 등 — "검수" 같은 동사 단독 라벨 회피).

#### 0.4.3 뷰어 READ 원칙

뷰어는 읽기만 가능하되, **민감 개인정보는 자동 마스킹**한다:

| 필드 | 뷰어 표시 |
|---|---|
| 이메일 | `fi****@example.com` (앞 2자 + 도메인만) |
| 전화번호 | `010-****-1234` (뒷 4자만) |
| 신고 상세 사유 본문 | 전문 표시(단, 작성자 개인정보는 위 규칙 적용) |

Phase 1에서는 뷰어가 없어 해당 규칙은 데이터 모델 단에서만 준비. Phase 2 권한 분리 시 UI 레이어에 마스킹 적용.

### 0.5 공통 비기능 요건

아래 SLO는 **런칭 전 백엔드 연동 후** 측정 기준. Phase 1 클라이언트 목업은 별도 SLO 없음.

| 항목 | P50 | P95 | P99 |

| 사이드바 네비 전환 | < 100ms | < 250ms | < 400ms |
| 테이블 로딩(100건 페이지 1장) | < 400ms | < 1000ms | < 2000ms |

### 0.6 감사 로그 (Audit Trail)

#### 0.6.1 기록 대상 액션

아래 액션은 수행 즉시 `admin_audit_log` 레코드를 생성한다. Phase 1은 클라이언트 기반이라 localStorage(`artier_admin_audit_log_v1`)에 기록하고, 런칭 전 백엔드 연동 후 서버 테이블로 이관한다.

| 분류 | 액션 | 기록 트리거 |
|---|---|---|
| **검수** | 승인·반려 | ADM-REV-02 판정 확정 |
| **신고 처리** | 삭제·기각·비공개 유지 | ADM-RPT-02 3 액션 각각 |
| **회원 제재** | (보류) | Phase 1 미사용 |
| **회원 계정** | 회원 영구 삭제(말소) | ADM-MBR-02 복구 불가 액션 |
| **콘텐츠** | 작품 영구 삭제 · 공개/비공개 토글 | ADM-RPT-01 (단일 모더레이션 큐) |
| **큐레이션** | Pick 선정·해제 · 기획전 생성·수정·삭제 | ADM-PCK-01 · ADM-CUR-01 |
| **배너·이벤트** | 배너 CRUD · 이벤트 CRUD · 선정작 지정·취소 | ADM-BNR-01 · ADM-EVT-01·02 |
| **공지·문의** | 공지 게시·게시중단 · 문의 답변·상태 변경 | ADM-NTC-01 · ADM-INQ-01 |
| **권한(Phase 2)** | 운영자 역할 부여·박탈 | (Phase 2 진입 시) |

#### 0.6.2 레코드 스키마

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string | 로그 ID (ULID 권장) |
| `actorId` | string | 액션 수행 운영자 ID |
| `actorRole` | string | 행위 시점의 역할 (Phase 1=`operator`, Phase 2=`super`/`editor`/`viewer`) |
| `action` | string | 액션 코드 (`review.approve`·`report.delete`·`member.suspend.perm` 등) |
| `targetType` | string | 대상 유형 (`work`·`artist`·`event`·`banner`·`pick`·`theme`·`notice`·`inquiry`) |
| `targetId` | string | 대상 고유 ID |
| `targetSnapshot` | object? | 액션 전 대상의 핵심 필드 스냅샷(복원 가능성 확보, 복구 불가 액션만) |
| `reason` | string? | 사유 메모(반려 사유·신고 처리 메모 등) |
| `metadata` | object? | 액션별 부가 데이터(예: 반려 사유 분류, 정지 기간 일수) |
| `ip` | string? | 운영자 IP (런칭 전 백엔드 연동 후 기록) |
| `userAgent` | string? | 운영자 브라우저 |
| `createdAt` | string | ISO 8601 타임스탬프 |

#### 0.6.3 보존·조회 정책

- **보존 기간**: 5년 (법정 감사 대응 상한 참고)
- **조회 권한**: Phase 1은 Operator 전체 조회. Phase 2는 **최고 운영자**만(에디터·뷰어 불가).
- **개인정보 포함 처리**: `targetSnapshot`에 이메일·전화가 들어가면 §0.4.3 마스킹 규칙 적용한 상태로 저장.
- **삭제·변조 불가**: append-only. 수정·삭제 API 제공하지 않음.
- **열람 화면**: Phase 2에 ADM-LOG-01 "감사 로그" 신설 예정(현재는 백엔드 DB 직접 조회로 대응).

#### 0.6.4 Phase 1 → 백엔드 이관

Phase 1 localStorage 감사 로그는 데모용이라 **런칭 전 백엔드 연동 시점에 이관되지 않는다**(데모 기간 기록은 실사용이 아니므로 폐기). 런칭 후부터의 액션이 감사 로그로 누적된다.

> **확장 규칙**: 새 액션 추가 시 본 표에 행 추가 + PRD_Admin 문서 이력 행에 명시. 액션 코드는 폐기되더라도 enum에서 제거하지 않고 `(deprecated)` 주석 유지(과거 로그 해석 보장).

#### 0.6.6 호출 계약 (`auditLogStore`)

스토어 API 시그니처·구현은 개발팀 영역(코드의 `auditLogStore`). 어드민 화면 코드는 액션 직후 1줄 호출로 기록한다:

```ts
auditLog({
  action: 'report.delete',
  targetType: 'work',
  targetId: work.id,
  targetSnapshot: { title: work.title, artistId: work.artistId, /* … */ },
  reason: '저작권 침해',
  metadata: { reportCount: 3 }
});
```

- 액션 실패 시(예: 스토어 mutation 실패) **로그를 먼저 기록하지 말 것** — 정상 mutation 직후에만 호출.
- 단, 실패도 추적이 필요한 액션(예: `member.delete` 실패)은 별도 `metadata: { result: 'failed', error }` 와 함께 기록.

## 1. ADM-DSH · 대시보드

### ADM-DSH-01 · 대시보드

**목적**: 런칭 준비 상태와 운영 건강성을 한 장으로 요약해 보여주기.
**트리거**: 어드민 진입 기본 경로(`/admin`), 사이드바 1순위.
**우선순위**: **P0**

#### 입력
- 작품 공개 상태 분포(공개·검수 대기·반려·자동 비공개·운영 비공개)
- 신고 큐 미결 건수 · 자동 비공개 전시 수
- 미결 이슈 수·체크리스트 진행률
- 블로커(차단됨 체크리스트·긴급 이슈)

#### 구성 (두 개 섹션으로 분리)

**1) 콘텐츠 운영** — 운영자가 매일 확인하는 지표
- **검수 대기 카드** (→ `/admin/content-review`) — pending 건수 + 재검수(rejectionHistory > 0) 건수 인라인 표시 + "SLA 24시간"
- **반려 상태 카드** (→ `/admin/content-review?status=rejected`) — rejected 건수 + "작가 수정 대기 중"
- **미결 신고 카드** (→ `/admin/reports`) — pending 신고 건수 + "긴급 건 24시간 SLA"
- **자동 비공개 카드** (→ `/admin/reports?auto=1`) — 2회 신고로 자동 비공개된 전시 수 + "검토 후 확정·복원"

**2) 런칭 준비** — 기존 지표 유지
- 미결 이슈 / 체크리스트 % / 블로커 경고 / 카테고리별 진행률 / 최근 업데이트

#### 수용기준
- AC-01: Given 체크리스트 중 "차단됨" 1건 / When 대시보드 로드 / Then 상단 블로커 경고 + 해당 항목 링크.
- AC-02: Given 검수 대기 23건(재검수 5건 포함) / When 로드 / Then 카드에 "23" + 서브 "재검수 5건 포함 · SLA 24시간" 문구 노출.
- AC-03: Given 모든 지표 0 / When 로드 / Then 카드 수치는 0이지만 카드 자체 비활성화 없이 표시.
- AC-04: Given "반려 상태" 카드 탭 / When 클릭 / Then ADM-REV-01에서 `?status=rejected` 필터 적용 상태로 이동.
- AC-05: Given 신고가 실시간 추가(다른 탭·사용자) / When `REPORTS_CHANGED_EVENT` 수신 / Then 미결 신고 카드 수치 자동 갱신(새로고침 없이).

#### 의존
- 엔티티: EXHIBITION, UNRESOLVED_ISSUE, LAUNCH_CHECK_ITEM, REPORT
- 연결 화면: ADM-REV-01, ADM-RPT-01, ADM-MBR-01, ADM-ISU-01, ADM-CKL-01

## 2. ADM-REV · 콘텐츠 검수

### ADM-REV-01 · 검수 큐

**목적**: 업로드된 전시를 상태별·기간별로 열람하고 상세 판정 진입점을 제공.
**트리거**: 사이드바 → 콘텐츠 검수. 대시보드 "검수 대기" 카드 탭.
**우선순위**: **P0**

#### 입력
- 전체 전시 중 검수 상태별 목록
- 필터: 상태(전체/대기/승인/반려), 업로드 기간(날짜 범위)
- 정렬: 업로드 시각 기본(최신순)

#### 구성
- **상단 필터 바**: 상태 토글(4종: 전체·대기중·승인·반려) + 기간 피커(시작~종료)
- **URL 동기화**: 상태 필터 ↔ `?status=all|pending|approved|rejected` 양방향 동기화(`replace` 모드, `all`은 쿼리 생략). 대시보드·외부 링크에서 특정 필터로 딥링크 가능(예: `/admin/content-review?status=rejected`).
- **대기 건수 배지**: 큐 상단에 "검수 대기 N건"
- **테이블 컬럼**: 썸네일 · 작품명 · 작가 · 업로드일 · 상태 · 액션(상세 진입)
- **딥링크**: 작품명 → 전시 상세 새 탭으로 열기(`/exhibitions/:id`). 작가명 → 회원 상세 모달(ADM-MBR-01 `?artist=<id>`). 검수 큐를 유지한 채 맥락을 빠르게 확인할 수 있게 함.
- **재검수 배지**: 작품명 옆에 노출. 검수 대기 상태이면서 반려 이력이 있는 경우 "재검수 요청"(1회차) 또는 "재검수 N회차"(2회 이상) 앰버 배지. **클릭 시 팝오버로 이력 전체**(사유·일시·회차) 조회 가능 — 운영자가 과거 판정을 참고해 결정할 수 있도록 함.
- **SLA 배지(2단계)**: 검수 대기 상태에 한해, 영업일 기준 24시간 경과 시 "**시한 초과**" 배지 표시(빨강). 그 미만은 배지 없음. 자동 비공개 큐(ADM-RPT-01)는 4단계 배지를 쓰지만 일반 검수는 긴급도가 낮아 2단계로 단순화. (Policy §22.1)
- **빈 상태**: "조건에 맞는 항목이 없습니다" + 필터 초기화 버튼

#### 처리
1. 필터·기간 적용 후 페이지네이션 20건/페이지.
2. 행 탭 → ADM-REV-02 상세 모달.
3. 상태 배지 색: 대기 회색 · 승인 초록 · 반려 빨강.
4. SLA 배지: 영업일 기준 24시간 경과 미처리 건만 "시한 초과" 빨강 배지(영업일 기준 — 주말·공휴일 제외).

#### 수용기준
- AC-01: Given 대기 23건 / When 기본 진입 / Then "검수 대기 23건" 배지 + 최신 업로드 순 20건.
- AC-02: Given 상태 "반려" 필터 / When 탭 / Then `rejected` 항목만 노출 + URL `?status=rejected`.
- AC-03: Given 필터 조합 결과 0건 / When 로드 / Then 빈 상태 UI + 필터 초기화 CTA.
- AC-04: Given `pending` + `rejectionHistory.length === 0` / When 행 렌더 / Then 재검수 배지 비노출.
- AC-05: Given `pending` + `rejectionHistory.length === 1` / When 행 렌더 / Then "재검수 요청" 앰버 배지 노출.
- AC-06: Given `pending` + `rejectionHistory.length === 3` / When 행 렌더 / Then "재검수 3회차" 앰버 배지 + 툴팁 "이전 반려 3회 후 수정되어 재검수 대기 중입니다".
- AC-07: Given `/admin/content-review?status=rejected` 직접 진입 / When 로드 / Then 반려 상태로 필터링 시작.
- AC-08: Given 상태 필터 "대기중" 선택 / When 선택 / Then URL이 `?status=pending`로 갱신되고 뒤로가기로 이전 상태 복원.
- AC-09: Given 검수 대기 상태이고 영업일 기준 24시간이 경과한 행 / When 큐 로드 / Then 행에 "시한 초과" 빨강 배지 노출.
- AC-10: Given 운영자가 승인 또는 반려 액션 실행 / When 완료 / Then 해당 행의 시한 초과 배지 즉시 제거.

#### 엣지케이스
- EC-01: 다른 운영자가 처리한 항목이 큐에 남아있음 → 행 새로고침 버튼 노출.

#### 의존
- 엔티티: EXHIBITION
- 연결 화면: ADM-REV-02

### ADM-REV-02 · 검수 상세·판정

**목적**: 검수 대기 전시의 모든 이미지·메타를 확인하고 승인 또는 반려 판정.
**트리거**: ADM-REV-01 행 탭.
**우선순위**: **P0**

#### 입력
- 선택된 전시 전체 데이터(이미지 배열·커버·작가·그룹명·이벤트 연결·설명)
- 운영자의 판정 액션

#### 구성
- **이미지 캐러셀**: 전체 이미지 좌우 네비, 커버 슬라이드 포함
- **메타 패널**: 작가 프로필 링크, 그룹명, 이벤트 배지(연결 시), 설명, 태그
- **이미지별 작가 지정**: 회원/비회원 구분 표시, 비회원은 전화번호 마스킹
- **하단 CTA**: "승인" 녹색 · "반려" 빨강
- **이력 블록**: 과거에 반려·재검수된 이력(있을 경우)

#### 처리

**승인 액션**
1. `feedReviewStatus: 'approved'`로 전환.
2. 비회원 슬롯이 1개 이상 있으면 초대 토큰을 활성화한다([Policy §3.1](Policy_v1.md#3-비가입자-초대-정책)). 회사가 외부 채널(SMS·알림톡·이메일)로 발송하지 않으며, 작가가 마이페이지의 "친구에게 알리기" 버튼으로 본인 채널에 직접 공유한다.
3. 팔로워(알림 설정 "팔로잉 신작" ON) 전원에 신작 알림 생성.
4. 토스트 "승인 처리되었습니다" + 큐 자동 갱신.

**반려 액션**
1. ADM-REV-03 반려 사유 모달 오픈.
2. 사유(4종: 저품질·스팸·부적절·저작권) 선택 필수.
3. 내부 메모(선택) 저장.
4. `feedReviewStatus: 'rejected'` + `rejectionReason` 기록.
5. **`rejectionHistory` 배열에 `{reason, rejectedAt}` 항목 append** — 작가가 수정 재발행해도 보존되는 이력([Policy §12.1.1](Policy_v1.md#1211-검수-반려-사유-이력-보존)).
6. 초대 토큰 비활성화(친구 링크 보존, 재승인 시 자동 활성화 — Policy §3.4).
7. 작가에게 시스템 알림 생성("전시명이 반려되었습니다 — 사유: …").
8. 토스트 "반려 처리되었습니다" + 큐 자동 갱신.

#### 수용기준
- AC-01: Given 승인 + 비회원 초대 2건 / When 확정 / Then 발송 큐 진입 + 팔로워 알림 생성(알림 설정 ON 대상).
- AC-02: Given 반려 + 사유 선택 / When 확정 / Then `rejectionReason` 기록 + 작가 알림 1건.
- AC-03: Given 동일 전시 이중 처리 시도 / When 다른 운영자가 먼저 처리 / Then 에러 "이미 처리된 항목입니다" + 큐 새로고침.
- AC-04: Given 작가가 반려 후 수정·재발행 / When 큐 재조회 / Then `pending`으로 재진입(기존 사유 제거).

#### 엣지케이스
- EC-01: 이벤트 연결 전시 반려 / When 이벤트 응모 자격 상실 / Then 이벤트 참여 이력에서 제거.
- EC-02: 삭제된 전시 상세 진입 시도 → "삭제된 전시" 안내 + 큐 복귀.

#### 의존
- 엔티티: EXHIBITION, INVITE, NOTIFICATION
- 정책: [Policy §12 신고·모더레이션](Policy_v1.md#12-신고모더레이션-정책-phase-1) · [Policy §3 비가입자 초대](Policy_v1.md#3-비가입자-초대-정책) · [Policy §22.1 검수 SLA](Policy_v1.md#22-운영-리스크모더레이션-sla)
- 연결 화면: ADM-REV-03, ADM-RPT-01

## 3. ADM-PCK · Pick 관리

### ADM-PCK-01 · Artier's Pick 관리

**목적**: 매주 선정하는 Artier's Pick 관리(최대 10개). 이력 배지는 영구 유지.
**트리거**: 사이드바 → Pick 관리.
**우선순위**: **P0**

#### 입력
- 승인된 전시 중 Pick 후보 검색(전시명·작가명·기간 필터)
- 현재 Pick 목록(최대 10개)
- Pick 이력(배지 수여 누적)

#### 처리
1. 현재 Pick 섹션: 최대 10개 카드 + 해제 버튼.
2. 추가: 검색 결과에서 "+ Pick 추가" — 상한 10 초과 시 불가.
3. 선정 시:
   - 해당 전시에 `pick: true` + `pickBadge: true` 동시 부여
   - 대상 작가에게 `type: pick` 시스템 알림 1건 — 문구는 [Policy §15.6](Policy_v1.md#156-pick-선정-알림-전송-왕복-피드백)
4. 해제 시: `pick: false`로 전환. `pickBadge`는 **영구 유지**(한 번 받은 이력 배지).
5. 히스토리 섹션: 최근 N주차 Pick 변화 로그.

#### 출력
- 현재 Pick 카드 그리드(최대 10)
- 추가·해제 테이블
- 히스토리 타임라인

#### 수용기준
- AC-01: Given Pick 10건 / When "+ 추가" 시도 / Then 에러 "Pick은 최대 10개입니다".
- AC-02: Given Pick 선정 / When 확정 / Then 해당 전시 피드에서 Pick 버킷 진입 + 카드 배지 + 작가 알림.
- AC-03: Given Pick 해제 / When 확정 / Then `pick: false`로 전환되지만 `pickBadge` 배지는 유지.
- AC-04: Given 반려·비공개·삭제된 전시 / When Pick 추가 시도 / Then 차단.

#### 엣지케이스
- EC-01: Pick 중인 전시가 반려 전환되면 자동 해제 + 알림.
- EC-02: 20일 내 같은 전시 재선정 시도 — Phase 1 미구현(운영 수동). 서버 스케줄러 도입 시 자동 차단([Policy §15.3](Policy_v1.md#15-큐레이션배지-정책)).

## 7. ADM-RPT · 신고 처리

### ADM-RPT-01 · 신고 큐

**목적**: 사용자 신고를 상태·유형·사유별로 열람하고 상세 처리 진입점 제공. 자동 비공개 건의 SLA(Policy §12.2.1) 준수 가시화.
**트리거**: 사이드바 → 신고 처리.
**우선순위**: **P0**

#### 입력
- 전체 신고 레코드
- 자동 비공개 발동 시각 (`autoHiddenAt: ISO 8601`, 2회 신고 트리거 시점 — `reportsStore` 또는 전시 메타에서 파생)
- 현재 시각(SLA 계산 기준)
- 필터: 상태(전체/대기/비공개/삭제/기각), 대상 유형(전체/작품/작가), 사유(5종), **SLA(전체 / SLA 초과 / 임박)**

#### 구성
- **필터 바**: 상태 · 대상 유형 · 사유 · 기간 · **SLA**
- **정렬 기본값**: 자동 비공개 발동 시각 오름차순(가장 오래된 미처리 건 먼저)
- **테이블**: 신고 대상(썸네일 또는 작가 아바타) · 유형 · 사유 · 신고일 · **자동 비공개 경과 시간** · 상태 · **SLA 배지** · 행별 상세 진입
- **신고 대상 컬럼의 작품(piece) 표기**: 다중 이미지 전시에서 사용자가 모달의 작품 선택 단계에서 명시 선택한 piece 인덱스가 함께 표시(예: "전시명 · 3번 작품"). 단일 이미지 전시는 piece 표기 생략. 자동 비공개·운영 처리 흐름 자체는 전시 단위 그대로(Policy §12.0·§12.2).
- **딥링크**: 신고 대상 셀 → 작품 신고는 전시 상세 새 탭으로, 프로필 신고는 회원 상세 모달(ADM-MBR-01 `?artist=<id>`)로 점프. 신고 큐를 유지한 채 콘텐츠·회원 컨텍스트 확인.
- **동일 대상 중복**: 집계 카운트 배지("N건 누적")
- **빈 상태**: "접수된 신고가 없습니다"
- **운영자 직권 신고 등록**(ADM-WRK 폐기 후속, 2026-04-26): 큐 상단 "+ 직권 신고 등록" 버튼. 클릭 시 모달 — 작품·회원 검색·선택 + 사유 4종 라디오 + 상세 메모(500자) → 운영자 본인이 신고자로 등록되어 큐에 추가. 사용자 신고와 동일하게 ADM-RPT-02에서 처리. 모더레이션 단일 큐 원칙.
- **"초대 매칭 거부" 카테고리 폐기** (2026-04-27 Policy v2.14 토큰 모델 전환): 가입자가 본인 작품을 직접 클릭해서 연결하는 모델로 전환되어, 매칭 후보 yes/no 게이트 자체가 사라졌다. 잘못 연결 시 가입자가 작가에게 직접 알리고 작가가 슬롯 편집에서 해제하는 방식이라 운영팀 큐에 별도 카테고리를 두지 않는다.

#### SLA 배지 규칙 (Policy §12.2.1 연동)

자동 비공개 발동(`autoHiddenAt`) 후 운영 판정 미완료 상태에 한해 행 배지를 표시한다.

| 경과 시간 | 배지 라벨 | 색상 | 동작 |
|---|---|---|---|
| 0~24h | (배지 없음) | — | SLA 목표 내 |
| 24~48h | "SLA 임박" | 노란 | 운영자 주의 환기. 정렬 우선순위 상향. |
| 48~72h | "**SLA 초과**" | 주황 | 우선 처리 유도. 행 배경 옅은 주황. |
| **>72h** | "**SLA 위반**" | 빨강 | 행 배경 빨강 강조 + 대시보드(ADM-DSH-01) 블로커 카드에 합산 노출. 운영 슬랙 알림 발송(런칭 후 백엔드 연동). |

- 영업일 기준 계산: 토·일·공휴일은 경과 시간에서 제외. Phase 1은 단순 경과 시간으로 시작, 백엔드 연동 시 영업일 캘린더 적용.
- 판정(삭제·기각·비공개 유지) 완료 시 배지 제거. 추후 동일 작품 재신고로 자동 비공개 재발동되면 새로 카운트 시작.

#### 수용기준
- AC-01: Given 상태 "대기" 필터 / When 탭 / Then pending 건만 노출.
- AC-02: Given 같은 `(신고자, 대상)` 반복 신고 / When 큐 로드 / Then 1건만 노출(중복 방지).
- AC-03: Given 대상이 삭제된 전시 / When 큐에 노출 / Then 회색 처리 + "삭제된 전시" 배지.
- AC-04: Given `autoHiddenAt`이 50시간 전 + 미처리 / When 큐 로드 / Then 행에 "SLA 초과" 주황 배지 노출.
- AC-05: Given `autoHiddenAt`이 80시간 전 + 미처리 / When 큐 로드 / Then 행에 "SLA 위반" 빨강 배지 + 대시보드 블로커 카운트 +1.
- AC-06: Given SLA 필터 "SLA 초과" / When 탭 / Then 48시간 이상 경과 + 미처리 건만 노출, `autoHiddenAt` 오름차순 정렬.
- AC-07: Given 운영자가 "기각" 액션 실행 / When 완료 / Then 해당 행의 SLA 배지 즉시 제거 + 작품 `isHidden: false` 복원.

#### 의존
- 엔티티: REPORT (필수 추가 필드: `autoHiddenAt`이 전시(EXHIBITION) 또는 첫 자동 비공개 트리거 신고에 기록)
- 정책: [Policy §12.2.1 SLA](Policy_v1.md#12-신고모더레이션-정책-phase-1)
- 연결 화면: ADM-RPT-02 · ADM-DSH-01

### ADM-RPT-02 · 신고 상세·처리

**목적**: 신고 1건의 상세를 확인하고 4가지 처리 액션 중 하나 선택.

- 자동 비공개 배너: 2회 신고 누적으로 자동 비공개 상태면 "이미 자동 비공개 상태" 배지 + "기각 시 즉시 복원" 힌트
- **3 액션 CTA**: 삭제(빨강·전시 한정) · 기각 · 비공개 유지(전시 한정)
- 목록 제거(레거시, 액션 없이 큐에서만 제거)

#### 처리

**삭제 액션** (전시 신고 한정)
- ADM-RPT-03 복구 불가 확인 다이얼로그 → 확정 시 전시 영구 삭제
- `adminStatus: 'deleted'`
- 연쇄 정리: 좋아요·저장·알림·Pick·기획전 참조 전부 정리
- 대상 작가에게 시스템 알림 "회원님의 전시 '{title}'이 신고 처리로 삭제되었습니다." ([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))
- 에러: 작가 신고에 삭제 시도 → "작품 신고에 한해 삭제할 수 있습니다"

**기각 액션**
- `adminStatus: 'dismissed'`
- 자동 비공개 상태였다면 **즉시 복원** (`isHidden` false, 피드·검색 재노출)
- 신고자에게 시스템 알림 "접수하신 신고가 운영팀 검토 결과 기각되었습니다." ([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))
- 복원 시 작가에게 추가 시스템 알림 "'{title}' 전시가 검토 결과 정상 복원되었습니다."
- Phase 1은 허위 신고 카운터를 두지 않는다(Policy §12.3).

**비공개 유지 액션** (전시 한정)
- `Exhibition.isHidden: true`
- `adminStatus: 'hidden'`
- 피드·검색·신고자 숨김 모두 비노출(작가 본인 프로필만 예외)
- 대상 작가에게 시스템 알림 "회원님의 전시 '{title}'이 비공개 처리되었습니다. 수정해 재검수를 요청할 수 있어요." ([Policy §12.1.3](Policy_v1.md#1213-신고-처리-알림-전송-왕복-피드백))
- 작가는 [Policy §12.1.2](Policy_v1.md#12-신고모더레이션-정책-phase-1)에 따라 수정 가능. 이미지 변경 시 `feedReviewStatus`가 `pending`으로 재설정되어 재검수 진입.

#### 수용기준
- AC-01: Given 같은 전시 2번째 신고 / When 접수 완료 / Then 즉시 자동 비공개 + 작가 알림 + 운영자 큐 표시.
- AC-02: Given 자동 비공개 상태 전시 기각 처리 / When 확정 / Then 즉시 복원 + 작가·신고자 알림 각각 발송.
- AC-03: Given 삭제 액션 / When 다이얼로그 / Then "되돌릴 수 없습니다" 문구 + 복구 불가 버튼.
- AC-04: Given 비공개 유지 / When 확정 / Then 피드·검색·신고자 숨김 전부 비노출, 본인 프로필만 노출.
- AC-05: Given 작가 신고에 삭제 시도 / When 확정 / Then 에러 토스트 + 삭제 미수행.

#### 엣지케이스
- EC-01: 이미 삭제된 전시 대상 신고 → 신고는 큐에 남지만 3 액션 중 "기각"만 유효.
- EC-02: 신고자 계정이 탈퇴 / When 처리 시도 / Then 알림 스킵. 처리는 정상.

## 8. ADM-MBR · 회원 관리

### ADM-MBR-01 · 회원 목록

**목적**: 가입한 모든 회원을 검색·필터·정렬하여 열람.
**트리거**: 사이드바 → 회원 관리.

#### 구성
- **검색 바**: 이름·이메일·전화 통합(부분 일치)
- **필터**: 가입 경로 · 강사 여부
- **테이블**: 프로필·이름·이메일·가입일·가입 경로·행별 "상세"

- **빈 상태**: "검색 결과가 없습니다"

> Phase 1은 계정 단위 제재(정지·경고 카운터)가 없다(Policy §12.3). 해당 필터·열은 추후 별도 로드맵에서 검토한다. 파트너 작가 트래킹은 운영팀이 외부 도구(스프레드시트·CRM)로 처리하며, 본 화면에는 포함하지 않는다(2026-04-26 결정).

#### 수용기준
- AC-01: Given 검색 "카테" / When 엔터 / Then 일치하는 회원만 표시.
- AC-02: Given 필터 "강사 여부 = 강사" / When 선택 / Then 강사 역할 업로드 이력이 있는 회원만 노출.

#### 의존

## 12.5 ADM-INQ · 문의함

### ADM-INQ-01 · 문의함

**목적**: 사용자(USR-INF-07)가 접수한 문의를 카테고리별로 처리. **개인정보 권리 행사 요청(Policy §30)** 우선 처리 큐 겸용.
**트리거**: 사이드바 → 문의함 / 대시보드 "미답변 문의 N건" 카드.
**우선순위**: **P0**

#### 입력
- 전체 `INQUIRY` 레코드
- 필터: 카테고리(USR-INF-07 정의 + **작품 문의**(Policy §33) 카테고리 추가), 상태(신규 / 처리 중 / 완료 / 보류), 기간, **개인정보 우선** 토글

#### 구성
- **상단 KPI 카드 4종**: 신규 N건 · 처리 중 N건 · **개인정보 요청 N건** · SLA 임박/위반 N건
- **필터 바**: 카테고리·상태·기간·우선 토글
- **테이블**: 접수 시각 · 카테고리 라벨(개인정보는 보라 강조) · 이름·이메일 · 본문 미리보기(60자) · 상태 · **SLA 배지** · 행 탭 시 상세 패널
- **작품 문의 행 표기**(Policy §33): 본문 미리보기 위에 "전시명 · N번 작품 · 세부 카테고리(구입·라이선스·전시 협업·작품 정보·그 외)" 한 줄 표시. 운영팀이 어느 작품·어떤 종류의 문의인지 즉시 인식.
- **정렬 기본값**: 카테고리=`privacy`인 신규 건 우선, 그 외는 접수 시각 오름차순.
- **검색**: 이메일·이름·본문 키워드.
- **빈 상태**: "접수된 문의가 없습니다".

#### 상세 패널 (행 클릭)
- 사용자 정보(이름·이메일·세션 가입 경로 표시 — 로그인 사용자였다면).
- 카테고리·접수 시각·SLA 잔여 시간.
- **본문 전문** + **첨부 파일 목록**(이미지 미리보기·PDF/문서 다운로드).
- **답변 입력 영역**:
  - 빠른 답변 템플릿 드롭다운(카테고리별 5~10종 — 개인정보·계정·업로드 등 표준 답변 사전 작성).
  - 자유 텍스트(최대 5000자).
  - 답변 발송 시 **사용자 이메일로 모의 발송**(Phase 1) → 런칭 후 실제 이메일 발송 서비스(SMTP) 연동.
- **상태 변경**: 신규 → 처리 중 → 완료 / 보류 (드롭다운).
- **개인정보 카테고리 추가 위젯**: "본인 확인 완료" 체크박스 + "처리 결과(열람·정정·삭제) 데이터 첨부" 업로드 슬롯. 처리 시한(접수+30일) 카운트다운 표시.
- **운영 메모**(내부 전용): 다른 운영자와 공유하는 메모. 사용자에게 노출 X.

#### SLA 배지 규칙

| 카테고리 | 정상 | 임박 | 초과 | 근거 |
|---|---|---|---|---|
| `privacy` | ≤24h 내 접수 확인 / ≤30일 처리 | 25일 경과 | 30일 경과 | Policy §30.3 + 개인정보보호법 §35~37 |
| 기타 | ≤5영업일 응답 | 4영업일 경과 | 5영업일 경과 | 운영 원칙 |

#### 처리
1. 사용자가 USR-INF-07에서 제출 → `INQUIRY` 저장.
2. 신규 건 알림: 운영팀 슬랙·이메일(런칭 후 백엔드 연동).
3. 운영자 답변 → `inquiry.reply` 감사 로그 기록 + 사용자 이메일 회신.
4. 상태 변경 → `inquiry.status_change` 감사 로그.
5. 완료 후에도 90일간 조회 가능, 그 이후 보관함 이동(법무 5년 보관은 백엔드).

#### 수용기준
- AC-01: Given 카테고리 `privacy` 신규 1건 / When 큐 로드 / Then 첫 행 노출 + KPI "개인정보 요청 1건" 강조.
- AC-02: Given `privacy` 신규 + 25일 경과 / When 로드 / Then "SLA 임박" 노란 배지.
- AC-03: Given `privacy` + 30일 경과 / When 로드 / Then "SLA 초과" 빨강 배지 + 대시보드 블로커 카운트 +1.
- AC-04: Given 기타 카테고리 + 5영업일 경과 / When 로드 / Then "SLA 초과" 배지.
- AC-05: Given 답변 발송 / When 완료 / Then `inquiry.reply` 감사 로그 1건 + 사용자 이메일 회신 + 상태 자동 "처리 중"으로 전환.
- AC-06: Given `privacy` 카테고리 답변 시 / When "본인 확인 완료" 미체크 / Then 발송 전 경고 모달 "본인 확인이 완료되지 않았습니다. 진행할까요?".
- AC-07: Given 첨부 파일 5MB 초과 / When 운영자가 응답 첨부 시도 / Then 에러 "5MB 이하만 가능합니다".

#### 의존
- 엔티티: INQUIRY (필수 필드: `id`·`name`·`email`·`category`·`message`·`attachments[]`·`status`·`createdAt`·`replies[]`·`assigneeId?`·`internalNotes?`·`privacy.subjectVerified?`·`privacy.responseAttachments?`)
- 정책: [Policy §30 개인정보 요청](Policy_v1.md#30-개인정보-열람정정삭제-요청-정책) · [Policy §22.6 운영 데이터 보관](Policy_v1.md#22-운영-리스크모더레이션-sla)
- 연결 화면: ADM-DSH-01 (블로커) · USR-INF-07 (사용자 진입)
- 감사 로그: `inquiry.reply` · `inquiry.status_change` (PRD §0.6.5)

## 13. 어드민 공통 동작

### 13.1 감사·이력

모든 복구 불가 액션(삭제·영구 정지)과 상태 변경(정지 단계 상승·기획전 비공개 전환)은 **이력에 기록**한다. 현재는 각 엔티티 내부의 `history` 필드에 스냅샷 저장. **런칭 전 백엔드 이관 시** 통합 감사 로그 테이블로 이관.

### 13.2 CSV 내보내기

회원·이벤트 참여자·문의 내역 등은 **CSV 내보내기**가 가능(후순위 개발). 현재는 UI만 노출하고 실제 다운로드는 모의.

### 13.3 운영 데이터 일괄 초기화

데모 환경에서 운영 데이터(배너·이벤트·Pick·기획전·체크리스트)를 스토리지 버전이 바뀌면 부팅 시 기본값으로 다시 설정한다. 운영 빌드에서는 수동 초기화가 필요하며, **런칭 전 백엔드 연동 후** 관리자 도구로 이관한다.

### 13.4 권한 관리 (Phase 2)

Phase 2 권한 3단계 분리 시 §0.4.2 표의 ● 분포가 그대로 활성화된다. 구현 순서:
1. 백엔드에 `operator_role` 필드 추가(enum: `super`/`editor`/`viewer`)
2. 모든 어드민 API가 역할 기반 가드 적용
3. 프론트에서 뷰어 역할일 때 §0.4.3 마스킹 규칙 적용
4. 감사 로그(§0.6)와 함께 권한 변경 이력을 별도 테이블에 기록

Policy §17.3 및 §0.4.1~0.4.3 참조.

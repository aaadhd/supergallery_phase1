# 1. 제품 개요

## 1.1 제품 정의

**Artier**는 그림을 그리는 누구나 작품을 올리고, 발견되고, 더 넓은 기회로 연결될 수 있는 **웹 기반** 디지털 아트·갤러리 플랫폼이다.

## 1.2 해결하는 문제

국내 디지털 드로잉 인구는 빠르게 성장하고 있으나, 중장년 창작자를 위한 **작품 발표·전시 공간**이 사실상 없다. 기존 플랫폼(노트폴리오, Behance 등)은 전문가 포트폴리오 중심이고, SNS(인스타그램 등)는 알고리즘 의존도가 높아 신규 작가의 발견이 어렵다.

| 문제 | 대상 | 현재 대안의 한계 |
| --- | --- | --- |
| 작품을 올릴 공간이 없다 | 비전공 창작자, 중장년 | 전문가용 플랫폼 진입 장벽 높음 |
| 올려도 발견되지 않는다 | 신규 작가 전반 | SNS 알고리즘 불리, 팔로워 없으면 노출 없음 |
| 그룹전시를 열 수 없다 | 드로잉 클래스 강사·수강생 | 별도 협업 도구 필요, 번거로움 |
| 창작이 기회로 연결되지 않는다 | 실력 있는 비전공 작가 | 포트폴리오·교육·커머스 연결 채널 없음 |

## 1.3 타겟 유저

| 세그먼트 | 정의 |
| --- | --- |
| 전공자 | 미술 전공·직업 화가 경험이 있는 창작자 |
| 비전공 실력자 | 전공자는 아니지만 자신만의 작업 세계를 갖춘 창작자 |
| 일반 창작자 | 취미·일상 속에서 꾸준히 그림을 그리는 사용자 |

Phase 1 초기 큐레이션은 **중장년(50+) 창작자 및 재도전형 창작자** 중심으로 운영하여 플랫폼의 분위기와 정체성을 형성한다.

## 1.4 핵심 가치 제안

| 가치 | 설명 |
| --- | --- |
| 내 작품을 공개 전시 | 오프라인 전시 없이도 온라인 갤러리에 작품을 올리고 공개적으로 선보일 수 있다 |
| 발견과 재도전의 기회 | 작품이 더 많은 사람에게 노출되고, 다시 활동을 이어갈 계기를 제공한다 |
| 감상과 교류 | 작품을 감상하고 의견을 나누며 작가와 사용자 간 연결을 만든다 |
| 확장 가능성 | 전시에 그치지 않고 교육·유통 등 더 넓은 기회로 연결된다 |

# 2. 범위 정의 (Scope)

## 2.1 Phase 1 포함 (In Scope)

- 회원가입 / 로그인 (소셜 + 이메일, 한국·글로벌 분기 UI)
- 작품 업로드 및 개인·그룹 전시 자동 구성
- 둘러보기 피드 (큐레이션·무한 스크롤)
- 작업물 상세 모달 (좋아요, 저장, 공유)
- 작가 프로필 (내 프로필 / 타인 프로필)
- 팔로우 / 언팔로우
- 앱 내 알림 (실시간 푸시는 범위 밖)
- 검색 (작품·작가·전시 관련)
- 이벤트 목록 및 상세
- 신고 및 어드민에서의 처리 흐름
- 법적 페이지 (이용약관, 개인정보처리방침, FAQ, 문의하기)
- 온보딩 플로우
- 포인트 시스템 (AP 중심, 원장 UI 포함)
- 이메일 알림용 템플릿·흐름 (실제 발송은 인프라 연동 시)
- 한국어 / 영어 UI
- 어드민 (배너, Pick, 신고, 공지, 회원·이벤트·큐레이션 등)
- 서비스 소개 (/about)
- 작품 비공개 전환

## 2.3 Phase 1 구현 전제 (아키텍처)

Phase 1 웹 앱은 **브라우저 localStorage·클라이언트 스토어**를 중심으로 동작하는 **프런트엔드 중심 데모·MVP**이다. 다음은 **문서상 목표**이나 현재 클라이언트만으로는 **실연동이 없거나 모의**인 영역이다.

- 실제 **OAuth**(카카오·구글·애플) 토큰 교환, **SMTP/SMS/알림톡** 발송, **중앙 DB·백엔드 검수 큐**는 **연동 시점(Phase 2 이후)** 에 재정의한다.
- 그에 따라 “서버에 검색 이력 저장” 등은 **로컬 키(계정별·게스트별)** 로 대체될 수 있다.

# 3. 성공 지표 (Success Metrics)

## 3.1 Phase 1 목표 (~오픈 후 3개월)

| 지표 | 목표값 | 측정 방법 |
| --- | --- | --- |
| 누적 가입 회원 수 | 800명 이상 | 서버·분석 연동 후 DB 집계 |
| 누적 업로드 작가 수 | 200명 이상 | 업로드 유저 기준 |
| 누적 작품 업로드 수 | 3,000점 이상 | DB 집계 |
| MAU | 500명 | 월간 활성 |
| 30일 재방문율 | 15% 이상 | 코호트 분석 |

## 3.2 핵심 가설

1. **콘텐츠 가설**: 파트너 작가 확보와 사전 업로드가 있으면 둘러보기 첫인상이 가입 전환을 높인다.
2. **바이럴 가설**: 전시 초대·공유 URL이 작가 SNS 홍보 수단이 되어 유입을 만든다.
3. **리텐션 가설**: 주간 테마·Artier's Pick이 재방문을 유도한다.
4. **그룹 유입 가설**: 그룹 전시가 강사·수강생 단위 유입을 만든다.

# 4. 사용자 스토리

## 4.1 일반 창작자

- 그림을 올릴 공간이 없어서 가입하고 첫 작품을 업로드하고 싶다.
- 내 작품이 얼마나 발견되는지 좋아요·팔로우로 확인하고 싶다.
- 지인에게 **“내 전시에 초대합니다”** 맥락으로 공유하고 싶다.

## 4.2 드로잉 클래스 강사

- 수강생 작품을 한곳에 모아 그룹 전시로 수업 결과를 보여주고 싶다.
- 프로필이 수업 홍보 도구가 되고, 수강생 작품을 모아보는 탭이 필요하다.
- 비가입 수강생에게 연락을 보내 가입 후 본인 작품을 확인하게 하고 싶다.

## 4.3 감상자 (비작가)

- 인스타그램처럼 스크롤하며 다양한 작가를 발견하고 싶다.
- 마음에 드는 작가를 팔로우하고 신작 알림을 받고 싶다.
- 좋아요를 누른 작품을 나중에 다시 보고 싶다.

## 4.4 운영팀

- 매주 Artier's Pick을 선정해 배지를 부여하고 싶다.
- 신고된 콘텐츠를 검토해 삭제·경고·기각·비공개 처리하고 싶다.
- 배너와 이벤트를 어드민에서 등록해 운영하고 싶다.

# 5. 기능 요구사항 (구현·제품 확정)

## 5.1 핵심 차별화

| 기능 | 정의 | 우선순위 |
| --- | --- | --- |
| 자동 전시 화면 | 이미지 업로드 후 배경·레이아웃이 감상에 맞게 구성된다. 배경은 원본 이미지 블러·스케일·투명도 등 **순수 프런트 연출**로 처리한다. | P0 |
| 전시 초대·공유 | `/exhibitions/:id` 단일 URL 체계. 쿼리로 **전시 단위 초대**, **비회원 작가 알림**, **작품 단위 공유** 랜딩을 구분한다. | P0 |
| 그룹 전시 | 업로드 시 **함께 올리기**로 그룹 전시. 이미지 **1장당 작가 1명**만 지정. 그룹 전시는 공동 창작이 아니라 **개별 작가 작품의 묶음**이다. **그룹명은 필수**. 강사(대리) 업로드 시 작가 지정·그룹명 규칙이 강화된다. | P0 |
| 발견형 피드 | 랜덤·큐레이션 혼합. **이전에 본 작품**은 뒤로 보내는 등 세션 단위 제어. **좋아요·저장·팔로우** 신호로 버킷 내 가중치를 준다. | P0 |
| Artier's Pick | 피드 상단 고정 레이어. 운영이 선정한 작품에 배지. | P1 |

## 5.2 작품 업로드

| 항목 | 요구사항 |
| --- | --- |
| 전시당 이미지 | **최대 10장** (초기 명세 20장 대비 **제품 확정 10장**) |
| 파일 크기 | **장당 10MB 초과 불가** |
| 형식 | JPEG, PNG, WEBP, GIF |
| 해상도 | 이미지 **짧은 변 최소 800px** 미만이면 업로드 불가 |
| 카메라 촬영 차단 | JPEG의 EXIF에서 **Make/Model**이 카메라로 판단되면 차단. EXIF 없음·분석 불가·PNG 등은 허용 흐름 유지 |
| 전시 제목 | **최대 20자** |
| 장별 작품 제목 | **최대 120자** |
| 그룹명 | 그룹 전시 시 **필수**. 강사 여부와 무관 |
| 작가 지정 | 그룹 전시에서는 **모든 이미지**에 회원 또는 비회원(실명·전화) 지정 |
| 비속어 | 전시명·그룹명 발행 시 차단. 가입 닉네임·프로필(닉·헤드라인·자기소개)·장 제목 편집에도 적용 |
| 그룹인데 비강사·실질 작가 1명 이하 | **개인 전시로 전환**할지 확인 대화 후 전환 가능 |
| 임시저장 | 초안 스토어로 저장 |
| 업로드 진입 | 비로그인 URL 직접 진입 시 로그인으로 보냄 |
| 대표 이미지 | 업로드한 이미지 중에서만 대표(커스텀) 지정. **외부 파일만으로 커버 구성 불가** |
| 발행 시 태그 | 현재 구현은 태그 배열 빈 값 고정(향후 확장 여지) |
| 검수 상태 | `pending` / `approved` / `rejected`. 기본 발행은 **검수 대기**. 개발·데모용으로 환경 변수 **`VITE_UPLOAD_AUTO_APPROVE=true`** 일 때만 즉시 승인 가능 |
| 미리보기 | 미리보기 모드 작품은 검수 상태와 무관하게 **통과로 보이는 경로**가 있음(실제 발행 객체와 분리) |
| 반려 사유 분류 | **4종**: 저품질·스팸·부적절·저작권 의심 (코드 상 상수·i18n 매핑) |

## 5.3 전시 URL·공유

| 경로·조건 | 동작 |
| --- | --- |
| `/exhibitions/:id` | 기본: 둘러보기 피드 맥락 + 작품 모달 딥링크 |
| 큐레이션 목업 ID | 별도 전시 상세 페이지 |
| `?from=invite` | **전시 단위** 초대 랜딩: 전시 헤더·작품 그리드·가입 유도 등 |
| `?from=credited` | 비회원 작가에게 보낸 연락 맥락(동일 랜딩 컴포넌트, 문구만 분기) |
| `?from=work` | **작품 한 점** 강조 랜딩 |
| 공유 URL 생성 | 작품 상세에서 전시 공유 시 `from=invite` 등 규칙 부여 |

한 공유 링크에 **여러 전시를 묶지 않는다**. 같은 전시 맥락의 작품만 묶어 보여준다.

## 5.4 비회원 작가·SMS·가입 매칭

| 항목 | 요구사항 |
| --- | --- |
| 발송 | Phase 1 클라이언트는 **실제 SMS/알림톡 게이트웨이 없이** 로컬 로그에 기록. 한국 번호는 채널을 알림톡 우선으로 **표시**. **5% 확률 실패 시뮬** 가능 |
| 중복 발송 방지 | 동일 **전화번호 + 작품 ID** 조합에서 **성공한 발송**이 있으면 재발송 생략 |
| 로그 상한 | 발송 로그·매칭 로그는 각각 상한(예: 300·200건)으로 잘라 저장 |
| 저장 | 발행 직후 작품 데이터에서 비회원 **전화번호는 제거**(스크럽)하여 저장 |
| 가입 후 매칭 | 초대 로그 중 **성공 건만** 대상. **전화 일치 + 초대 시 표시명과 온보딩 실명 일치** 시 해당 작품의 비회원 슬롯을 회원으로 승격 |
| 불일치 | 전화만 맞고 이름이 다르면 **작품 연결 안 함**, 로그만 남김 |
| 초대 유입 | 전시 랜딩에서 가입 시 **초대 플래그** 설정. 온보딩에선 실명·전화가 **모든 경로에서 필수**이므로 초대 여부와 무관하게 매칭 대상. 초대 플래그는 배너 문구·에러 메시지 분기에만 사용 |

## 5.5 둘러보기 피드

| 항목 | 요구사항 |
| --- | --- |
| 공개 풀 | 검수 `pending`·`rejected`는 공개 피드에서 제외. 작가가 비공개로 둔 작품 제외 |
| 버킷 순서 | **Pick → 테마전 → 추천 작가 → 최근 14일 이내 업로드 → 일반** |
| 중복 | 동일 작품은 **한 버킷에만** |
| 정렬 가중 | 좋아요×2 + 저장×3 + (로그인 시) 팔로우 중인 작가 작품 **+8**, 약한 랜덤성 추가 |
| 댓글 | 랭킹에 **포함하지 않음**(Phase 2 댓글 범위) |
| 카드 | 좋아요·저장 **숫자는 표시하지 않고** 상태 아이콘만 |

## 5.6 Artier's Pick · 이벤트 · 배너 · 큐레이션

| 영역 | 요구사항 |
| --- | --- |
| Pick 어드민 | 선정 목록 **최대 10건**. 동일 작품 중복 추가 불가. 검색용 후보는 **목업 카탈로그**가 섞일 수 있음. **목록 내 드래그로 순서 변경은 UI만 있고 실제 순서 반영은 미연동**(번호 표시 중심) |
| Pick 정책(문서 목표) | 주당 4~8점·동일 작가 20일 중복 제외 등 **운영 규칙**은 있으나, **클라이언트에 20일 자동 제한·주간 자동 초기화 스케줄은 없음**. 화면에 “월요일 자정 초기화” 안내가 있어도 **실제 cron은 없음** |
| 피드 반영 | 실제 피드 상단 Pick은 작품의 `pick` / `editorsPick` **플래그 및 시드 데이터**에 의존. 어드민 목록과 **완전 자동 연동은 아님** |
| 이벤트 | 단일 이벤트 스토어·영속 키로 목록·상세·어드민 CRUD 일관 |
| 배너 | 스토어에 순서 필드. 어드민에서 **드래그로 순서 변경**(dnd-kit 기반) |
| 큐레이션 | 테마전(작품 ID 목록)·추천 작가 집합을 어드민에서 설정 |

## 5.9 포인트 (AP)

활동 포인트(AP)는 **클라이언트 원장**에 적립·표시한다. 상용 서버 이관 시 규칙은 재검증한다.

| 이벤트 | AP | 조건 |
| --- | ---: | --- |
| 회원가입 완료 | +50 | 계정당 1회 |
| 온보딩 1단계 완료 | +30 | 1회 |
| 첫 작품 발행 | +100 | 최초 1회 |
| 이후 발행 | +20 | **같은 UTC 날짜**에 업로드 AP는 **최대 2회** |
| 월간 4번째 발행 | +50 | 해당 달 기준 |
| 그룹 전시 생성(강사) | +30 | 월·강사 조합당 1회 |
| 그룹 참여(비강사) | +15 | 월 **최대 5회** |
| 팔로워 10/50/100 | +30 / +100 / +200 | 마일스톤당 1회 |
| 일일 둘러보기 | +5 | **일 1회** |
| 발행 후 24시간 이내 삭제 | -20 | 빠른 삭제 회수 |
| 원장 길이 | 최대 **500건** 단위로 유지 |

PP(프로모션 포인트)는 데모 잔액 키 등으로 별도. **만료 배치**는 Phase 1에서 **시뮬레이션·로그 수준**일 수 있으며 실차감은 백엔드 연동 후 정의한다.

- 카메라 촬영 이미지 차단(§5.2).
- 비밀번호 규칙(§5.7).

- 설정에서 **글자 크기 작게/보통/크게** 선택.

## 6.4 플랫폼

- 웹 반응형(PC 우선·모바일 대응). 네이티브 앱 없음.
- 한국어·영어 전환.

## 6.5 운영·법무 (목표와 Phase 1 격차)

아래는 **사업·운영 목표**이며, Phase 1 클라이언트만으로는 **전부 자동화되지 않을 수 있다**.

- 신고 처리 **SLA**(일반 7일·불법 24시간)는 **운영 프로세스**로 두고, 시스템은 타임스탬프·상태 저장 수준.
- 야간 마케팅 메일 금지 시간대 등은 **발송 인프라 연동 후** 적용.
- 이용약관·개인정보 처리방침 **법무 확정**, DPO, 이메일 인증, 데이터 내보내기 등은 **런칭 전 과제**로 별도 체크리스트를 둔다.

# 7. 정보 구조 (IA)

## 7.1 사이트맵 (요약)

```
Artier (웹)
├── 둘러보기 (/)
│   ├── 배너 슬라이더
│   ├── 탭: 전체 / 개인 전시 / 그룹 전시
│   └── 작품 카드 피드 (무한 스크롤)
├── 이벤트 (/events, /events/:id)
├── 작품 올리기 (/upload) — 비로그인 시 로그인 유도
├── 검색 (/search)
├── 알림 (/notifications)
├── 포인트 (/points) — Phase 1에서 UI 노출
├── 내 프로필 (/me)
│   ├── 전시 / 작품 관리 / 좋아요 / 저장 / 초안
├── 타인 프로필
├── 전시 (/exhibitions/:id)
│   └── 쿼리: from=invite | credited | work 에 따라 랜딩 분기
├── 계정 설정 (/settings)
├── 서비스 소개 (/about)
├── 공지·FAQ·약관·개인정보·문의
├── 데모·검수용 (/demo, /demo/reference) — 운영상 숨김 여부 결정
└── 어드민 (/admin 및 하위: 배너, 이벤트, 신고, 회원, Pick, 큐레이션 등)
```

## 7.2 데이터 모델 (Phase 1 클라이언트 관점)

- **작품(Work)** 이 핵심 단위. 전시명·그룹명·이미지 배열·검수 상태·Pick 플래그·업로드 시각 등이 한 객체에 담긴다.
- **전시(Exhibition)** 를 별도 서버 엔티티로 두지 않고도, 동일 URL로 개인·그룹 전시를 **작품 필드**로 구분해 렌더링한다.
- v1.3 초안에 있던 **`User.is_instructor` 고정 필드는 쓰지 않는다**. 강사 노출은 **강사 대리 업로드 플래그가 있는 작품 존재 여부**로만 판단한다.

## 7.3 주요 로컬 저장 키 (예시)

운영·디버깅 시 참고. **단일 진실 공급원은 코드 저장소**이나, 문서상 개념만 적는다.

| 초안 | `artier_drafts` |
| 초대 발송 로그 | `artier_invite_messaging_log` |

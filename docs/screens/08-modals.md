# 08. 주요 모달 (화면급 상호작용)

> 이 문서 단독으로 이해 가능하도록 작성. 레퍼런스 명세를 이 문서 기준으로 업데이트해주세요.
> 라우트는 아니지만 기능·복잡도가 화면급인 모달 12개.

---

## 사전 지식

### 공통 정책

- **시니어 친화 터치 영역**: 모달 안 버튼도 44×44 픽셀 유지
- **브라우저 네이티브 금지**: `confirm()`, `alert()` 등 사용 불가. 모든 확인 다이얼로그는 앱 자체 디자인
- **XSS 방지**: 본문 HTML 주입 금지 (필요하면 살균 처리 후 사용)

### 모달 마운트 전략

| 모달 | 마운트 위치 |
|---|---|
| 공통 확인 다이얼로그 | 앱 루트에 한 인스턴스만 — 함수 호출로 오픈 |
| 가입 후 본인 확인 게이트 | 앱 루트 — 트리거 감지로 자동 오픈 |
| 그 외 (소셜 가입·작품 상세·세부정보·로그인 유도·신고) | 호출하는 화면 내부에 개별 마운트 |

## MOD-01 · 소셜 첫 가입 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 로그인 화면에서 소셜 버튼(카카오·구글·애플) 첫 클릭 시 |

### 모달의 목적
소셜 첫 가입자에게 약관 동의 + 닉네임을 빠르게 받음. 재방문자는 이 모달 없이 즉시 로그인.

### 사용자 시나리오
- 주 사용자: 처음 소셜로 가입하는 사용자
- 기대: 빠른 약관 확인 + 닉네임만 정하고 본 서비스 진입

### 모달 구성

| 영역 | 요소 |
|---|---|
| 헤더 | provider 로고 + "○○로 가입" 타이틀 |
| 닉네임 입력 | 2~20자 + 비속어 차단 |
| 약관 동의 | 전체동의 마스터 + 필수 2개(이용약관·개인정보) + 선택 2개(마케팅 이메일·푸시) |
| 연령 확인 | "만 14세 이상입니다" 체크 (필수) |
| [가입 완료] 버튼 | 모든 검증 통과해야 활성화 |
| 닫기 | X 아이콘 또는 ESC |

### 검증
- 닉네임: 길이 + 비속어 + 중복 방지
- 필수 약관 3개 모두 체크 (이용약관 + 개인정보 + 14세 이상)

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 전체동의 체크 | 필수 2 + 선택 2 모두 토글 |
| 이용약관 "(보기)" | 새 탭 또는 같은 탭에서 `/terms` |
| 개인정보처리방침 "(보기)" | `/privacy` |
| [가입 완료] | 검증 통과 시 모달 닫힘 → **온보딩(`/onboarding`)**으로 이동 |
| X 아이콘 / ESC / 배경 탭 | 모달 닫힘 (로그인 화면 상태 그대로) |

### 완료 시 동작
1. 로그인 처리 + 세션 저장
2. 해당 provider 첫 가입 완료 마킹 (다음 방문부터는 이 모달 건너뜀)
3. 닉네임·provider 이메일(있으면)을 온보딩 prefill용으로 보관
4. 온보딩 화면으로 이동

### 수용기준 (Acceptance Criteria)

- AC-01: Given 소셜 첫 가입 모달 / When 닉네임을 2~20자로 입력하고 필수 약관 3개(이용약관·개인정보·14세 이상)를 모두 체크 / Then [가입 완료] 버튼이 활성화된다
- AC-02: Given 닉네임 입력란 / When 비속어를 입력 / Then 에러 메시지가 표시되고 [가입 완료] 비활성 유지
- AC-03: Given 닉네임 입력란 / When 1자 또는 21자 이상 입력 / Then 길이 검증 에러 표시
- AC-04: Given 전체동의 마스터 체크 / When 클릭 / Then 필수 2개 + 선택 2개 모두 토글된다
- AC-05: Given 필수 약관 중 "만 14세 이상" 미체크 / When [가입 완료] 시도 / Then 버튼 비활성 상태 유지
- AC-06: Given 모든 검증 통과 후 [가입 완료] 클릭 / When 처리 완료 / Then 모달 닫히고 온보딩(`/onboarding`)으로 이동하며 해당 provider 첫 가입 마킹된다

### 엣지케이스

- EC-01: 이미 가입된 provider로 다시 소셜 버튼 클릭 → 이 모달 없이 즉시 로그인
- EC-02: 다른 사용자와 동일한 닉네임 입력 → 중복 에러 표시
- EC-03: X/ESC로 모달 닫기 → 로그인 화면 상태 그대로 유지, 가입 처리 없음

### Phase 2 보강
- 실 OAuth provider로부터 받은 이메일·이름·프로필 사진 prefill
- 14세 확인을 provider 생년월일 정보로 자동 검증

## MOD-02 · 가입 후 본인 확인 게이트

| 항목 | 값 |
|---|---|
| 마운트 위치 | 앱 루트 (항상 마운트) |
| 트리거 | 가입 흐름에서 SMS 초대 자동 매칭이 이름 불일치로 차단된 경우 |

### 모달의 목적
SMS 초대 매칭에서 **전화는 일치하지만 이름이 불일치**해 차단된 초대를 사용자에게 보여주고, 본인 확인 받아 수동으로 승격시키거나 신고 처리.

### 이게 뜨는 상황 (예시)

작가 A가 친구 B를 초대하면서 "민수쌤"으로 이름 입력 → SMS 받은 B가 가입 시 본명 "김민수"로 입력 → 전화는 같지만 이름이 달라 자동 매칭 실패 → 그렇다고 그냥 무시하면 작품 연결이 안 됨 → 이 모달로 본인 확인.

### 트리거 흐름

1. 온보딩 완료 시 SMS 매칭 함수가 전화 일치/이름 불일치 항목을 발견
2. 해당 항목을 잠시 보관 (세션 동안)
3. 홈 도착 → 루트에 마운트된 이 게이트가 감지 → 모달 자동 오픈
4. 각 차단된 초대마다 작품·작가 정보를 해석해 표시

### 모달 구성

| 영역 | 요소 |
|---|---|
| 타이틀 | "회원님 번호로 받은 초대가 있어요" |
| 설명 | "작가가 입력한 이름이 회원님 실명과 달라 자동 연결되지 않았어요. 본인이 맞으면 연결, 모르는 초대면 신고해주세요." |
| 각 초대 카드 | 작품 제목 / 보낸 작가 이름 / 초대 시 이름 (예: "[민수쌤]") + 두 버튼 |
| **[네, 제 작품이에요]** | 수락 |
| **[아니요, 모르는 초대]** | 거부 (신고) |
| 하단 | **[나중에]** — 보관된 차단 정보 비우고 모달 닫음. 같은 세션 동안 다시 안 뜸 |

### 탭·액션 결과 (네비게이션)

이 모달은 사용자가 **홈(둘러보기)에 도착한 직후** 자동으로 열림 — 홈 화면 위에 겹쳐 표시. 모달을 닫으면 홈(둘러보기) 화면이 그대로 노출됨. 별도 페이지 이동 없음.

| 버튼 | 동작 |
|---|---|
| **[네, 제 작품이에요]** | 해당 초대 카드만 모달에서 제거 + 승격 토스트. 남은 초대가 있으면 모달 유지, 없으면 자동 닫힘 |
| **[아니요, 모르는 초대]** | 해당 카드만 제거 + 경고 등록 토스트. 동일한 자동 닫힘 규칙 |
| **[나중에]** | 전체 보관 정보 비우고 모달 닫음 (다음 세션엔 다시 뜨지 않음) |
| X 아이콘 / ESC | [나중에]와 동일 동작 |

### 수용기준 (Acceptance Criteria)

- AC-01: Given 가입 후 SMS 초대 자동 매칭에서 이름 불일치로 차단된 초대가 존재 / When 홈 도착 / Then 본인 확인 게이트 모달이 자동으로 열린다
- AC-02: Given 초대 카드에서 [네, 제 작품이에요] 클릭 / When 수락 / Then 해당 초대가 수동 승격(매칭) 처리되고 토스트가 표시되며 해당 카드가 모달에서 제거된다
- AC-03: Given 초대 카드에서 [아니요, 모르는 초대] 클릭 / When 거부 / Then 발신 작가에게 경고 +1이 등록되고 해당 카드가 모달에서 제거된다
- AC-04: Given 모든 초대 카드 처리 완료 / When 마지막 카드 수락 또는 거부 / Then 모달이 자동으로 닫힌다
- AC-05: Given [나중에] 버튼 탭 / When 클릭 / Then 보관된 차단 정보가 비워지고 모달이 닫히며 같은 세션에서 다시 뜨지 않는다
- AC-06: Given X 아이콘 또는 ESC / When 닫기 / Then [나중에]와 동일하게 동작한다

### 엣지케이스

- EC-01: 차단된 초대가 여러 건일 때 → 각 카드를 개별적으로 수락/거부 가능, 남은 카드가 있으면 모달 유지
- EC-02: 차단된 초대가 0건일 때 → 모달이 열리지 않음
- EC-03: 세션 재시작 후 → 세션 동안만 보관이므로 다시 뜨지 않음 (일회성 트리거)

### Phase 2 보강
- **초대 토큰 기반 매칭**으로 전환하면 이 게이트 폐기 가능 (이름 불일치 케이스 자체가 사라짐)
- 프로필에 "미연결 초대" 섹션 — 사용자가 언제든 다시 확인

## MOD-03 · 작품 상세 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 둘러보기 · 프로필 · 전시 상세의 작품 카드 탭 |

### 모달의 목적
작품을 풀스크린 유사 모달로 크게 감상. 시니어 작가의 작품 디테일 보존.

### 모달 구성

| 영역 | 요소 |
|---|---|
| 배경 | **원본 이미지를 흐림 + 확대 + 투명도 조정**으로 작품 톤이 자연스럽게 묻어남 (별도 색상 추출 알고리즘 없이 단순 시각 효과) |
| 중앙 이미지 | 원본 비율 유지 (최대 화면 높이 90%) |
| 헤더 좌측 | 작가 아바타+이름 (개인전시) 또는 그룹 아이콘+그룹명 (그룹전시). 그룹 전시는 게시자 라인 추가 표시 |
| 헤더 우측 | Artier's Pick 배지 (해당 시) + 팔로우 버튼 |
| 우측/하단 정보 | 제목 · 작가 아바타+이름 · 업로드일 · 좋아요/저장 버튼 |
| 액션 | 좋아요 · 저장 · 공유 · 신고 |
| 좌우 네비 (선택) | 같은 전시의 이전/다음 작품 |
| 닫기 | ESC · 배경 탭 · X 아이콘 |

### 공유 URL 규칙
공유 버튼 탭 시 **초대 진입 파라미터가 붙은 URL**을 기기 공유 시트 또는 클립보드로 제공:
- 받은 사람이 탭하면 → 전시 초대 랜딩으로 진입 → 가입 유도
- 이미 회원이면 전시 상세로 자동 전환되며 그 작품에 포커스

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 좋아요 | 모달 유지 — 상태 토글만 |
| 저장 | 모달 유지 — 토글 |
| 공유 | 모달 유지 — 기기 공유 시트 or 클립보드 복사 |
| 신고 | 모달 위에 **신고 모달** 겹쳐 열림 (MOD-07) |
| 작가 아바타/이름 | 모달 닫힘 → **작가 프로필(`/profile/:artistId`)** 로 이동 |
| 좌우 화살표 | 같은 모달에서 이전/다음 작품으로 전환 (전시 범위 안에서) |
| X 아이콘 / ESC / 배경 탭 | 모달 닫힘 → 이전 화면(둘러보기·프로필·전시 상세 등) 유지 |

### 수용기준 (Acceptance Criteria)

- AC-01: Given 작품 카드 탭 / When 작품 상세 모달 열림 / Then 원본 비율 유지된 이미지가 중앙에 표시되고 배경에 원본 이미지 흐림 효과가 적용된다
- AC-02: Given 작품 상세 모달 / When 좋아요 버튼 탭 / Then 모달 유지 상태에서 좋아요 상태만 토글된다
- AC-03: Given 작품 상세 모달 / When 저장 버튼 탭 / Then 모달 유지 상태에서 저장 상태만 토글된다
- AC-04: Given 작품 상세 모달 / When 공유 버튼 탭 / Then 초대 진입 파라미터(`?from=invite`)가 포함된 URL이 기기 공유 시트 또는 클립보드로 복사된다
- AC-05: Given 작품 상세 모달 / When 신고 버튼 탭 / Then 모달 위에 신고 모달(MOD-07)이 겹쳐 열린다
- AC-06: Given 같은 전시의 작품이 여러 개 / When 좌우 화살표 탭 / Then 같은 모달에서 이전/다음 작품으로 전환된다
- AC-07: Given 그룹 전시 작품 / When 모달 헤더 확인 / Then 그룹 아이콘+그룹명 + 게시자 라인이 표시된다
- AC-08: Given Artier's Pick으로 선정된 작품 / When 모달 열림 / Then 헤더 우상단에 Pick 배지가 표시된다
- AC-09: Given 작가 아바타/이름 탭 / When 클릭 / Then 모달이 닫히고 작가 프로필(`/profile/:artistId`)로 이동한다

### 엣지케이스

- EC-01: 전시에 작품 1개만 있을 때 → 좌우 네비 화살표 미표시 또는 비활성
- EC-02: 작품 이미지가 극단적 종횡비(매우 길거나 매우 넓음)일 때 → 최대 화면 높이 90% 제한으로 비율 유지 축소
- EC-03: 비로그인 사용자가 좋아요/저장 탭 시 → 로그인 유도 모달(MOD-06) 발동

### Phase 2 보강
- 핀치 줌 (모바일)
- 고해상도 원본 다운로드 옵션 (권한 기반 — 작가가 허용한 경우만)
- 작품 영역 주석(annotation)

## MOD-04 · 업로드 세부정보 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 업로드 화면에서 [발행하기] 버튼 탭 시 |

### 모달의 목적
발행 직전 **작품별 제목 설정 + 대표(커버) 이미지 선정**. 두 카드로 나뉨.

### 카드 1: 작품 제목

| 요소 | 비고 |
|---|---|
| 작품 카드별 제목 입력 | 최대 20자 |
| 자동생성 안내 | "비워두면 자동으로 제목을 만들어요" |
| 개별 카드 미리보기 | 썸네일 + 입력 필드 |

**자동 제목 생성 규칙**:
- 그룹 + 작가 지정 있음 → "[작가명] 1번 그림"
- 그룹 + 작가 자유 → "1번 그림"
- 개인 + 1장 → 전시명 사용
- 개인 + 여러 장 → "(무제)"

### 카드 2: 대표 이미지

| 요소 | 비고 |
|---|---|
| 업로드한 이미지 중 선택 (라디오) | 그리드 |
| **외부 파일 커스텀 커버 업로드** | 전시 작품 수에 포함되지 않는 별도 커버. 카드 표지로만 사용 |

**커스텀 커버 패턴**:
- 작품 이미지 배열은 전시 작품들만
- 커스텀 커버는 별도 보관 — 카드 표지 우선순위에서 사용
- 썸네일 조회 순서: 커스텀 커버 있으면 그것, 없으면 첫 이미지

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 작품별 제목 입력 | 모달 유지 — 입력 |
| 커버 이미지 라디오 선택 | 모달 유지 — 선택 표시 |
| 외부 파일 커버 업로드 | 기기 파일 선택 창 → 모달 안에서 미리보기 |
| **[완료]** | 모달 닫힘 → 업로드 화면의 발행 흐름 계속 (비회원 초대 확인 → 작품 저장 → SMS 모의 발송 → 프로필 전시 탭으로 이동) |
| X 아이콘 / ESC / 배경 탭 | 모달 닫힘 (발행 흐름 중단, 업로드 화면 유지) |

### 완료 시
[완료] 후 → 업로드 화면의 발행 흐름 계속 (02 문서의 **SCR-CORE-03** 참조: 비회원 초대 확인 → 작품 저장 → SMS 모의 발송 → 내 프로필 전시 탭으로 이동하며 검수 대기 배너)

### 수용기준 (Acceptance Criteria)

- AC-01: Given 업로드 세부정보 모달 카드 1 / When 작품별 제목 입력란에 20자 이내 제목 입력 / Then 입력이 반영된다
- AC-02: Given 작품별 제목을 비워둔 경우 / When [완료] 클릭 / Then 자동 제목 생성 규칙에 따라 제목이 생성된다
- AC-03: Given 업로드 세부정보 모달 카드 2 / When 업로드한 이미지 중 하나를 라디오로 선택 / Then 해당 이미지가 대표(커버) 이미지로 설정된다
- AC-04: Given 카드 2에서 / When 외부 파일 커스텀 커버 업로드 / Then 별도 커버 이미지가 등록되고 미리보기가 표시되며, 전시 작품 수에는 포함되지 않는다
- AC-05: Given [완료] 클릭 / When 처리 완료 / Then 모달 닫히고 발행 흐름이 계속된다 (비회원 초대 확인 → 저장 → SMS → 프로필 전시 탭)
- AC-06: Given X/ESC/배경 탭 / When 모달 닫기 / Then 발행 흐름이 중단되고 업로드 화면이 유지된다

### 엣지케이스

- EC-01: 작품 제목에 21자 이상 입력 시도 → 20자 제한에 의해 추가 입력 차단
- EC-02: 이미지 1장만 업로드한 경우 → 커버 선택이 해당 이미지로 자동 설정, 라디오 1개만 표시
- EC-03: 커스텀 커버 업로드 후 삭제 → 첫 이미지가 기본 커버로 복귀

## MOD-05 · 비회원 초대 발송 확인 (공통 확인 다이얼로그 활용)

| 항목 | 값 |
|---|---|
| 구현 | 공통 확인 다이얼로그(MOD-08)를 업로드 발행 흐름에서 호출 |
| 트리거 | 그룹 전시 발행 흐름에서 비회원 작가가 1명 이상 포함되어 있을 때 |

### 모달의 목적
SMS 초대가 발송되기 **직전에 한 번 더** 실명 정확성을 상기시키고 수신자 목록을 확인. 잘못된 이름(별명·호칭)으로 발송되면 가입 후 매칭이 실패하므로.

### 모달 본문

```
[타이틀]
초대장을 발송할까요?

[설명]
다음 분께 SMS/알림톡으로 초대장이 발송됩니다.
• 김민수
• 이영희

⚠️ 입력하신 이름이 그분의 실명(가입 시 입력할 이름)과 정확히 같아야 
작품이 자동 연결됩니다. 호칭·별명(예: "민수쌤", "엄마")으로는 
연결되지 않아요.

[취소] [발송]
```

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| **[취소]** | 모달 닫힘 + **발행 자체 중단**. 작품 저장 안 됨, SMS 발송 안 됨. 업로드 화면 유지 |
| **[발송]** | 모달 닫힘 → 작품 저장 → SMS 모의 발송 → 발송 후 작품에서 전화번호 제거 → 내 프로필 전시 탭(`/me?tab=exhibition`)으로 이동 |
| X / ESC / 배경 탭 | [취소]와 동일 |

### 동작
- **취소** → 발행 자체 중단. 작품 저장 안 됨, SMS 발송 안 됨
- **발송** → 나머지 발행 흐름 진행: 작품 저장 → SMS 모의 발송 → 발송 후 작품에서 전화번호 제거

### 참고: 본문 줄바꿈·목록
설명 텍스트가 여러 줄 + 불릿을 포함하므로, 공통 확인 다이얼로그의 설명 영역이 **줄바꿈을 그대로 렌더**해야 함. (기본 설정이 한 줄로 합쳐지는 경우 설정 필요.)

### 수용기준 (Acceptance Criteria)

- AC-01: Given 그룹 전시 발행 중 비회원 작가가 1명 이상 포함 / When 발행 흐름 진행 / Then 비회원 초대 발송 확인 다이얼로그가 표시되고 수신자 이름 목록이 본문에 나열된다
- AC-02: Given 다이얼로그에서 [취소] 클릭 / When 취소 / Then 발행 자체가 중단되고 작품 저장·SMS 발송 모두 일어나지 않으며 업로드 화면이 유지된다
- AC-03: Given 다이얼로그에서 [발송] 클릭 / When 발송 / Then 작품 저장 → SMS 모의 발송 → 전화번호 제거 → 내 프로필 전시 탭으로 이동
- AC-04: Given X/ESC/배경 탭으로 닫기 / When 닫기 / Then [취소]와 동일하게 발행 중단

### 엣지케이스

- EC-01: 비회원 작가가 0명인 그룹 전시 → 이 확인 다이얼로그 자체가 뜨지 않고 바로 발행 진행
- EC-02: 비회원 작가 이름에 호칭·별명이 포함된 경우 → 다이얼로그 본문 경고 문구로 실명 확인 상기

## MOD-06 · 로그인 유도 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 둘러보기·프로필·전시 상세·이벤트 상세 등에서 비로그인 사용자가 좋아요·저장·팔로우·업로드 시도 시 |

### 모달의 목적
비로그인 사용자가 인증이 필요한 액션을 시도할 때 로그인 유도. **세션당 1회 모달**, 이후는 토스트만.

### 모달 구성

| 영역 | 요소 |
|---|---|
| 타이틀 | "로그인이 필요해요" |
| 설명 | 맥락별 다른 카피 (좋아요: "이 작품에 좋아요 표시하려면..." / 업로드: "작품을 올리려면..." 등) |
| 버튼 | [로그인] (redirect 파라미터로 현재 URL 전달) · [회원가입] · [닫기] |

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| **[로그인]** | 모달 닫힘 → **로그인(`/login?redirect=현재URL`)** 으로 이동. 로그인 성공 시 원 화면으로 복귀 |
| **[회원가입]** | 모달 닫힘 → **가입 화면(`/signup`)** 으로 이동 |
| **[닫기]** / ESC / 배경 탭 | 모달 닫힘 + 세션 1회 플래그 세팅 — 이후 같은 세션 시도는 토스트로만 |

### 세션 1회 정책 (시니어 UX)

1. 첫 시도 → 모달 표시
2. [닫기] 또는 ESC → 세션 플래그 세팅
3. 같은 세션에서 다음 시도 시 → 모달 안 뜨고 **토스트만**: "로그인 후 사용 가능합니다"
4. 새 세션(브라우저 재시작) → 다시 모달

시니어가 같은 모달을 반복해 보지 않게 함.

### 예외: 둘러보기 맥락 (감상 액션 후 자동 온보딩 차단)

**감상성 액션**(좋아요·저장·팔로우)으로 로그인하면 → 로그인 후 원 화면 유지 (온보딩으로 튕기지 않음)
**창작/관리 액션**(업로드·신고)으로 로그인하면 → 기존대로 온보딩 유도 (온보딩 미완료자인 경우)

### 수용기준 (Acceptance Criteria)

- AC-01: Given 비로그인 사용자 / When 좋아요·저장·팔로우·업로드 등 인증 필요 액션 최초 시도 / Then 로그인 유도 모달이 표시된다
- AC-02: Given 모달에서 [로그인] 클릭 / When 이동 / Then 현재 URL이 redirect 파라미터로 전달되어 로그인 화면으로 이동하고, 로그인 성공 시 원 화면으로 복귀한다
- AC-03: Given 모달에서 [닫기]/ESC/배경 탭 / When 닫기 / Then 세션 1회 플래그가 세팅되고 이후 같은 세션에서는 토스트만 표시된다
- AC-04: Given 세션 1회 플래그 세팅 후 / When 다시 인증 필요 액션 시도 / Then 모달 대신 "로그인 후 사용 가능합니다" 토스트만 표시된다
- AC-05: Given 감상성 액션(좋아요·저장·팔로우)으로 로그인 완료 / When 복귀 / Then 온보딩으로 튕기지 않고 원 화면 유지

### 엣지케이스

- EC-01: 새 브라우저 세션 시작 → 세션 1회 플래그 초기화되어 다시 모달 표시
- EC-02: [회원가입] 버튼 탭 → 가입 화면(`/signup`)으로 이동
- EC-03: 창작/관리 액션(업로드·신고)으로 로그인한 온보딩 미완료자 → 온보딩 유도

## MOD-07 · 신고 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 작품 카드 "…" 메뉴 → "신고" · 프로필 "…" → "신고" · 작품 상세 모달 "신고" |

### 모달의 목적
작품·작가에 대한 신고 접수. 어드민 신고 관리에서 4가지 액션으로 처리.

### 모달 구성

| 영역 | 요소 |
|---|---|
| 헤더 | 깃발 아이콘(빨강) + "신고하기" 타이틀 |
| 대상 표시 | 작품 신고면 썸네일+제목, 작가 신고면 아바타+이름 |
| 사유 (드롭다운) | 저작권 / 부적절 / 스팸 / 기타 |
| 상세 입력 (선택) | 추가 설명 텍스트 영역 |
| [신고하기] 빨간 버튼 | |
| [취소] | |

### 동작

1. [신고하기] 탭
2. **중복 검사**: (신고자 + 대상 + 사유) 조합으로 이미 신고된 경우 → 토스트 "이미 신고하셨습니다"
3. 신고 접수
4. **본인 피드 숨김**: 같은 사용자에겐 그 작품·작가가 둘러보기에서 안 보이게 됨
5. 운영팀 알림 (Phase 1은 모의)
6. 모달 닫기 + 토스트 "신고가 접수되었습니다"

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 사유 드롭다운 | 모달 유지 — 선택 |
| 상세 입력 | 모달 유지 — 입력 |
| **[신고하기]** | 중복이면 토스트 "이미 신고하셨습니다" (모달 유지). 통과 시 모달 닫힘 + 토스트 "신고가 접수되었습니다" + 이후 본인 피드에서 해당 작품·작가 자동 숨김 |
| **[취소]** / X / ESC | 모달 닫힘 (아무 변화 없음) |

### 어드민 측 처리 흐름 (참고)

어드민 신고 관리 화면에서 이 신고에 4가지 액션 가능:
- **삭제**: 작품 영구 삭제 (작품 신고 한정)
- **경고**: 작가에게 경고 +1 (3회 누적 → 자동 7일 정지)
- **기각**: 신고자에게 허위 신고 +1 (3회 누적 → 7일 차단)
- **비공개**: 작품을 피드·검색에서 숨김

### 수용기준 (Acceptance Criteria)

- AC-01: Given 신고 모달 / When 사유 드롭다운에서 사유 선택 + [신고하기] 클릭 / Then 신고가 접수되고 토스트 "신고가 접수되었습니다"가 표시되며 모달이 닫힌다
- AC-02: Given 같은 사용자가 동일 대상에 같은 사유로 이미 신고한 경우 / When [신고하기] 클릭 / Then 토스트 "이미 신고하셨습니다"가 표시되고 모달은 유지된다
- AC-03: Given 신고 접수 완료 / When 이후 둘러보기 피드 조회 / Then 신고한 작품·작가가 본인 피드에서 자동 숨김 처리된다
- AC-04: Given 사유 미선택 상태 / When [신고하기] 확인 / Then 버튼 비활성 또는 사유 선택 안내

### 엣지케이스

- EC-01: 상세 입력 없이 사유만 선택하고 신고 → 정상 접수 (상세는 선택 사항)
- EC-02: 작품 상세 모달(MOD-03) 위에서 신고 모달 열린 경우 → 신고 모달이 최상위에 겹쳐 표시
- EC-03: [취소]/X/ESC → 모달 닫히고 아무 신고도 접수되지 않음

### Phase 2 보강
- 신고 처리 결과 **신고자에게도 알림** (현재는 "안 보내기" 정책 — 운영 부담 고려)
- 신고 카테고리 세분화 (저작권 → DMCA 양식)
- 첨부 증거 (스크린샷)

## MOD-08 · 공통 확인 다이얼로그

| 항목 | 값 |
|---|---|
| 마운트 | 앱 루트에 한 인스턴스만 — 함수 호출로 오픈 |

### 모달의 목적
브라우저 네이티브 confirm/alert를 대체하는 **프로미스 기반** 확인 다이얼로그. 앱 전역 정책으로 브라우저 네이티브 사용 금지 — 시각 일관성과 다국어 지원을 위함.

### API 개념

```
const ok = await 확인다이얼로그열기({
  title: '삭제하시겠어요?',
  description: '복구할 수 없습니다.',  // 여러 줄 지원
  confirmLabel: '삭제',                  // 기본 "확인"
  cancelLabel: '취소',                   // 기본 "취소"
  destructive: true,                      // true면 확인 버튼 빨간색
});
if (!ok) return;
// 확인됐을 때 진행할 로직
```

### 사용 예 (전체 앱)

| 사용처 | 옵션 |
|---|---|
| 작품 삭제 (프로필·어드민) | 파괴적 |
| 이벤트·배너 삭제 (어드민) | 파괴적 |
| 신고 삭제 (어드민) | 파괴적 |
| 업로드 취소 (편집 모드) | 일반 |
| **비회원 초대 발송** (업로드 발행) | 일반 (MOD-05) |
| 모두 읽음 처리 (알림) | 일반 |
| 회원 탈퇴 최종 확인 (설정) | 파괴적 |

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| **[확인]** | 모달 닫힘 → 호출한 곳의 후속 로직 실행 (예: 삭제·탈퇴·발송). 화면 이동은 호출한 화면의 정책에 따름 |
| **[취소]** / X / ESC / 배경 탭 | 모달 닫힘 → 호출한 곳에서 아무 동작 없음. 원 화면 유지 |

### 동작 원리

- 모듈 레벨에 **단일 프로미스 참조**. 동시 여러 호출 시 이전 것은 자동 취소(`false` resolve) 후 새 모달 표시
- 닫는 방법: ESC · 배경 탭 · [취소] 버튼 → `false` resolve
- [확인] 버튼 → `true` resolve
- 모달 닫힌 후 참조는 정리

### 설명 영역 멀티라인 지원

설명 텍스트에 줄바꿈(\n) + 불릿이 포함될 수 있음 — 시각적으로 그대로 렌더되어야 함. 예: 비회원 초대 발송 확인 모달(MOD-05).

### 수용기준 (Acceptance Criteria)

- AC-01: Given `openConfirm()` 호출 / When title·description 전달 / Then 앱 자체 디자인의 확인 다이얼로그가 표시된다
- AC-02: Given `destructive: true` 옵션 / When 다이얼로그 열림 / Then 확인 버튼이 빨간색으로 표시된다
- AC-03: Given [확인] 버튼 클릭 / When 클릭 / Then Promise가 `true`로 resolve되고 호출한 곳의 후속 로직이 실행된다
- AC-04: Given [취소]/X/ESC/배경 탭 / When 닫기 / Then Promise가 `false`로 resolve되고 아무 동작 없음
- AC-05: Given description에 줄바꿈(\n) + 불릿 포함 / When 다이얼로그 표시 / Then 멀티라인이 시각적으로 그대로 렌더된다
- AC-06: Given 동시에 여러 `openConfirm()` 호출 / When 이전 것이 아직 열려 있을 때 새 호출 / Then 이전 것은 자동 취소(`false` resolve)되고 새 다이얼로그가 표시된다

### 엣지케이스

- EC-01: description 미전달 시 → title만으로 다이얼로그 표시
- EC-02: confirmLabel/cancelLabel 커스텀 → 기본 "확인"/"취소" 대신 전달된 라벨 사용

### Phase 2 보강
- 추가 옵션 (예: 입력 필드 — "비밀번호 입력 후 확인" 같은 패턴) — 현재는 별도 다이얼로그 사용
- 키보드 단축키 (Enter = 확인, ESC = 취소) 일관성

## MOD-09 · 반려 사유 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 본인 프로필 작품 탭에서 **반려 배지** 탭 시 |

### 모달의 목적
운영팀이 반려한 작품에 대해 작가에게 **사유·메시지를 정중하게 전달** + **수정해서 다시 올릴 수 있는 즉시 동선** 제공. 시니어 작가가 거절감을 받지 않도록 톤·색조 신중하게.

### 사용자 시나리오
- 주 사용자: 본인 작품이 반려된 작가
- 기대: 무엇이 문제였는지 명확히, 어떻게 고치면 되는지 안내, 바로 수정 가능

### 모달 구성

| 영역 | 요소 |
|---|---|
| 헤더 | "검수 결과 안내" (중립 톤. "반려"라는 단어 강조 X) |
| 작품 미리보기 | 썸네일 + 전시명 |
| **사유 라벨** | 4분류 중 하나 (저작권 의심 / 부적절한 콘텐츠 / 품질 미달 / 기타) — 색은 amber/주황 (빨강 X) |
| **운영팀 메시지** | 운영팀이 작성한 구체적 안내 (예: "단변 800px 미만으로 보입니다. 더 큰 사이즈로 다시 올려주세요") |
| 안내 카피 | "수정 후 다시 올리시면 검수해드려요" |
| 버튼 | [수정해서 다시 올리기] (primary) · [닫기] |

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| **[수정해서 다시 올리기]** | 모달 닫힘 → **업로드 화면 편집 모드(`/upload?editWorkId=...`)** 로 이동. 기존 이미지·전시명·그룹명 등 모두 prefill |
| [닫기] / X / ESC / 배경 탭 | 모달 닫힘. 프로필 작품 탭에 반려 배지는 그대로 표시 (작가가 또 누를 수 있게) |

### 정책
- 같은 작품을 여러 번 다시 올려도 반려 배지·이 모달은 검수 상태에 따라 다시 표시
- 작가가 작품을 삭제하면 배지·모달도 함께 사라짐

### 수용기준 (Acceptance Criteria)

- AC-01: Given 본인 프로필 작품 탭에서 반려 배지가 있는 작품 / When 배지 탭 / Then 반려 사유 모달이 열리고 사유 라벨(4분류 중 1개) + 운영팀 메시지가 표시된다
- AC-02: Given 모달에서 [수정해서 다시 올리기] 클릭 / When 클릭 / Then 모달 닫히고 업로드 화면 편집 모드(`/upload?editWorkId=...`)로 이동하며 기존 이미지·전시명 등이 prefill된다
- AC-03: Given [닫기]/X/ESC/배경 탭 / When 닫기 / Then 모달 닫히고 프로필 작품 탭에 반려 배지는 그대로 유지된다
- AC-04: Given 사유 라벨 표시 / When 모달 렌더 / Then 색상이 amber/주황으로 표시된다 (빨강 아님)

### 엣지케이스

- EC-01: 같은 작품을 수정 후 다시 올려 다시 반려된 경우 → 새로운 사유·메시지로 모달 갱신
- EC-02: 반려된 작품을 삭제한 경우 → 배지·모달 모두 사라짐
- EC-03: 운영팀 메시지가 비어 있는 경우 → 사유 라벨만 표시하고 메시지 영역은 생략

### Phase 2 보강
- 운영팀이 첨부 이미지로 안내 (예: "이 부분이 잘렸어요" 마킹)
- 이의제기 채널 (모달 안에서 1:1 문의로 연결)

## MOD-10 · 프로필 편집 다이얼로그

| 항목 | 값 |
|---|---|
| 호출 위치 | 본인 프로필에서 "편집" 버튼 · URL `?edit=1`로 직진입 |

### 모달의 목적
프로필 메타 정보(닉네임·실명·헤드라인·자기소개·위치·외부 링크)를 한 화면에서 편집. 별도 편집 페이지를 두지 않고 다이얼로그로 처리 — 이전 상태 비교가 쉽고 취소 시 되돌리기 자연스러움.

### 모달 구성 (단일 폼, 위→아래)

| 필드 | 규칙 |
|---|---|
| 닉네임 | 2~20자 + 비속어 검사 + 중복 차단 |
| 실명 | 2자 이상 (비공개 — 본인만 보임) |
| 헤드라인 (Prominent) | 최대 20자 — 프로필 헤더에 큰 글씨로 노출되는 한 줄 |
| 자기소개 | 글자 제한 없음 + 비속어 검사 |
| 위치 | 국가 드롭다운 (대한민국·미국·일본·중국·영국·프랑스·독일·캐나다·호주·대만·기타) |
| **외부 링크 영역** | 6개 플랫폼 고정 행 (인스타·유튜브·페이스북·트위터·핀터레스트·비메오) — 각 행에 핸들/ID만 입력 (URL 자동 조립) |
| 버튼 | [저장] (primary) · [취소] |

### 외부 링크 입력 패턴

각 플랫폼 행의 입력란 placeholder가 형식 안내:
- 인스타그램: "@username" 또는 "username"
- 유튜브: "@channel" 또는 "채널 ID"
- 페이스북: "username"
- 트위터: "@handle"
- 핀터레스트: "username"
- 비메오: "username"

빈 칸은 프로필에 노출 안 됨. 입력된 행만 아이콘 + 링크로 표시.

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 입력란 변경 | 같은 다이얼로그에서 검증 (실시간) — 에러는 해당 입력란 아래 빨간 텍스트 |
| [저장] | 검증 통과 시 다이얼로그 닫힘 + 프로필 즉시 갱신 + 토스트 "프로필이 저장되었어요" |
| [취소] / X / ESC / 배경 탭 | 변경 사항이 있으면 **파괴적 확인 다이얼로그** ("저장하지 않은 변경사항이 있어요. 정말 닫을까요?") → 확인 시 다이얼로그 닫힘. 변경 없으면 즉시 닫힘 |

### 검증·정책
- 비속어: 닉네임·자기소개에 적용
- 중복 닉네임: 다른 사용자 닉네임과 같으면 차단
- 외부 링크 입력값: URL 형식이 아닌 핸들/ID 권장 — 사용자가 풀 URL 붙여넣어도 시스템이 핸들로 정규화

### 수용기준 (Acceptance Criteria)

- AC-01: Given 프로필 편집 다이얼로그 / When 닉네임을 2~20자로 입력하고 비속어·중복 검사 통과 / Then 에러 없이 입력 반영
- AC-02: Given 닉네임에 비속어 입력 / When 실시간 검증 / Then 해당 입력란 아래 빨간 에러 텍스트 표시
- AC-03: Given 다른 사용자와 동일 닉네임 입력 / When 실시간 검증 / Then 중복 에러 표시
- AC-04: Given 헤드라인 입력 / When 20자 초과 입력 시도 / Then 20자 제한에 의해 추가 입력 차단
- AC-05: Given 외부 링크 입력란 / When 핸들/ID 또는 풀 URL 입력 / Then 시스템이 핸들로 정규화하여 저장
- AC-06: Given 모든 검증 통과 후 [저장] 클릭 / When 저장 / Then 다이얼로그 닫히고 프로필 즉시 갱신 + 토스트 "프로필이 저장되었어요"
- AC-07: Given 변경 사항이 있는 상태에서 [취소]/X/ESC/배경 탭 / When 닫기 시도 / Then 파괴적 확인 다이얼로그 ("저장하지 않은 변경사항이 있어요. 정말 닫을까요?") 표시
- AC-08: Given 변경 사항이 없는 상태에서 [취소]/X/ESC / When 닫기 / Then 즉시 닫힘

### 엣지케이스

- EC-01: 외부 링크 6개 행 모두 비워두기 → 프로필에 링크 영역 미노출
- EC-02: 자기소개에 비속어 포함 → 저장 차단 + 에러 표시
- EC-03: 닉네임 1자 입력 → 최소 2자 검증 에러

### Phase 2 보강
- 프로필 헤드 배경 이미지 편집
- 자기소개 리치 포맷 (간단한 줄바꿈·강조)
- 작가 활동 영역 태그 (수채화·유화·드로잉 등)

## MOD-11 · 프로필 이미지 모달

| 항목 | 값 |
|---|---|
| 호출 위치 | 프로필 헤더의 아바타 탭 시 (본인·타인 모두) |

### 모달의 목적
프로필 이미지를 큰 사이즈로 감상 + 본인일 경우 변경. 시니어가 작은 아바타를 잘 알아보지 못할 수 있어 별도 모달.

### 모달 구성

| 영역 | 요소 |
|---|---|
| 헤더 | 작가 이름 + X 닫기 |
| 큰 이미지 | 원본 비율 유지, 화면 80% 차지 |
| (본인만) [이미지 변경] 버튼 | 파일 선택 → 미리보기로 즉시 교체 |
| (본인만) [기본 이미지로] | 업로드한 이미지 제거 → 이니셜 아바타로 |
| (본인만) [저장] / [취소] | 변경 시에만 표시 |

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 큰 이미지 (타인 진입 시) | 모달 유지 (감상만) |
| [이미지 변경] | 기기 파일 선택 창 → 같은 모달에서 미리보기 |
| [기본 이미지로] | 미리보기를 이니셜 아바타로 변경 (저장 안 누르면 적용 X) |
| [저장] | 다이얼로그 닫힘 + 프로필 즉시 갱신 + 토스트 "프로필 사진이 변경되었어요" |
| [취소] / X / ESC / 배경 탭 | 변경 있으면 파괴적 확인 → 확인 시 닫힘. 없으면 즉시 닫힘 |

### 검증·정책
- 파일 크기 제한 (예: 5MB)
- 정사각형 권장 안내 (자동 크롭 X — 사용자가 직접)
- 이미지 타입: JPEG / PNG / WebP

### 수용기준 (Acceptance Criteria)

- AC-01: Given 본인 프로필 헤더의 아바타 탭 / When 클릭 / Then 프로필 이미지 모달이 열리고 큰 이미지 + [이미지 변경]·[기본 이미지로] 버튼이 표시된다
- AC-02: Given 타인 프로필 아바타 탭 / When 클릭 / Then 큰 이미지만 표시되고 변경 버튼은 미노출
- AC-03: Given [이미지 변경] 클릭 / When 파일 선택 후 / Then 같은 모달에서 미리보기가 즉시 교체된다
- AC-04: Given [기본 이미지로] 클릭 / When 클릭 / Then 미리보기가 이니셜 아바타로 변경된다 (저장 전까지 실 적용 안 됨)
- AC-05: Given [저장] 클릭 / When 저장 / Then 다이얼로그 닫히고 프로필 즉시 갱신 + 토스트 "프로필 사진이 변경되었어요"
- AC-06: Given 변경 있는 상태에서 [취소]/X/ESC/배경 탭 / When 닫기 시도 / Then 파괴적 확인 다이얼로그 표시

### 엣지케이스

- EC-01: 5MB 초과 파일 선택 시 → 크기 제한 에러 안내
- EC-02: 지원하지 않는 파일 형식(GIF, BMP 등) 선택 시 → 타입 에러 안내 (JPEG/PNG/WebP만 허용)
- EC-03: 변경 없이 [취소] → 즉시 닫힘 (확인 다이얼로그 없음)

### Phase 2 보강
- 이미지 크롭 도구 (원형 마스크 미리보기)
- AI 배경 제거
- 갤러리에서 선택 (본인이 올린 작품 중)

## MOD-12 · 회원 탈퇴 다이얼로그

| 항목 | 값 |
|---|---|
| 호출 위치 | 설정 화면 위험구역의 "회원 탈퇴" 탭 시 |

### 모달의 목적
회원 탈퇴 의사를 정중히 확인하고 **사유 수집** + 비밀번호 확인 (이메일 가입자) + 최종 파괴적 확인. 시니어가 실수로 탈퇴하지 않도록 단계 구분.

### 사용자 시나리오
- 주 사용자: 더 이상 서비스를 사용하지 않으려는 회원
- 기대: 명확한 안내 + 사유 선택만으로 빠른 진행. 강제 다단계 X

### 모달 구성

| 영역 | 요소 |
|---|---|
| 헤더 | "정말 탈퇴하시겠어요?" |
| 안내 | "탈퇴 시 작품·프로필이 다른 사용자에게 보이지 않게 됩니다" |
| **사유 라디오 (5종)** | 잘 안 쓰여서 / 다른 서비스 이용 / 개인정보 우려 / 사용이 어려움 / 기타 |
| (이메일 가입자) 비밀번호 확인 | 본인 확인용 |
| 안내 텍스트 | "Phase 2부터 데이터 삭제 요청을 별도로 받습니다. 현재는 비공개 처리만 됩니다." |
| 버튼 | [탈퇴하기] (파괴적/빨강) · [취소] |

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 사유 라디오 | 같은 다이얼로그에서 선택 |
| 비밀번호 입력 | 같은 다이얼로그에서 입력 |
| **[탈퇴하기]** | 사유 미선택이면 비활성. 통과 시 → **파괴적 최종 확인 다이얼로그** ("정말 탈퇴할까요? 되돌릴 수 없어요") → 확인 시 탈퇴 처리 → 로그아웃 → **로그인(`/login`)** 으로 이동 |
| [취소] / X / ESC / 배경 탭 | 다이얼로그 닫힘 (설정 화면으로 돌아감) |

### 탈퇴 처리 동작

1. 사용자 상태를 "탈퇴" 표시 (다른 사용자가 이 프로필 진입 시 빈 화면 + 안내)
2. 즉시 로그아웃
3. (Phase 1) 로컬 작품·프로필 데이터는 유지 (디버깅·복구 가능성)
4. (Phase 2) 서버 측 영구 삭제 또는 익명화

### 검증·정책
- 사유 미선택 시 [탈퇴하기] 비활성
- 비밀번호 틀리면 토스트 "비밀번호가 일치하지 않습니다"
- 탈퇴 후 같은 이메일·전화로 30일 내 재가입 차단 (Phase 2 정책)

### 수용기준 (Acceptance Criteria)

- AC-01: Given 회원 탈퇴 다이얼로그 / When 사유 라디오 5종 중 하나 선택 / Then 해당 사유가 선택 상태로 표시된다
- AC-02: Given 사유 미선택 상태 / When [탈퇴하기] 확인 / Then 버튼이 비활성 상태이다
- AC-03: Given 이메일 가입자 / When 비밀번호를 틀리게 입력 후 [탈퇴하기] / Then 토스트 "비밀번호가 일치하지 않습니다" 표시
- AC-04: Given 사유 선택 + (이메일 가입자는 비밀번호 확인) 통과 후 [탈퇴하기] 클릭 / When 클릭 / Then 파괴적 최종 확인 다이얼로그 ("정말 탈퇴할까요? 되돌릴 수 없어요") 표시
- AC-05: Given 최종 확인 다이얼로그에서 확인 / When 확인 / Then 탈퇴 처리 → 즉시 로그아웃 → 로그인(`/login`)으로 이동
- AC-06: Given 탈퇴 완료 후 / When 다른 사용자가 탈퇴 회원 프로필 진입 / Then 빈 화면 + 안내 표시
- AC-07: Given [취소]/X/ESC/배경 탭 / When 닫기 / Then 다이얼로그 닫히고 설정 화면으로 돌아감

### 엣지케이스

- EC-01: 소셜 가입자의 경우 비밀번호 확인란 대신 소셜 재인증 필요 (Phase 1에서는 생략 가능)
- EC-02: 최종 확인 다이얼로그에서 취소 → 탈퇴 다이얼로그로 복귀 (사유 선택 유지)
- EC-03: 탈퇴 후 같은 이메일·전화로 즉시 재가입 시도 → Phase 2에서 30일 재가입 차단 적용 예정

### Phase 2 보강
- 데이터 내보내기 옵션 ("탈퇴 전에 내 작품·기록 다운로드받기")
- 일시 비활성화 (탈퇴 대신 일시 휴면)
- 이유별 운영팀 분석 → 이탈 원인 개선

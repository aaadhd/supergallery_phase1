| 레이아웃 | 공통 Layout |

### 화면의 목적
1:1 문의 접수.

### 화면 구성

| 필드 | 규칙 |
|---|---|
| 카테고리 (드롭다운) | 계정 / 업로드 / 신고 / 제안 / 버그 / 기타 |
| 제목 | 필수 |
| 내용 (텍스트 영역) | 필수, 최소 10자 |
| 이메일 | 비로그인이면 필수, 로그인이면 자동 기입 |
| **첨부 파일** | 이미지 업로드 가능. 파일 크기 초과 시 토스트 안내 |
| 제출 버튼 | 제출 중엔 비활성화 |

### 탭·액션 결과 (네비게이션)

| 탭 대상 | 결과 |
|---|---|
| 카테고리 드롭다운 | 같은 화면에서 선택 |
| 파일 첨부 영역 | 기기 파일 선택 창 |
| 파일 크기 초과 | 화면 이동 없음 — 토스트 "파일이 너무 커요" |
| **[제출]** | 검증 통과 시 "제출 동작" 흐름 → 제출 완료 후 폼 초기화 (화면 유지). 사용자가 수동으로 다른 화면으로 이동 |

### 제출 동작

1. 검증 (필수 필드 + 최소 길이)
2. 문의 접수
3. **이중 토스트**:
   - 즉시: "접수되었습니다"
   - 5초간 유지: "24시간 내 답변드릴게요" (자동 응답 안내)
4. 폼 초기화 (화면 이동 없음)

### Phase 2 보강

## SCR-ERR-02 · 500 Server Error

| 항목 | 값 |
|---|---|
| URL | `/500` (수동 라우팅) |
| 접근 권한 | 모두 |
| 레이아웃 | 공통 Layout |

### 화면의 목적
백엔드·네트워크 에러 시 안내. 수동 라우팅 전용.

### 화면 구성
- 큰 "500" 숫자
- 메시지 + 힌트
- [새로고침] / [홈으로] 버튼

### 탭·액션 결과
| 탭 대상 | 결과 |
|---|---|
| [새로고침] | 현재 URL 재로드 |
| [홈으로] | 둘러보기(`/`)로 이동 |

### 앱 런타임 에러는 별도 처리

JavaScript 런타임 에러가 발생했을 때 이 500 페이지로 자동 이동하지 **않음** (라우팅 자체가 깨진 상황이라 불안정). 대신 **앱 최상위에 에러 경계(boundary)**가 있어 인플레이스 fallback UI를 렌더:

- 빨간 경고 아이콘 + "오류가 발생했어요" 타이틀
- 짧은 설명 + (개발 모드면) 에러 메시지
- [다시 시도] (화면 재시도) / [홈으로] 버튼

이 fallback은 다국어 리소스를 사용. 500 라우트는 **백엔드가 5xx 응답 시 서버에서 리다이렉트**하는 시나리오용.
